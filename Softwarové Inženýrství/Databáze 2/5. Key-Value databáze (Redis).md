- **[[5. Key-Value databáze (Redis)#Obecné principy|Obecné principy]]:** datový model (klíč-hodnota), správa klíčů (generované, strukturované, s předponami), základní CRUD operace, TTL.
- **[[5. Key-Value databáze (Redis)#Redis|Redis]]:** in-memory vlastnosti, objekty a datové typy (string, list, set, sorted set, hash).
- **[[5. Key-Value databáze (Redis)#Operace a příkazy|Operace a příkazy]]:**
    - String: SET, GET, INCR, DEL
    - List: LPUSH, RPUSH, LPOP, RPOP, LINDEX, LRANGE
    - Set: SADD, SISMEMBER, SUNION, SINTER, SDIFF, SREM
    - Sorted set: ZADD, ZRANGE, ZINCRBY, ZREM, ZUNIONSTORE
    - Hash: HSET, HMSET, HGET, HKEYS, HVALS, HDEL
    - Základní a TTL: EXISTS, RENAME, EXPIRE, PERSIST.

Key-value databáze jsou nejjednodušší typ NoSQL databází. Celá myšlenka je postavená na dvojici **klíč -> hodnota**. Klíč jednoznačně identifikuje záznam a hodnota je uložený obsah. Když klíč znám, databáze mi hodnotu velmi rychle vrátí. Když klíč neznám a chci hledat "uvnitř" hodnoty, key-value model přestává být pohodlný.

Redis je v předmětu hlavní zástupce key-value databází. Je důležitý nejen jako cache, ale i jako ukázka toho, že moderní key-value store nemusí ukládat jen obyčejné bajty. Redis umí strings, lists, sets, sorted sets, hashes, streams, geospatial indexy a další struktury. Pořád ale platí základní filozofie: nejdřív musím znát klíč.

# Obecné principy

Key-value databázi si lze představit jako obrovskou hash tabulku. Na jedné straně je klíč, na druhé hodnota. Klíč bývá textový identifikátor, ale obecně může jít o libovolnou binární sekvenci. Hodnota může být cokoliv: string, číslo, serializovaný objekt, JSON, obrázek nebo binární blob. U čistého key-value modelu databáze hodnotě nerozumí. Bere ji jako černou skříňku.

> **Hash tabulka:** Datová struktura, která podle klíče velmi rychle najde odpovídající hodnotu. Klíč se převede hash funkcí na pozici v interní struktuře.
>
> **Blob:** Binary Large Object. Obecné označení pro větší binární data, například obrázek nebo serializovaný soubor.
>
> **Serializace:** Převod objektu nebo datové struktury do podoby, kterou lze uložit jako bytes/string a později znovu načíst.

Tohle jednoduché schéma je zároveň největší síla i největší slabina. Síla je výkon. Pokud databáze nemusí řešit složitý dotazovací jazyk, joiny, optimalizátor a porozumění schématu hodnot, může být extrémně rychlá. Slabina je vyjadřovací schopnost. Pokud chci "najdi všechny uživatele starší 18 let z Prahy", key-value store mi s tím v základním modelu nepomůže, protože nevidí do hodnoty nebo nad ní nemá indexy.

> **Index:** Pomocná struktura pro rychlé vyhledávání podle určitého atributu. Relační databáze mají indexy nad sloupci; čistý key-value store typicky vyhledává hlavně podle klíče.

### Datový model

Základní záznam má tvar:

```text
key -> value
```

Klíč je identifikátor. V přednášce se objevují termíny `id`, `identifier`, `primary key`. Hodnota je objekt uložený pod tímto klíčem. U jednoduchého key-value systému je hodnota binární objekt a databáze neřeší jeho vnitřní strukturu.

Moderní key-value systémy, hlavně Redis, tento model rozšiřují. Hodnota už nemusí být jen obyčejný byte array. Může to být seznam, množina, sorted set nebo hash. Redis pak umí dělat operace přímo nad těmito strukturami, aniž by aplikace musela celou hodnotu přečíst, upravit a zapsat zpět.

> **Byte array:** Posloupnost bajtů. Databáze nemusí vědět, jestli bajty reprezentují text, obrázek, číslo nebo serializovaný objekt.

### Kdy key-value databáze použít

Key-value store se hodí tam, kde aplikace přistupuje k hodnotám přes známý klíč a potřebuje extrémně rychlé čtení nebo zápis. Typické příklady jsou cache, session data, nákupní košíky, uživatelské preference, dočasné tokeny, čítače, rate limiting, fronty úloh nebo rychlé lookupy jednotlivých objektů.

> **Cache:** Rychlé dočasné úložiště pro data, která lze případně znovu spočítat nebo načíst z pomalejšího zdroje.
>
> **Session:** Stav konkrétního uživatele mezi požadavky, například přihlášení, token, poslední aktivita nebo obsah košíku.
>
> **Rate limiting:** Omezování počtu požadavků, které může uživatel nebo IP adresa poslat za určitý čas.

Key-value databáze se naopak nehodí tam, kde hlavní práce spočívá v komplexních dotazech nad obsahem hodnot, analytice přes mnoho záznamů, joinování vztahů mezi entitami nebo ad-hoc filtrování. Když potřebuji často dotazovat podle mnoha atributů uvnitř dokumentu, je vhodnější dokumentová databáze. Když řeším vztahy, je vhodnější grafová databáze. Když řeším rozsáhlou analytiku podle přístupových vzorů, může dávat smysl wide-column databáze nebo Hadoop/Spark.

### Správa klíčů

U key-value databáze je návrh klíčů stejně důležitý jako návrh tabulek u relační databáze. Pokud klíč neznám, často se k datům rozumně nedostanu.

Klíče mohou být přirozené identifikátory, například e-mail nebo login. Mohou být generované, například UUID. V distribuovaném peer-to-peer prostředí je potřeba dávat pozor na auto-increment integer, protože centrální čítač se špatně škáluje a může být bottleneck.

> **UUID:** Universally Unique Identifier. Dlouhý identifikátor navržený tak, aby byl prakticky unikátní bez centrálního čítače.
>
> **Bottleneck:** Úzké hrdlo systému. Část, která omezuje výkon celku.

V praxi se často používají strukturované klíče s prefixy:

```text
user:1000:profile
user:1000:email
cart:1234
session:abc123
movie:603:rating
```

Prefix říká typ objektu a další části klíče identifikují konkrétní entitu nebo pole. Doporučené schéma z přednášky je například `object-type:id:field`. Důležitá je konzistence: když se jednou rozhodnu pro dvojtečky jako oddělovače, měl bych je používat všude stejně.

Redis klíče jsou **binary-safe**, takže mohou obsahovat libovolná binární data. Prázdný string je také validní klíč, ale prakticky se mu chceme vyhnout, protože je nečitelný a snadno matoucí. Délka klíče je kompromis: příliš dlouhý klíč žere paměť, příliš krátký klíč je nečitelný a zvyšuje riziko kolizí v návrhu aplikace.

> **Binary-safe:** Systém bere data jako libovolnou posloupnost bajtů a nespoléhá se na ukončovací znak nebo textové kódování.
>
> **Kolize v návrhu klíčů:** Situace, kdy dvě různé věci v aplikaci nechtěně používají stejný nebo nejasně odlišitelný klíč.

### CRUD a vyhledávání

Základní CRUD operace v key-value modelu jsou jednoduché: vytvořit hodnotu pro klíč, načíst hodnotu podle klíče, změnit hodnotu podle klíče a smazat hodnotu podle klíče.

> **CRUD:** Create, Read, Update, Delete. Čtyři základní typy práce s daty: vytvoření, čtení, úprava a smazání.

Zásadní omezení je, že tyto operace vyžadují znalost klíče. Redis sice umí procházet klíče příkazem `SCAN`, ale to není náhrada za dotazovací jazyk. Nad obrovskou databází je procházení klíčů pořád drahé a musí se používat opatrně.

> **SCAN:** Redis příkaz pro postupné procházení klíčů bez dlouhého blokování serveru. Je bezpečnější než hromadné `KEYS`, ale u velkých databází je stále nákladný.

### TTL a volatilní objekty

Velká výhoda key-value databází je práce s dočasnými daty. **TTL** znamená Time-To-Live, tedy doba, po které se klíč automaticky smaže. Hodí se pro sessions, ověřovací kódy, dočasné tokeny, cache, rate limiting nebo shopping carty, které po určité době neaktivity zmizí.

> **TTL (Time-To-Live):** Životnost klíče. Po uplynutí nastaveného času databáze objekt automaticky odstraní.

V Redis lze expiraci nastavit samostatně příkazem `EXPIRE`, v milisekundách přes `PEXPIRE`, nebo přímo při zápisu přes `SET key value EX seconds`.

Redis expirované klíče maže dvěma způsoby. **Passive expiration** se spustí při přístupu ke klíči: Redis zkontroluje, jestli už neměl vypršet, a pokud ano, smaže ho. **Active expiration** běží průběžně na pozadí: Redis náhodně vzorkuje klíče s expirací a maže ty prošlé.

> **Passive expiration:** Mazání expirovaného klíče až ve chvíli, kdy se k němu někdo pokusí přistoupit.
>
> **Active expiration:** Průběžné aktivní hledání a mazání expirovaných klíčů na pozadí.

Redis může být použit jako cache, a proto řeší i situaci, kdy paměť dochází. V přednášce se zmiňuje LFU eviction policy: když je cache plná, Redis může odstraňovat nejméně často používané položky. U velkých objektů pomáhá také lazy freeing, kdy se objekt označí ke smazání a paměť se uvolní na pozadí, aby mazání neblokovalo běžné příkazy.

> **Eviction policy:** Pravidlo, podle kterého databáze odstraňuje položky, když dochází paměť.
>
> **LFU (Least Frequently Used):** Strategie, která odstraňuje nejméně často používané položky.
>
> **Lazy freeing:** Odložené uvolnění paměti na pozadí, aby mazání velkého objektu nezpůsobilo vysokou latenci.

# Redis

Redis znamená **Remote Dictionary Server**. Je to open-source in-memory data structure store implementovaný v C. Může fungovat jako databáze, cache, message broker i streaming engine. Je známý velmi nízkou latencí a vysokou propustností; v přednášce se uvádí řádově více než 100 000 read/write operací za sekundu.

> **In-memory databáze:** Databáze, která drží hlavní pracovní dataset v operační paměti. Díky tomu je rychlá, ale musí řešit, jak data případně uložit trvale na disk.
>
> **Message broker:** Systém, přes který aplikace posílají zprávy jiným aplikacím nebo workerům.

Redis není jen "rychlá mapa". Má bohaté datové struktury, TTL, transakce, persistence, cluster, Pub/Sub, Streams, Lua scripting a další funkce. Přesto je dobré ho vnímat jako nástroj pro rychlé, dobře ohraničené přístupové vzory, ne jako univerzální náhradu relační databáze.

### Redis datový model

Redis instance obsahuje databáze, databáze obsahují objekty. Databáze v Redis nemají jména, ale číselné identifikátory. Objekt je key-value pár. Klíč je string, respektive binary-safe hodnota, a value může mít různé Redis datové typy.

Redis hodnoty nelze libovolně vnořovat. Například neexistuje "list hashů" jako jeden nativní vnořený typ. Pokud takovou strukturu potřebujeme, musíme ji modelovat pomocí více klíčů, serializace nebo jiné databáze.

> **Nativní datový typ:** Typ, kterému databáze přímo rozumí a poskytuje nad ním vlastní operace. Redis nativně rozumí např. listům a setům.

### Redis datové typy

**String** je jediný atomický datový typ. Může obsahovat text, číslo, serializovaný objekt, JSON, binární data nebo třeba obrázek. Maximální velikost string hodnoty je 512 MB. String se používá pro jednoduché hodnoty, cache, tokeny, čítače i binární objekty.

> **Atomický datový typ:** V tomto kontextu hodnota, kterou Redis nečlení na vnitřní prvky jako list nebo hash. Redis s ní pracuje jako s jedním kusem dat, i když nad stringem umí operace jako append nebo substring.

**List** je oboustranný seznam stringů. Operace na začátku a konci jsou rychlé, typicky `LPUSH`, `RPUSH`, `LPOP`, `RPOP`. Operace uprostřed seznamu jsou pomalejší, protože je potřeba prvky projít. Listy se hodí pro fronty, timeline a jednoduché sekvence.

> **Fronta (queue):** Datová struktura, do které se prvky vkládají na jednu stranu a odebírají na druhou. Typicky pro zpracování úloh.

**Set** je neuspořádaná množina unikátních stringů. Hodí se pro členství, unikátní položky, tagy, uživatele online nebo množinové operace. Umí rychle testovat členství (`SISMEMBER`) a provádět sjednocení, průnik a rozdíl (`SUNION`, `SINTER`, `SDIFF`).

**Sorted set** je množina unikátních členů, kde každý člen má číselné skóre. Redis podle skóre udržuje pořadí. Hodí se pro leaderboardy, rankingy, prioritní seznamy, časové osy nebo rotaci bannerů. Členové jsou unikátní, skóre se opakovat může.

> **Leaderboard:** Žebříček, například hráči se skóre ve hře. Sorted set je na to přirozený, protože umí řadit podle skóre.

**Hash** je mapa `field -> value` uložená pod jedním Redis klíčem. Hodí se pro objekty, například uživatele, session nebo košík. Místo mnoha klíčů `user:1000:name`, `user:1000:email`, `user:1000:verified` lze použít jeden hash `user:1000` s poli `name`, `email`, `verified`.

> **Hash v Redis:** Strukturovaná hodnota pod jedním klíčem. Uvnitř obsahuje pole a hodnoty, podobně jako jednoduchý objekt nebo slovník.

Redis má i další typy. **Bitmaps** a **bitfields** ukládají kompaktně bity nebo malé integery, což se hodí pro příznaky, feature flags, přítomnost uživatelů nebo úsporné čítače. **HyperLogLog** slouží pro odhad počtu unikátních prvků s malou paměťovou stopou. **Geospatial indexes** ukládají souřadnice a umožňují hledat objekty v okolí. **Streams** ukládají časově uspořádané záznamy pro stream processing.

> **Feature flag:** Přepínač, kterým lze zapnout nebo vypnout funkci aplikace bez nasazení nové verze.
>
> **HyperLogLog:** Pravděpodobnostní datová struktura pro odhad počtu unikátních prvků. Je velmi úsporná, ale výsledek je aproximace.
>
> **Geospatial index:** Index pro práci se souřadnicemi, například hledání nejbližších restaurací.
>
> **Stream:** Časově uspořádaný proud záznamů. Redis Streams se hodí pro event logy, messaging a real-time analytiku.

### Redis jako cache a primární úložiště

Redis se velmi často používá jako cache před pomalejší databází. Aplikace se nejdřív podívá do Redis. Pokud tam data jsou, odpoví rychle. Pokud tam nejsou, načte je z primární databáze a uloží do Redis pro příště.

To ale neznamená, že Redis je vždy jen cache. Redis může být i primární úložiště pro data, která přirozeně odpovídají jeho strukturám: session management, leaderboardy, realtime čítače, rate limiting, fronty, konfigurace, Pub/Sub notifikace nebo krátkodobá data s TTL.

> **Primární úložiště:** Místo, kde aplikace považuje data za hlavní zdroj pravdy. U cache lze data při ztrátě znovu načíst odjinud; u primárního úložiště je ztráta vážnější.

### Redis transakce

Redis příkazy se na jedné instanci vykonávají sekvenčně. Jednotlivý příkaz jako `INCR`, `HSET` nebo `ZINCRBY` je atomický: žádný jiný příkaz se nevmísí doprostřed jeho provedení.

> **Atomická operace:** Operace, která se provede jako nedělitelný celek. Ostatní klienti nevidí její mezistav.

Více příkazů lze seskupit pomocí `MULTI` a `EXEC`. Po `MULTI` se příkazy jen řadí do fronty. `EXEC` je provede sekvenčně bez prokládání příkazů jiných klientů. `DISCARD` frontu zahodí.

Redis transakce ale nejsou totéž co relační ACID transakce. Redis nemá rollback úspěšně provedených příkazů při execution-time chybě. Pokud se chyba zjistí už při zařazování příkazů, například špatný počet argumentů, `EXEC` celou transakci zruší. Pokud se chyba stane až při vykonávání, například operace nad špatným typem hodnoty, předchozí úspěšné příkazy zůstanou provedené.

> **Rollback:** Vrácení transakce zpět, jako by se vůbec nestala. Redis ho u execution-time chyb v transakci neposkytuje.

Pro optimistické zamykání slouží `WATCH`. Klient označí klíče, které sleduje. Pokud je někdo změní před `EXEC`, transakce selže a klient může výpočet zopakovat.

> **Optimistické zamykání:** Přístup, kde se data předem nezamknou. Systém jen při zápisu ověří, zda se mezitím nezměnila; pokud ano, operace se zopakuje nebo odmítne.

V Redis Clusteru jsou transakce omezené na jednu instanci. `MULTI/EXEC` funguje jen tehdy, když všechny klíče leží ve stejném hash slotu. K tomu se používají hash tagy, například `{movie:603}:counter` a `{movie:603}:log`.

> **Hash slot:** Redis Cluster rozděluje klíče do 16384 slotů. Slot se počítá z klíče pomocí CRC16 modulo 16384.
>
> **Hash tag:** Část klíče v `{...}`, kterou Redis Cluster použije pro výpočet hash slotu. Pomáhá uložit související klíče na stejný shard.

### Persistence

Redis je in-memory, ale umí data ukládat na disk. Persistence je důležitá, pokud Redis není jen cache, ale drží data, o která nechceme přijít.

Redis má několik možností:

**RDB** dělá point-in-time snapshot datasetu. Výsledkem je kompaktní binární soubor vhodný pro zálohy a rychlé restarty. Nevýhoda je možná ztráta změn od posledního snapshotu.

**AOF** loguje každou write operaci. Při restartu Redis příkazy přehraje a rekonstruuje dataset. Má lepší durability, ale soubor bývá větší a obnova může být pomalejší.

> **Snapshot:** Uložený obraz dat v určitém okamžiku.
>
> **AOF (Append Only File):** Redis log zapisovacích příkazů. Při obnově se příkazy znovu přehrají.
>
> **Durability:** Schopnost systému neztratit potvrzená data při pádu.

Redis může používat RDB, AOF, kombinaci RDB + AOF, nebo žádnou persistence, což je běžné u čisté cache. S RDB souvisí příkazy `BGSAVE`, `SAVE`, `LASTSAVE`. `SAVE` blokuje server, proto se v produkci nepoužívá běžně. `BGSAVE` ukládá na pozadí. U AOF je důležitý `BGREWRITEAOF`, který vytvoří kompaktnější verzi logu.

> **fsync:** Operační systémová operace, která vynutí zapsání dat z bufferů na disk. Častější fsync zvyšuje durability, ale zhoršuje výkon.

### Redis architektura, sharding a replikace

Redis podporuje horizontální škálování přes Redis Cluster. Klíče jsou rozdělené do 16384 hash slotů. Hash slot se počítá jako:

```text
HASH_SLOT = CRC16(key) mod 16384
```

Administrátor nastaví uzly a jejich role, Redis rozdělí sloty mezi primary uzly. Při přidání nebo odebrání uzlu se sloty rebalancují.

> **CRC16:** Hashovací/kontrolní algoritmus, který Redis Cluster používá pro výpočet hash slotu.
>
> **Rebalancing:** Přesun části dat nebo slotů mezi uzly, aby se zátěž po změně clusteru znovu rozložila.

Redis používá primary-replica replikaci. Primary přijímá zápisy, replica drží kopii. Replikace často probíhá asynchronně, takže může vzniknout zpoždění mezi primary a replicou. Pokud primary spadne dřív, než zápis dorazí na repliku, může se zápis ztratit.

> **Asynchronní replikace:** Primary nečeká na potvrzení repliky a odpoví klientovi dřív. Je rychlá, ale může ztratit poslední změny při výpadku.

Pro vyšší jistotu lze použít příkaz `WAIT`, který počká, až zápis potvrdí určitý počet replik. Tím se zvyšuje konzistence a durability, ale roste latence.

Redis Sentinel řeší monitoring a failover. Sleduje primary i replica uzly, posílá notifikace, při výpadku primary vybere vhodnou repliku, povýší ji na primary a klientům pomáhá najít aktuální primary.

> **Failover:** Přepnutí provozu na náhradní uzel po výpadku hlavního uzlu.
>
> **Quorum:** Minimální počet uzlů, které se musí shodnout na rozhodnutí, například že primary opravdu selhal.

Z pohledu CAP není Redis jednovětá odpověď. Jedna instance má silnou konzistenci v rámci svého event loopu. Cluster a replikace zavádí trade-off mezi dostupností a konzistencí. Asynchronní replikace preferuje výkon a dostupnost; synchronnější potvrzování přes `WAIT` preferuje jistotu za cenu latence.

> **Event loop:** Smyčka, ve které Redis postupně zpracovává příkazy. Příkazy nad daty se vykonávají sekvenčně, proto jsou jednotlivé příkazy atomické.

### Pub/Sub, Streams, geospatial a bitfields

Redis umí i funkce, které přesahují klasický key-value model.

**Pub/Sub** umožňuje posílat zprávy do kanálů. Klient se přihlásí pomocí `SUBSCRIBE news`, jiný klient pošle `PUBLISH news "hello"` a všichni odběratelé kanálu zprávu dostanou. Hodí se pro notifikace a jednoduchý messaging, ale zprávy se v klasickém Pub/Sub modelu neukládají pro pozdější čtení.

> **Pub/Sub:** Publish/Subscribe model. Publisher posílá zprávy do kanálu, subscribers je dostávají, pokud jsou právě přihlášeni.

**Redis Streams** jsou persistentnější proud záznamů. Každý záznam má ID založené na timestampu a sekvenčním čísle. Streams podporují consumer groups, čtení rozsahů a potvrzování zpracovaných zpráv přes `XACK`.

> **Consumer group:** Skupina konzumentů, kteří společně zpracovávají stream. Každá zpráva se typicky přiřadí jednomu členu skupiny.
>
> **Event sourcing:** Architektonický přístup, kde se stav systému odvozuje z posloupnosti událostí, ne jen z aktuálního stavu uloženého v tabulce.

**Geospatial příkazy** ukládají souřadnice a hledají body v okolí. Redis používá souřadnice v pořadí longitude, latitude. Příklady příkazů jsou `GEOADD`, `GEOPOS`, `GEODIST`, `GEOSEARCH`. Interně se používá geohashing a přesnost je omezená přibližně na 5-6 desetinných míst.

> **Longitude / latitude:** Zeměpisná délka a šířka. Redis u geospatial příkazů očekává nejdřív délku, potom šířku.
>
> **Geohash:** Způsob zakódování geografické polohy do řetězce tak, aby blízké body měly podobné prefixy.

**Bitfields** umožňují do jednoho Redis klíče zabalit více malých integer hodnot a manipulovat s nimi na bitové úrovni. Hodí se pro feature flags, malé čítače, rate limiting nebo tracking přítomnosti. Příkaz `BITFIELD` umí `GET`, `SET`, `INCRBY` a nastavitelné chování při overflow.

> **Overflow:** Přetečení číselného rozsahu, například když 8bit unsigned integer překročí maximum 255.

# Operace a příkazy

Tahle část je praktická. Ke státnicím není nutné memorovat každý detail syntaxe, ale je potřeba vědět, jaký datový typ který příkaz obsluhuje a jaký přístupový vzor tím Redis podporuje.

### Základní příkazy nad klíči

`SET key value` nastaví string hodnotu. `GET key` ji načte. `EXISTS key` ověří existenci. `TYPE key` vrátí typ hodnoty. `DEL key` smaže klíč. `RENAME key newkey` přejmenuje klíč.

```redis
SET demo:greeting "Hello, Redis!"
GET demo:greeting
EXISTS demo:greeting
TYPE demo:greeting
DEL demo:greeting
```

TTL nastavíme například přes `EXPIRE` nebo přímo při `SET`:

```redis
SET session:abc123 "user:1000" EX 3600
TTL session:abc123
PERSIST session:abc123
```

`TTL` vrací zbývající životnost v sekundách, `PTTL` v milisekundách. `PERSIST` odstraní expiraci a udělá klíč trvalým.

### String

String je základní hodnota. Používá se pro cache, tokeny, jednoduché atributy i čítače. Redis nad stringem umí atomické inkrementace, což je velmi častý pattern pro statistiky a rate limiting.

```redis
SET visits:total 0
INCR visits:total
INCRBY visits:total 5
GET visits:total
DEL visits:total
```

Další operace: `DECR`, `DECRBY`, `STRLEN`, `APPEND`, `GETRANGE`, `SETRANGE`.

> **Atomické inkrementace:** Zvýšení čítače jedním nedělitelným příkazem. Je bezpečné i při paralelních klientech na jedné Redis instanci.

### List

List je vhodný pro fronty a sekvence. `LPUSH` přidává na začátek, `RPUSH` na konec. `LPOP` odebírá ze začátku, `RPOP` z konce. Operace na krajích listu jsou rychlé.

```redis
RPUSH queue:tasks "send_email"
RPUSH queue:tasks "resize_image"
LPOP queue:tasks
LRANGE queue:tasks 0 -1
```

`LINDEX` vrátí prvek podle indexu, `LRANGE` rozsah prvků. `LTRIM` ořízne list na určený rozsah, což je užitečné třeba pro udržování posledních `N` záznamů. `LLEN` vrátí délku listu.

Z přednášky: `LPUSH`, `RPUSH`, `LPOP`, `RPOP` jsou `O(1)`, zatímco přístup doprostřed listu je pomalejší.

> **Časová složitost `O(1)`:** Operace trvá přibližně konstantní čas bez ohledu na velikost struktury.

### Set

Set ukládá unikátní členy bez pořadí. Hodí se pro členství ve skupině, unikátní návštěvníky, tagy nebo množinové operace.

```redis
SADD online:users alice bob charlie
SISMEMBER online:users alice
SREM online:users bob
SCARD online:users
```

Množinové operace:

```redis
SUNION set1 set2
SINTER set1 set2
SDIFF set1 set2
```

`SUNION` vrátí sjednocení, `SINTER` průnik a `SDIFF` rozdíl množin. Díky tomu lze jednoduše řešit otázky typu "kteří uživatelé jsou v obou skupinách".

### Sorted set

Sorted set ukládá unikátní členy se skóre. Redis je drží seřazené podle skóre. To je ideální pro žebříčky.

```redis
ZADD leaderboard 1000 "player1" 2000 "player2"
ZINCRBY leaderboard 500 "player1"
ZRANGE leaderboard 0 -1 WITHSCORES
ZREVRANGE leaderboard 0 2 WITHSCORES
ZREM leaderboard "player2"
```

`ZRANGE` čte podle pořadí, `ZRANGEBYSCORE` podle rozsahu skóre, `ZCOUNT` počítá členy v rozsahu skóre, `ZCARD` vrací počet členů. `ZUNIONSTORE` umí spojit více sorted setů do nového. To se hodí například pro pre-aggregované denní žebříčky, ze kterých vytvoříme týdenní top list.

> **Pre-aggregace:** Předpočítání výsledků dopředu, aby se při dotazu nemusely znovu procházet syrové události.

### Hash

Hash je vhodný pro objekty se sadou polí. Typický příklad je session, uživatel nebo košík.

```redis
HSET user:1000 username "antirez" birthyear 1977 verified 1
HGET user:1000 username
HMGET user:1000 username verified
HGETALL user:1000
HKEYS user:1000
HVALS user:1000
HDEL user:1000 verified
```

Pro čítače uvnitř hashe se používá `HINCRBY`:

```redis
HINCRBY stats:genre "Action" 1
HINCRBY stats:genre "Drama" 1
HGETALL stats:genre
```

Hash je často lepší než tisíce malých klíčů, pokud data logicky patří k sobě. Šetří režii klíčů a zpřehledňuje model.

### Typické Redis vzory

**Session management:** Session uložíme jako hash a nastavíme expiraci.

```redis
HSET session:abc123 user_id 1000 last_access 1631234567 is_logged_in 1
EXPIRE session:abc123 3600
HGETALL session:abc123
```

**Cache:** Hodnotu uložíme jako string s TTL.

```redis
SET user:profile:1000 "{\"name\":\"John\",\"email\":\"john@example.com\"}" EX 300
GET user:profile:1000
```

**Rate limiting:** Pro IP adresu inkrementujeme čítač a nastavíme expiraci okna.

```redis
INCR rate:ip:192.168.1.1
EXPIRE rate:ip:192.168.1.1 60
GET rate:ip:192.168.1.1
```

**Jednoduchá fronta:** Přidáváme úlohy do listu a worker je odebírá.

```redis
LPUSH queue:tasks "{\"task\":\"send_email\"}"
RPOP queue:tasks
```

**Leaderboard:** Sorted set drží hráče a skóre.

```redis
ZADD leaderboard 1000 "player1" 2000 "player2"
ZINCRBY leaderboard 500 "player1"
ZREVRANGE leaderboard 0 9 WITHSCORES
```

### Shrnutí ke státnicím

Key-value databáze jsou nejjednodušší NoSQL model: rychlé vyhledání hodnoty podle klíče. Díky jednoduchosti mají výborný výkon a dobře se hodí pro cache, sessions, košíky, čítače, fronty a dočasná data. Nehodí se pro složité dotazy podle obsahu hodnot, vztahy mezi entitami a rozsáhlou analytiku bez předem navržených klíčů nebo struktur.

Redis je key-value store rozšířený o bohaté datové struktury. String slouží pro jednoduché hodnoty a čítače, list pro fronty a sekvence, set pro unikátní množiny, sorted set pro žebříčky a hash pro objekty. TTL je klíčový pro cache a dočasná data.

Redis je velmi rychlý, protože pracuje hlavně v paměti a příkazy na jedné instanci vykonává sekvenčně. Pro trvalost nabízí RDB a AOF. Pro škálování má Redis Cluster s hash sloty, pro replikaci primary-replica model a pro failover Redis Sentinel. Transakce přes `MULTI/EXEC` zajišťují neprokládání příkazů, ale nejsou stejné jako relační transakce s rollbackem.
