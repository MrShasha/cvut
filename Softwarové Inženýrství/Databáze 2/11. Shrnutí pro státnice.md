# 1. Big Data a NoSQL

→ Detaily: [[1. Úvod do Big Data a NoSQL databází]]

**Big Data = 5V:**

| **V** | **Popis** | **Příklad** |
|---|---|---|
| **Volume** | Obrovské objemy dat (TB, PB, EB) | Logy, IoT senzory |
| **Velocity** | Rychlost generování a zpracování | Burzovní data, stream processing |
| **Variety** | Strukturovaná, semistrukturovaná, nestrukturovaná | SQL, JSON, video, text |
| **Veracity** | Kvalita a důvěryhodnost dat | Šum, chybějící hodnoty |
| **Value** | Byznysová hodnota z dat | Predikce, doporučení |

**Trendy:** Data Lakehouse, vektorové databáze (AI/LLM), Data Mesh, MLOps, stream processing (Kafka/Flink).

**NoSQL motivace** [[1. Úvod do Big Data a NoSQL databází#NoSQL databáze|→ sekce]]:
- Relační DB nezvládají horizontální škálování, striktní schéma, heterogenní data
- NoSQL: schemalessness, agregáty (čtení/zápis jako celek), automatická údržba replik
- **Typy:** key-value (Redis), wide-column (Cassandra), document (MongoDB), graph (Neo4j)

**Polyglot persistence:** Různé části aplikace používají různé databáze podle potřeby (Redis pro cache, Cassandra pro logy, Neo4j pro vztahy, relační pro transakce).

> [!note] Klíčová zkratka ke státnicím
> 5V: Volume, Velocity, Variety, Veracity, Value. NoSQL typy: KV, Wide-column, Document, Graph. Polyglot = různé DB pro různé účely.

# 2. Škálování a distribuce

→ Detaily: [[3. Principy distribuovaných systémů a NoSQL]]

**Vertikální škálování (scale-up):** silnější hardware, jednodušší konzistence, fyzické limity.
**Horizontální škálování (scale-out):** více uzlů, lineárně levnější, složitá konzistence a distribuce.

**Network fallacies** (falešné předpoklady o síti) [[3. Principy distribuovaných systémů a NoSQL#Škálování|→ sekce]]:
Síť není spolehlivá, latence není nulová, bandwidth není nekonečný, síť není bezpečná, topologie se mění.

**Sharding** [[3. Principy distribuovaných systémů a NoSQL#Distribuce dat (Sharding)|→ sekce]] — horizontální rozdělení dat:

| **Strategie** | **Princip** | **Pro / Proti** |
|---|---|---|
| Range-based | Rozsahy hodnot klíče | ✅ range queries, ❌ hotspoty |
| Hash sharding | `hash(key) mod N` | ✅ rovnoměrné, ❌ rozbití range queries |
| Consistent hashing | Uzly i klíče na kruhu | ✅ min přesun při změně, ❌ složitější |
| Directory-based | Explicitní adresář (config server) | ✅ flexibilní, ❌ critical point |

**Shard key:** atribut pro rozdělení dat; musí mít vysokou kardinalitu, odpovídat dotazům, nevytvářet hotspot.

**Replikace** [[3. Principy distribuovaných systémů a NoSQL#Replikace dat|→ sekce]]:
- **Master-slave:** zápis přes master, čtení ze slave; master je single point of failure
- **Peer-to-peer:** každý uzel může přijmout čtení i zápis (Cassandra); lepší HA

**Replication factor (RF):** počet kopií záznamu. RF = 3 → přežije výpadek 1 uzlu.

> [!note] Klíčová zkratka ke státnicím
> Scale-up = silnější server. Scale-out = více serverů + sharding. Consistent hashing = min přesun. Master-slave vs. peer-to-peer. RF = počet kopií.

# 3. CAP teorém a Konzistence

→ Detaily: [[3. Principy distribuovaných systémů a NoSQL#Teorémy a garance]]

**CAP teorém:** Distribuovaný systém může zaručit nejvýše **2 ze 3** vlastností:
- **C**onsistency — každé čtení vrátí aktuální data
- **A**vailability — každý požadavek dostane odpověď
- **P**artition tolerance — systém funguje i při výpadku části sítě

Protože network partition (P) je reálné a nelze ho ignorovat, volba je vždy mezi **CP** a **AP**:
- **CP** (Consistency + Partition): HBase, ZooKeeper — raději odmítne požadavek
- **AP** (Availability + Partition): Cassandra, CouchDB — raději vrátí starší data
- **CA** (bez distribuce): PostgreSQL, MySQL na jednom uzlu

**ACID vs. BASE** [[3. Principy distribuovaných systémů a NoSQL#Teorémy a garance|→ sekce]]:

| | **ACID** | **BASE** |
|---|---|---|
| **Filozofie** | Striktní konzistence | Eventual consistency |
| **Dostupnost** | Nižší (zámky, rollback) | Vyšší |
| **Kde** | Relační DB | Většina NoSQL |
| **Co zaručuje** | Atomicita, konzistence, izolace, trvalost | Basically Available, Soft state, Eventually consistent |

**Konzistence** [[3. Principy distribuovaných systémů a NoSQL#Konzistence|→ sekce]]:
- **Silná (strong):** čtení vždy vrátí nejnovější zapsaná data; dražší (quorum)
- **Případná (eventual):** repliky se nakonec shodnou; může vrátit starší data
- **Tunable consistency** (Cassandra): `W + R > RF` = strong, jinak eventual

**Read/Write Quorum:** `W` = počet potvrzení zápisu, `R` = počet odpovědí čtení, `RF` = replikační faktor.
- Strong consistency: $W + R > RF$, např. RF=3, W=2, R=2

**Výkon** [[3. Principy distribuovaných systémů a NoSQL#Ladění výkonu|→ sekce]]:
- **Amdahlův zákon:** max zrychlení omezeno sériovou částí kódu: $S = \frac{1}{(1-p) + p/N}$
- **Littleův zákon:** $L = \lambda W$ (průměrný počet položek = propustnost × průměrný čas)
- **Message cost model:** cena distribuce ≠ jen výpočet; zahrnuje i síťový overhead

> [!note] Klíčová zkratka ke státnicím
> CAP: max 2 ze 3, reálně volba CP vs AP. ACID = striktní. BASE = eventual. Quorum: W+R>RF → strong. Amdahl: sériová část limituje škálování.

# 4. MapReduce a Hadoop

→ Detaily: [[4. MapReduce a Hadoop]]

**MapReduce** [[4. MapReduce a Hadoop#MapReduce|→ sekce]] — batch processing nad distribuovanými daty:

```text
map(input_key, input_value)   -> list(intermediate_key, intermediate_value)
reduce(intermediate_key, list(values)) -> list(output_values)
```

**Fáze zpracování:**
1. **Parsing** — rozdělení vstupu na input splity
2. **Mapping** — každý mapper zpracuje svůj split, vyprodukuje (key, value) páry
3. **Combining** — lokální mini-reduce na každém mapperu (musí být komutativní a asociativní)
4. **Shuffling & Partitioning** — přesun mezivýsledků ke správným reducerům (by key)
5. **Merging** — slučování vstupu pro reducer
6. **Reducing** — finální výstup

**Stragglers** — pomalé uzly způsobují zdržení celé jobu. Řeší se spekulativním spuštěním zálohy.

**Combine funkce:** stejná logika jako reduce, ale lokální → šetří síť. Podmínka: **komutativita a asociativita** (výsledek musí být stejný bez ohledu na pořadí).

**Hadoop** [[4. MapReduce a Hadoop#Hadoop a HDFS|→ sekce]]:

| **Modul** | **Úloha** |
|---|---|
| **HDFS** | Distribuovaný souborový systém, bloky (128 MB), rack-aware replikace |
| **YARN** | Resource manager; oddělení plánování zdrojů od logiky jobu |
| **MapReduce** | Framework pro spouštění MapReduce jobů nad HDFS |

**HDFS architektura:**
- **NameNode:** metadata (umístění bloků, namespace); single point — chráněn standby
- **DataNode:** ukládá fyzické bloky; periodicky posílá HeartBeat
- **FsImage + EditLog:** NameNode drží snapshot (FsImage) + log změn (EditLog)
- **Rack-aware replikace:** 1. replika lokálně, 2. v jiném racku, 3. ve stejném racku jako 2.

> [!note] Klíčová zkratka ke státnicím
> MapReduce: map → shuffle → reduce. Combine: komutativní + asociativní. HDFS: NameNode (metadata) + DataNode (data). YARN: resource management. Stragglers → spekulativní spuštění.

# 5. Redis (Key-Value)

→ Detaily: [[5. Key-Value databáze (Redis)]]

**Redis** = in-memory key-value databáze. Hodnoty nejsou jen byty — Redis má bohaté datové typy:

| **Typ** | **Klíčové operace** | **Typické použití** |
|---|---|---|
| **String** | SET, GET, INCR, DEL | Cache, čítače, tokeny |
| **List** | LPUSH, RPUSH, LPOP, RPOP, LRANGE | Fronty, log, Pub/Sub |
| **Set** | SADD, SISMEMBER, SUNION, SINTER, SDIFF | Tagy, unikátní položky |
| **Sorted set** | ZADD, ZRANGE, ZINCRBY, ZUNIONSTORE | Žebříčky, prioritní fronty |
| **Hash** | HSET, HGET, HKEYS, HVALS, HDEL | Uživatelský profil, session |

**TTL (Time To Live)** [[5. Key-Value databáze (Redis)#Operace a příkazy|→ sekce]]:
- `EXPIRE key seconds` — nastav exspiraci
- `PERSIST key` — zruš expiraci
- `TTL key` — kolik sekund zbývá

**Správa klíčů:** strukturované prefixové klíče (`user:42:profile`), pattern matching (`KEYS user:*`).

**Persistence Redis:**
- **RDB (Snapshotting):** periodický dump na disk; rychlý restart, může ztratit poslední data
- **AOF (Append Only File):** log každého příkazu; bezpečnější, pomalejší restart

> [!note] Klíčová zkratka ke státnicím
> Redis = in-memory. 5 typů: String, List, Set, Sorted Set, Hash. TTL: EXPIRE/PERSIST. Persistence: RDB (snapshot) vs AOF (log příkazů).

# 6. Cassandra (Wide Column)

→ Detaily: [[6. Wide Column Stores (Cassandra)]]

**Datový model** [[6. Wide Column Stores (Cassandra)#Obecné principy|→ sekce]]:
```text
instance → keyspace → table → row → column
```
Návrh tabulek se řídí **dotazy** (query-driven design), ne normalizací.

**Primární klíč v Cassandře:**
- **Partition key** → určuje, na který uzel data patří (hashing)
- **Clustering columns** → řadí data uvnitř partition; umožňují range queries

```sql
PRIMARY KEY ((sensor_id, day), timestamp)
-- partition: (sensor_id, day), clustering: timestamp
```

**Architektura:** peer-to-peer, consistent hashing, virtual nodes, gossip protokol. Žádný single master.

**Consistency levels (CQL):** `ONE`, `QUORUM`, `ALL` — volba pro každý dotaz. `QUORUM + QUORUM` → strong.

**CQL — klíčové příkazy** [[6. Wide Column Stores (Cassandra)#Cassandra Query Language (CQL)|→ sekce]]:
```sql
CREATE KEYSPACE ks WITH replication = {'class': 'SimpleStrategy', 'replication_factor': 3};
CREATE TABLE ks.events (sensor_id text, day date, ts timestamp, val float,
  PRIMARY KEY ((sensor_id, day), ts)) WITH CLUSTERING ORDER BY (ts DESC);
INSERT INTO ks.events (sensor_id, day, ts, val) VALUES ('s1', '2024-01-01', now(), 23.5);
SELECT * FROM ks.events WHERE sensor_id='s1' AND day='2024-01-01' LIMIT 10;
```

**WHERE omezení:** v CQL lze filtrovat v WHERE pouze přes partition key + clustering columns. `ALLOW FILTERING` obejde omezení, ale je pomalé (full scan).

**Tombstones:** logické značky smazaných dat (DELETE nevymaže ihned). Fyzicky se mažou při compaction.

**Datové typy:** primitiva (text, int, float, boolean, uuid, timestamp, date), kolekce (list, set, map), tuple, UDT (user-defined type). Kolekce ve `frozen` režimu = neměnitelné, uložené jako blob.

> [!note] Klíčová zkratka ke státnicím
> Cassandra: peer-to-peer, consistent hashing, query-driven design. Primary key = partition key + clustering columns. CQL WHERE: jen přes PK. Tombstones = logické mazání. Tunable consistency: W+R>RF → strong.

# 7. MongoDB (Document)

→ Detaily: [[7. Dokumentové databáze (MongoDB)]]

**Datový model** [[7. Dokumentové databáze (MongoDB)#Obecné principy|→ sekce]]:
```text
instance → database → collection → document (BSON)
```
Dokumenty jsou JSON-like, `_id` je primární klíč (automaticky `ObjectId` pokud nevyplněn).

**Modelování:**
- **Embedding (vnořené dokumenty):** data, která se čtou společně → jeden dokument. Výhoda: 1 dotaz. Nevýhoda: duplicita, velké dokumenty.
- **Reference:** odkaz na `_id` jiného dokumentu. Výhoda: normalizace. Nevýhoda: nutnost join přes `$lookup`.

**CRUD operace** [[7. Dokumentové databáze (MongoDB)#Dotazování a CRUD operace|→ sekce]]:
```javascript
db.col.insertOne({...})  / insertMany([...])
db.col.find({field: value}, {projection: 1})
db.col.updateOne({filter}, {$set: {field: val}}, {upsert: true})
db.col.deleteOne({filter})
```

**Operátory dotazů** [[7. Dokumentové databáze (MongoDB)#Operátory|→ sekce]]:

| **Kategorie** | **Operátory** |
|---|---|
| Porovnávací | `$eq`, `$ne`, `$gt`, `$gte`, `$lt`, `$lte`, `$in`, `$nin` |
| Logické | `$and`, `$or`, `$not`, `$nor` |
| Prvek | `$exists`, `$type` |
| Pole | `$all`, `$elemMatch`, `$size` |
| Projekce | `$slice`, `$elemMatch` |

**Update operátory:** `$set`, `$unset`, `$inc`, `$push`, `$pull`, `$addToSet`, `$rename`

**Aggregační pipeline** [[7. Dokumentové databáze (MongoDB)#Aggregační framework|→ sekce]]:
```javascript
db.col.aggregate([
  { $match: { year: { $gt: 2000 } } },
  { $group: { _id: "$genre", count: { $sum: 1 } } },
  { $sort: { count: -1 } },
  { $limit: 5 }
])
```
Hlavní stages: `$match`, `$group`, `$project`, `$sort`, `$limit`, `$skip`, `$unwind`, `$lookup`, `$count`.

**Indexování** [[7. Dokumentové databáze (MongoDB)#Indexování|→ sekce]]:
- Typy: **single field**, **compound**, **multikey** (pole), **hashed**, **text**, **geospatial**
- Pravidlo **ESR** (pro compound indexy): Equality → Sort → Range
- Vlastnosti: `unique`, `partial` (podmíněný), `sparse` (jen nenull), `TTL` (automatické mazání)

> [!note] Klíčová zkratka ke státnicím
> MongoDB: BSON dokumenty, `_id` = ObjectId. Embedding vs. reference. Aggregační pipeline: match → group → sort. ESR pravidlo pro složené indexy. `$lookup` = JOIN.

# 8. Neo4j a Cypher (Grafová DB)

→ Detaily: [[8. Grafové databáze (Neo4j)]]

**Property graph model** [[8. Grafové databáze (Neo4j)#Neo4j Datový model|→ sekce]]:
- **Uzly (nodes):** mají labely (`:Person`, `:Movie`) a properties (`{name: "Ivan"}`)
- **Vztahy (relationships):** orientované, mají typ (`:ACTED_IN`) a properties (`{role: "Ondřej"}`)
- Multi-hop traversal je přirozený — žádný JOIN, přímé procházení hran

**Traversal framework** [[8. Grafové databáze (Neo4j)#Traversal framework|→ sekce]]:
- **Traversal description:** konfigurace procházení (start uzel, expander, uniqueness, evaluátor)
- **Expander:** definuje, které hrany a uzly procházet
- **Uniqueness:** `NODE_GLOBAL`, `RELATIONSHIP_GLOBAL`, `NODE_PATH` — jak hlídat cykly
- **Evaluátor:** `ALL`, `ALL_BUT_START_NODE`, filtrace dle hloubky/podmínky → vrátit/prořezat
- **Traverser:** výsledek — iterable přes Node nebo Path

**Cypher** [[8. Grafové databáze (Neo4j)#Cypher Query Language|→ sekce]]:

```cypher
// Vyhledávání vzorů
MATCH (p:Person)-[:ACTED_IN]->(m:Movie {year: 2000})
WHERE p.name STARTS WITH 'Ivan'
RETURN p.name, m.title

// Délky cest
MATCH path = (a)-[:KNOWS*1..3]->(b)
RETURN path

// Zápis
CREATE (p:Person {name: 'Jan'})
MERGE (p)-[:KNOWS]->(q)
SET p.age = 30
DELETE r  // vztah

// Agregace
MATCH (p:Person)-[:ACTED_IN]->(m)
RETURN m.title, count(p) AS actors
ORDER BY actors DESC LIMIT 5

// WITH pro řetězení
MATCH (p:Person)-[:ACTED_IN]->(m)
WITH p, count(m) AS filmCount
WHERE filmCount > 5
RETURN p.name, filmCount
```

**Klauzule přehled:**

| **Typ** | **Klauzule** |
|---|---|
| Čtení | `MATCH`, `OPTIONAL MATCH`, `WHERE` |
| Zápis | `CREATE`, `MERGE`, `SET`, `REMOVE`, `DELETE`, `DETACH DELETE` |
| Obecné | `RETURN`, `WITH`, `ORDER BY`, `LIMIT`, `SKIP`, `UNWIND` |
| Agregace | `count()`, `sum()`, `avg()`, `min()`, `max()`, `collect()` |

> [!note] Klíčová zkratka ke státnicím
> Neo4j: property graph, uzly + vztahy (orientované) + properties. Traversal: description → expander → evaluátor → traverser. Cypher: MATCH vzory, `-[:TYP*min..max]->` pro délky cest, WITH pro řetězení.

# 9. Grafové datové struktury

→ Detaily: [[9. Pokročilé aspekty grafových databází]]

**Reprezentace grafu** $G = (V, E)$, $n = |V|$, $m = |E|$:

| **Struktura** | **Existence hrany** | **Sousedé uzlu** | **Paměť** | **Vhodné pro** |
|---|---|---|---|---|
| **Matice sousednosti** | $O(1)$ | $O(n)$ | $O(n^2)$ | Husté grafy, maticové operace |
| **Seznam sousedů** | $O(\deg(v))$ | $O(\deg(v))$ | $O(n+m)$ | Řídké grafy (typické v praxi) |
| **Matice incidence** | $O(m)$ | $O(m)$ | $O(n \cdot m)$ | Hypergrafy, analýza hran |
| **Laplaceova matice** | — | — | $O(n^2)$ | Spektrální analýza, clustering |

**Matice sousednosti:** $A[i][j] = 1$ iff hrana $i \to j$. Neorientovaný graf → symetrická matice.

**Laplaceova matice:** $L = D - A$, kde $D$ = diagonální matice stupňů. Vlastní čísla $\lambda_1 = 0$ (vždy), $\lambda_2 > 0$ iff graf je souvislý.

**Lokalita dat (BFS layout)** [[9. Pokročilé aspekty grafových databází#Lokalita dat a optimalizace|→ sekce]]:
- Uzly blízké v grafu by měly být blízké v paměti → méně cache miss
- **BFS layout:** přečísluj uzly v pořadí BFS traversalu → sousedé blízko v paměti
- **Bandwidth minimization:** $\min \max_{(i,j) \in E} |i - j|$ (NP-těžké obecně)
- **Cuthill-McKee algoritmus:** heuristika pro snížení bandwidth matice → reverse Cuthill-McKee (RCM) dává lepší výsledky

**Grafové dělení (Partitioning)** [[9. Pokročilé aspekty grafových databází#Zpracování velkých grafů|→ sekce]]:
- **1D partitioning** (vertex-cut): každý uzel patří jednomu procesoru; hrany přes procesory se duplikují
- **2D partitioning** (edge-cut): matice sousednosti se rozdělí na 2D bloky; vyvažuje komunikaci lépe pro husté grafy

> [!note] Klíčová zkratka ke státnicím
> Matice sousednosti: O(1) hrana, O(n²) paměť. Seznam sousedů: O(n+m). Laplaceova matice: L=D-A, λ₂>0 ↔ graf je souvislý. BFS layout = data locality. Cuthill-McKee = snížení bandwidth. 1D = vertex-cut, 2D = edge-cut.

# 10. SPARQL a RDF

→ Detaily: [[10. RDF Stores a Sémantický web (SPARQL)]]

**RDF datový model** [[10. RDF Stores a Sémantický web (SPARQL)#Linked Data|→ sekce]]:
- Všechno jako **trojice:** `(subjekt, predikát, objekt)`
- Subjekt a predikát jsou IRI; objekt je IRI nebo literál
- Graf trojic = orientovaný popsaný graf (RDF graph)
- **Blank node:** anonymní uzel bez globálního IRI (lokální pomocná struktura)

**Formáty zápisu:**
- **N-Triples:** jedna trojice na řádek, nejjednodušší
- **Turtle:** kompaktní zápis s prefixy, čitelný

```turtle
@prefix m: <http://example.org/movie/> .
@prefix i: <http://example.org/property/> .

m:medvidek i:title "Medvídek" ;
           i:year  2007 ;
           i:actor m:trojan .
```

**SPARQL** [[10. RDF Stores a Sémantický web (SPARQL)#SPARQL|→ sekce]] — dotazovací jazyk nad RDF:

**Formy dotazů:**
- `SELECT` — vrátí tabulku výsledků (variabilní vazby)
- `ASK` — vrátí true/false
- `DESCRIBE` — vrátí RDF graf o zdrojích
- `CONSTRUCT` — vrátí nový RDF graf dle šablony

**Grafové vzory:**
```sparql
PREFIX m: <http://example.org/movie/>
PREFIX i: <http://example.org/property/>

SELECT ?title ?actor
WHERE {
  ?movie i:title ?title .          # basic graph pattern
  ?movie i:actor ?actor .
  OPTIONAL { ?movie i:year ?year . # optional pattern
             FILTER (?year > 2000) }
  FILTER (lang(?title) = "cs")
}
ORDER BY ?title
LIMIT 10
```

**Vzory přehled:**

| **Vzor** | **Popis** |
|---|---|
| Basic Graph Pattern (BGP) | Sada trojic s proměnnými; vše musí matchovat |
| `OPTIONAL` | Levý outer join; proměnné zůstávají nevázané pokud vzor nesedí |
| `UNION` | Alternativní vzory; vrátí výsledky z obou větví |
| `MINUS` | Odečtení výsledků jiného vzoru |
| `GRAPH` | Dotaz přes pojmenovaný graf |

**Modifikátory:** `DISTINCT`, `REDUCED`, `ORDER BY`, `LIMIT`, `OFFSET`, `GROUP BY`, `HAVING`, `BIND`.

**Linked Data principy:** identifikace přes IRI, HTTP dereference, propojování dat přes IRI, otevřené formáty a licence.

> [!note] Klíčová zkratka ke státnicím
> RDF: trojice (S, P, O). IRI = globální ID. Blank node = anonymní. SPARQL: SELECT/ASK/DESCRIBE/CONSTRUCT. BGP = sada trojic. OPTIONAL = outer join. FILTER = podmínka. UNION = alternativa.

# Přehled systémů a jejich vlastností

| **Systém** | **Typ** | **Datový model** | **Škálování** | **Konzistence** | **Vhodné pro** |
|---|---|---|---|---|---|
| **Redis** | Key-Value | klíč → hodnota (str/list/set/hash) | Cluster (hash slots) | Strong (single node) | Cache, session, čítače |
| **Cassandra** | Wide Column | keyspace → table → row → column | Peer-to-peer, consistent hashing | Tunable (quorum) | Logy, časové řady, high write |
| **MongoDB** | Document | DB → collection → BSON document | Sharding + replication | Tunable (write concern) | Heterogenní data, flexibilní schéma |
| **Neo4j** | Graph | Uzly + vztahy (property graph) | Replikace (ne nativní sharding) | ACID (single node) | Vztahy, doporučení, fraud detection |
| **Hadoop/HDFS** | Batch | Soubory v blocích | NameNode + DataNodes | — (batch, immutable) | ETL, dávková analytika |
| **RDF store** | Triple store | Trojice (S, P, O) | Varies | — | Sémantický web, znalostní grafy |

## Big Data, NoSQL, Škálování, CAP

**Typické okruhové otázky:**
- Co jsou Big Data? Popište 5V.
- Jaké jsou typy NoSQL databází a jaká jsou jejich typická použití?
- Vysvětlete rozdíl mezi vertikálním a horizontálním škálováním.
- Co je CAP teorém? Jaký je rozdíl mezi CP a AP systémy?
- Co je BASE? Jak se liší od ACID?
- Co je eventual consistency? Kdy je to přijatelné?
- Co je quorum? Kdy platí strong consistency?
- Vysvětlete Amdahlův zákon.

**Big Data 5V:** Volume (objem), Velocity (rychlost), Variety (rozmanitost), Veracity (věrohodnost), Value (hodnota). NoSQL vzniklo kvůli nutnosti horizontálního škálování, schemalessness a různorodosti dat.

**Škálování:** Scale-up = silnější HW (limity, drahé). Scale-out = více serverů (levné, složitá konzistence). Síť není spolehlivá (network fallacies).

**CAP:** Konsistence + Dostupnost + Partition tolerance → max 2. Partition nelze ignorovat → volba CP (HBase) nebo AP (Cassandra).

**ACID vs. BASE:** ACID = striktní (relační DB). BASE = Basically Available, Soft state, Eventually consistent (NoSQL).

**Quorum:** W + R > RF → strong consistency. Příklad RF=3: W=2, R=2 → 2+2>3, strong.

**Amdahlův zákon:** $S(N) = \frac{1}{(1-p) + p/N}$, kde $p$ = paralelizovatelný podíl, $N$ = počet procesorů. Sériová část limituje maximální zrychlení.

## MapReduce a Hadoop

**Typické okruhové otázky:**
- Popište fáze MapReduce výpočtu.
- Jaké jsou vlastnosti combine funkce? Proč musí být komutativní a asociativní?
- Popište architekturu Hadoop clusteru (HDFS, YARN).
- Co jsou stragglers a jak se řeší?

**MapReduce fáze:** input splits → map (key, value) → combine (lokální, volitelné) → shuffle/partition (klíče ke správným reducerům) → merge → reduce → výstup.

**Combine:** lokální mini-reduce, šetří síť. Musí být komutativní ($a \oplus b = b \oplus a$) a asociativní ($(a \oplus b) \oplus c = a \oplus (b \oplus c)$) — výsledek nesmí záviset na pořadí. Například `sum`, `min`, `max` jsou v pořádku; průměr (`avg`) nikoliv.

**HDFS:** NameNode (metadata, namespace, seznam bloků) + DataNodes (data, bloky 128 MB). HeartBeat = DataNode hlásí živost. Rack-aware: 1. replika lokálně, 2. jiný rack, 3. stejný rack jako 2.

**Stragglers:** pomalé uzly zdržují celý job. Řeší se spekulativním spuštěním duplikátu tasku na jiném uzlu; kdo skončí první, vyhraje.

## Redis

**Typické okruhové otázky:**
- Jaké datové typy podporuje Redis?
- Jak funguje TTL? K čemu se používá?
- Jaký je rozdíl mezi RDB a AOF persistencí?

**Datové typy:** String, List (LPUSH/RPOP = fronta), Set (unikátní, množinové operace), Sorted Set (ZADD s score → žebříčky), Hash (objekt jako mapa).

**TTL:** `EXPIRE key 3600` → klíč expiruje za hodinu. `PERSIST` = zruš expiraci. Vhodné pro session, cache, rate limiting.

**Persistence:** RDB = pravidelný snapshot (kompaktní, rychlý restart, může ztratit posledních X sekund). AOF = log každého příkazu (bezpečnější, větší soubor, pomalejší restart). Lze kombinovat.

## Cassandra

**Typické okruhové otázky:**
- Jak funguje primární klíč v Cassandře (partition key, clustering columns)?
- Co je tunable consistency? Jak zajistit strong consistency?
- Proč nelze v CQL filtrovat libovolně v WHERE?
- Co jsou tombstones?

**Primární klíč:** `PRIMARY KEY ((partition_key), clustering_col1, clustering_col2)`. Partition key → hashing → uzel. Clustering columns → řazení uvnitř partition. Špatný partition key → hotspot.

**Tunable consistency:** každý dotaz specifikuje consistency level (ONE, QUORUM, ALL). Strong: $W + R > RF$. Cassandra je AP systém, ale tunable → dočasně CP chování.

**WHERE v CQL:** povoleno jen přes partition key + clustering columns (v pořadí). `ALLOW FILTERING` = full scan, pomalé, nevhodné v produkci.

**Tombstones:** DELETE v Cassandře nevymaže data okamžitě — zapíše tombstone (logická značka smazání). Fyzicky se maže při compaction. Příliš mnoho tombstones může snížit výkon.

## MongoDB

**Typické okruhové otázky:**
- Jaký je rozdíl mezi embedding a reference v MongoDB?
- Popište aggregační pipeline.
- Co je ESR pravidlo pro indexy?
- Jaké operátory MongoDB znáte?

**Embedding vs. reference:** Embedding = vnořené dokumenty (1 dotaz, vhodné pro data čtená společně). Reference = odkaz přes `_id` (normalizace, vhodné pro sdílená nebo velká data, nutný `$lookup`).

**Aggregační pipeline:** `$match` (filtruj) → `$group` (agreguj) → `$project` (projekce) → `$sort` → `$limit`. Každý stage transformuje stream dokumentů.

**ESR pravidlo:** Compound index: nejdřív pole pro **E**quality, pak **S**ort, pak **R**ange. Optimalizuje využití indexu.

**Klíčové operátory:** `$eq/$ne/$gt/$lt` (porovnání), `$in/$nin` (množina), `$and/$or/$not`, `$exists`, `$elemMatch` (pole), `$set/$inc/$push/$pull` (update).

## Neo4j a Cypher

**Typické okruhové otázky:**
- Popište property graph model.
- Jak funguje traversal framework v Neo4j?
- Napište Cypher dotaz pro nalezení přátel přátel.
- Jaký je rozdíl mezi `MATCH` a `OPTIONAL MATCH`?

**Property graph:** Uzly (labely + properties) + vztahy (orientované, typ + properties). Multi-hop traversal přirozeně efektivní — přímé ukazatele, ne JOINy.

**Traversal framework:** Traversal description specifikuje start uzel, expander (které hrany procházet), uniqueness (jak hlídat cykly: NODE\_GLOBAL, RELATIONSHIP\_GLOBAL) a evaluátor (kdy vrátit/prořezat uzel). Traverser je výsledný iterátor.

**Přátelé přátel (Cypher):**
```cypher
MATCH (me:Person {name: 'Jan'})-[:KNOWS]->(friend)-[:KNOWS]->(fof)
WHERE NOT (me)-[:KNOWS]->(fof) AND me <> fof
RETURN DISTINCT fof.name
```

**MATCH vs. OPTIONAL MATCH:** MATCH = inner join (nepasující řádky vypadnou). OPTIONAL MATCH = left outer join (proměnné zůstanou null pokud vzor nesedí).

## SPARQL

**Typické okruhové otázky:**
- Jaké jsou formy SPARQL dotazů?
- Jaký je rozdíl mezi BGP a OPTIONAL?
- Jak funguje FILTER?

**Formy:** SELECT (tabulka), ASK (boolean), DESCRIBE (RDF graf o zdrojích), CONSTRUCT (nový RDF graf).

**BGP vs. OPTIONAL:** BGP = všechny trojice musí matchovat (inner join). OPTIONAL = přidá vzor jako left join; pokud nesedí, proměnné jsou unbound (null).

**FILTER:** podmínky nad proměnnými ve WHERE: `FILTER (?age > 18)`, `FILTER (lang(?label) = "cs")`, `FILTER regex(?name, "^Jan")`. Logické spojky: `&&`, `||`, `!`.

**UNION:** alternativní vzory — vrátí výsledky z obou větví (sjednocení).
