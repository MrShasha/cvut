Dokumentové databáze stojí mezi jednoduchým key-value modelem a relačními databázemi. Stejně jako key-value databáze ukládají záznam pod identifikátorem, ale hodnota už není černá skříňka. Databáze rozumí vnitřní struktuře dokumentu, umí filtrovat podle polí, procházet vnořené objekty, dotazovat se do polí a vytvářet indexy nad konkrétními atributy.

MongoDB je v předmětu hlavní zástupce dokumentových databází. Je důležité ho nebrat jen jako "JSON v databázi". Podstatné je, že dokument se má navrhovat podle toho, jak aplikace data čte a mění. Někdy je správně data vnořit do jednoho dokumentu, protože se čtou společně. Jindy je správně použít reference, protože by vnoření vedlo k příliš velkým dokumentům, časté duplikaci nebo těžkým update operacím.

> **Dokumentová databáze:** NoSQL databáze, která ukládá záznamy jako strukturované dokumenty. Dokument může obsahovat skalární hodnoty, objekty, pole i další vnořené dokumenty.

# Obecné principy

Dokument v MongoDB je hierarchická datová struktura. Zvenku se podobá JSON objektu, ale interně se ukládá jako BSON. Dokument může obsahovat běžné hodnoty, vnořené objekty, pole, pole objektů a další typy, které čistý JSON nezná.

```javascript
{
  _id: ObjectId("64b000000000000000000001"),
  title: "Samotáři",
  year: 2000,
  director: { firstname: "David", lastname: "Ondříček" },
  actors: [
    { firstname: "Ivan", lastname: "Trojan" },
    { firstname: "Jiří", lastname: "Macháček" }
  ],
  genres: ["comedy", "drama"]
}
```

> **JSON:** Textový formát pro zápis objektů, polí a jednoduchých hodnot. Je dobře čitelný pro člověka i stroj, ale má menší množinu typů než BSON.
>
> **BSON:** Binary JSON. Binární formát používaný MongoDB pro ukládání dokumentů. Přidává typy jako `ObjectId`, `Date`, binární data nebo přesnější číselné typy.

Hierarchie v MongoDB vypadá takto:

```text
instance -> databases -> collections -> documents
```

**Database** je logický kontejner. **Collection** je kolekce dokumentů, typicky podobné struktury. MongoDB sice nevyžaduje pevné schéma jako relační databáze, ale v praxi nějaké implicitní schéma stejně existuje: aplikace očekává, že film má název, rok, herce a podobně. Bez takové disciplíny by se kolekce rychle změnila na nečitelnou skládku náhodných tvarů dokumentů.

> **Implicitní schéma:** Schéma, které databáze striktně nevynucuje, ale aplikace s ním počítá. Například kód očekává, že pole `year` je číslo a `actors` je pole.
>
> **Schema-on-read:** Přístup, kdy se struktura dat interpretuje až při čtení. Je flexibilní, ale analytika a validace musí s tvarem dat počítat pokaždé znovu.

MongoDB podporuje i JSON schema validation, tedy pravidla pro validaci dokumentů. To je užitečné, když chceme zachovat flexibilitu dokumentové databáze, ale zároveň nechceme dovolit úplně libovolný obsah.

> **Validace schématu:** Kontrola, zda dokument splňuje očekávanou strukturu, typy a povinná pole.

## Dokumenty, kolekce a identifikátory

Každý dokument v MongoDB má primární klíč v poli `_id`. Toto pole je povinné, unikátní v rámci kolekce a po přiřazení se nemá měnit. Pokud `_id` při vložení neuvedeme, MongoDB ho automaticky vygeneruje jako `ObjectId`.

> **Primární klíč:** Hodnota, která jednoznačně identifikuje záznam. V MongoDB je primárním klíčem pole `_id`.

`ObjectId` je 12bajtový BSON typ. Skládá se z timestampu, identifikace stroje, identifikace procesu a lokálního čítače. Díky tomu je malý, rychle generovatelný, prakticky unikátní a přibližně časově seřaditelný. Přednáška výslovně nedoporučuje používat jednoduché inkrementální `_id` typu `1`, `2`, `3`, protože v distribuovaném prostředí vytváří centrální závislost a hůře se škáluje.

```javascript
db.actors.insertOne({
  firstname: "Anna",
  lastname: "Geislerová"
})
```

MongoDB pak dokument uloží zhruba jako:

```javascript
{
  _id: ObjectId("..."),
  firstname: "Anna",
  lastname: "Geislerová"
}
```

> **ObjectId:** Výchozí typ identifikátoru v MongoDB. Má 12 bajtů a obsahuje mimo jiné čas vytvoření a lokální čítač.
>
> **UUID:** Univerzálně unikátní identifikátor. Lze ho použít místo `ObjectId`, když chce aplikace generovat identifikátory sama a nechce záviset na databázi.

Dokument má velikostní limit 16 MB. Pokud potřebujeme ukládat větší soubory, používá se GridFS, které soubor rozdělí na menší části uložené v kolekcích.

> **GridFS:** Mechanismus MongoDB pro ukládání souborů větších než limit jednoho dokumentu. Soubor se rozdělí na chunks a uloží se jako více dokumentů.
>
> **Chunk:** Menší část většího objektu. V MongoDB se pojem používá jak u GridFS, tak u shardingu.

MongoDB má také omezení pro názvy polí. Top-level pole `_id` je rezervované pro primární klíč. Názvy polí by neměly začínat znakem `$` a neměly by obsahovat tečku, protože `$` se používá pro operátory a tečka pro přístup k vnořeným polím. Pořadí polí v dokumentu se zachovává, s výjimkou `_id`, které se přesouvá na začátek. Názvy polí v jednom dokumentu musí být unikátní.

## Vnořené dokumenty vs. reference

Nejdůležitější návrhová otázka v dokumentových databázích zní: má být související informace vložená dovnitř dokumentu, nebo má být v samostatném dokumentu a propojená referencí?

**Vnořený model** ukládá související data přímo do jednoho dokumentu:

```javascript
{
  _id: ObjectId("..."),
  title: "Samotáři",
  year: 2000,
  actors: [
    { firstname: "Jitka", lastname: "Schneiderová" },
    { firstname: "Ivan", lastname: "Trojan" },
    { firstname: "Jiří", lastname: "Macháček" }
  ]
}
```

Výhoda je jednoduché čtení. Pokud stránka filmu vždy zobrazuje i seznam herců, stačí jeden dotaz a jeden dokument. Nevýhoda je duplicita. Když stejný herec hraje ve stovkách filmů a změníme mu jméno, musíme upravit mnoho dokumentů.

> **Vnořený dokument:** Dokument uložený uvnitř jiného dokumentu. V MongoDB je to běžný způsob modelování přirozených agregátů.
>
> **Agregát:** Skupina dat, která dává smysl číst a měnit jako jeden celek. Například objednávka a její položky jsou často jeden agregát.

**Referenční model** uloží související entity zvlášť a propojí je identifikátory:

```javascript
{
  _id: ObjectId("movie2"),
  title: "Samotáři",
  year: 2000,
  actors: [ObjectId("actor6"), ObjectId("actor4"), ObjectId("actor5")]
}
```

Herci jsou v jiné kolekci:

```javascript
{
  _id: ObjectId("actor4"),
  firstname: "Ivan",
  lastname: "Trojan"
}
```

Výhoda je menší duplicita a vyšší flexibilita u vztahů many-to-many. Nevýhoda je, že aplikace musí udělat další dotazy nebo použít agregaci s `$lookup`.

> **Reference:** Uložení identifikátoru jiného dokumentu. MongoDB pro běžné reference nemá speciální relační konstrukci; technicky je to obyčejná hodnota, se kterou pracuje aplikace nebo agregace.
>
> **Many-to-many vztah:** Vztah, kde jeden záznam může souviset s mnoha jinými a naopak. Například film má mnoho herců a herec hraje v mnoha filmech.

Pravidlo pro státnice: vnořujeme, když se data čtou společně, přirozeně patří k sobě, mají omezenou velikost a změny uvnitř dokumentu nejsou drahé. Referencujeme, když je vztah mnoho-na-mnoho, data často měníme samostatně, dokument by rostl bez limitu nebo by duplicita byla příliš nákladná.

## MongoDB jako distribuovaný systém

MongoDB podporuje automatický sharding, primary-secondary replikaci, automatický failover, secondary indexes a eventual consistency podle konfigurace. Tím se dostává z úrovně "datový formát dokumentu" na úroveň distribuované databáze.

> **Failover:** Automatické převzetí role při výpadku uzlu. Například když v replica setu selže primary, ostatní uzly zvolí nový primary.

U shardingu se data rozdělují podle **shard key** mezi více shardů. Sharded cluster má tři důležité komponenty: **shards**, které drží data; **config servers**, které drží metadata clusteru; a **mongos**, což je router mezi klientem a shardy.

> **Shard key:** Pole nebo kombinace polí, podle kterých MongoDB rozděluje dokumenty mezi shardy.
>
> **Mongos:** Router proces v MongoDB sharded clusteru. Klient posílá dotazy na `mongos`, ten je směruje na správné shardy.
>
> **Config server:** Server uchovávající metadata sharded clusteru, například mapování chunků na shardy. V produkci běží jako replica set.

MongoDB používá několik sharding strategií. **Range-based sharding** rozděluje data podle rozsahů shard key. Hodí se pro range dotazy, časové řady nebo geografická data, ale může vytvářet hotspoty, pokud hodnota monotonně roste. **Hash-based sharding** počítá hash shard key a rozděluje data rovnoměrněji; hodí se pro write-heavy workload, ale zhoršuje range dotazy. **Zone-based sharding** dovoluje řídit umístění dat podle zón, například kvůli regionům, legislativě nebo různému hardwaru.

> **Monotonně rostoucí hodnota:** Hodnota, která se stále zvyšuje, například čas nebo inkrementální číslo. U range shardingu může způsobit, že nové zápisy padají pořád na stejný shard.
>
> **Zone-based sharding:** Sharding, kde určité rozsahy dat připíšeme konkrétním zónám a zóny konkrétním shardům. Hodí se pro regionální nebo regulační požadavky.

Replikace v MongoDB používá **replica set** s primary-secondary architekturou. Primary přijímá zápisy a zapisuje operace do oplogu. Secondary uzly kopírují data z primary a mohou sloužit pro čtení, pokud to konfigurace dovolí. Když primary vypadne, replica set provede volbu nového primary.

> **Replica set:** Skupina MongoDB uzlů držících stejná data. Jeden uzel je primary, ostatní jsou secondary.
>
> **Oplog:** Operation log. Záznam operací na primary, podle kterého se secondary uzly synchronizují.

Z pohledu CAP není MongoDB jednoznačně jen CP nebo AP. Chování závisí na nastavení. Čtení z primary s `writeConcern: "majority"` a `readConcern: "majority"` se chová více CP-like: raději obětuje dostupnost než aby četlo nepotvrzená data. Čtení ze secondary nebo slabší write/read concern se chová více AP-like: dostupnost a latence mohou být lepší, ale čtení může být starší.

> **Read concern:** Nastavení, jakou úroveň potvrzení musí mít data, aby je MongoDB vrátilo při čtení.
>
> **Write concern:** Nastavení, kolik uzlů musí potvrdit zápis, aby ho klient považoval za úspěšný.
>
> **Read preference:** Nastavení, zda klient preferuje čtení z primary, secondary nebo jiného typu uzlu.

# Dotazování a CRUD operace

MongoDB query language je založený na JavaScriptovém zápisu objektů. Jeden příkaz může být jednoduchý dotaz, ale v `mongosh` lze psát i delší skripty. Výsledkem `find()` není okamžitě pole všech dokumentů, ale cursor, přes který se iteruje.

> **Cursor:** Objekt reprezentující výsledek dotazu. Dokumenty se z něj načítají postupně, což je důležité u velkých výsledků.
>
> **mongosh:** Interaktivní shell pro práci s MongoDB.

Základní CRUD operace jsou:

```text
Create -> insertOne(), insertMany()
Read   -> find(), findOne()
Update -> replaceOne(), updateOne(), updateMany()
Delete -> deleteOne(), deleteMany()
```

Kolekce vznikají automaticky při prvním použití. Ke kolekci se dá přistupovat například přes `db.movies`, přes `db.getCollection("movies")` nebo uložením reference do proměnné.

## Insert

`insertOne()` vloží jeden dokument, `insertMany()` vloží více dokumentů:

```javascript
db.actors.insertOne({
  firstname: "Anna",
  lastname: "Geislerová"
})
```

```javascript
db.movies.insertMany([
  {
    _id: ObjectId("64b000000000000000000009"),
    title: "Želary",
    year: 2003,
    actors: [ObjectId("64b000000000000000000004")]
  },
  {
    title: "Anthropoid",
    year: 2016
  }
])
```

Pokud dokument obsahuje `_id`, musí být v kolekci unikátní. Pokud `_id` chybí, MongoDB ho vygeneruje. U `insertMany()` je důležitá volba `ordered`. Výchozí `ordered: true` znamená, že se dokumenty vkládají v pořadí a první chyba zastaví zbytek dávky. `ordered: false` dovolí pokračovat i po chybě a umožňuje efektivnější paralelní zpracování.

> **Dávka (batch):** Skupina více operací poslaných najednou. U `insertMany()` je dávkou pole dokumentů.

`writeConcern` určuje, jak silné potvrzení zápisu požadujeme. Při slabším nastavení může být zápis rychlejší, ale méně bezpečný při výpadcích. Při `majority` se čeká na potvrzení většinou uzlů replica setu.

## Update: replace, update, multi a upsert

MongoDB rozlišuje nahrazení dokumentu a částečnou aktualizaci. `replaceOne()` nahradí celý dokument novým dokumentem, ale zachová `_id`:

```javascript
db.actors.replaceOne(
  { _id: ObjectId("64b000000000000000000008") },
  { firstname: "Aňa", lastname: "Geislerová" }
)
```

Naproti tomu `updateOne()` a `updateMany()` mění jen části dokumentu pomocí update operátorů:

```javascript
db.movies.updateMany(
  { year: { $gt: 2015 } },
  {
    $set: { new: true },
    $inc: { rating: 3 }
  }
)
```

> **Replace:** Režim aktualizace, kdy se nahradí celý dokument kromě `_id`.
>
> **Update operátor:** Operátor jako `$set`, `$unset` nebo `$inc`, který říká, jak přesně změnit konkrétní pole dokumentu.

`updateOne()` změní první odpovídající dokument, `updateMany()` změní všechny odpovídající dokumenty. Starší materiály někdy mluví o příznaku **multi**; v moderním API je tato volba rozdělená do metod `updateOne()` a `updateMany()`.

> **Multi update:** Aktualizace více dokumentů odpovídajících podmínce. V moderním MongoDB API ji reprezentuje hlavně `updateMany()`.

Často používané field operátory:

```javascript
db.users.updateOne(
  { username: "jirka" },
  {
    $set: { active: true },
    $unset: { temporaryToken: "" },
    $inc: { loginCount: 1 },
    $currentDate: { lastLogin: true }
  }
)
```

`$set` nastaví nebo vytvoří pole. `$unset` pole odstraní. `$rename` přejmenuje pole. `$inc` inkrementuje číslo, `$mul` ho vynásobí a `$currentDate` uloží aktuální datum nebo timestamp.

> **Inkrementace:** Zvýšení číselné hodnoty o daný přírůstek, například `rating += 3`.

Pro pole existují array operátory. `$push` přidá prvek na konec pole, `$addToSet` přidá prvek jen tehdy, pokud tam ještě není, `$pop` odstraní první nebo poslední prvek a `$pull` odstraní prvky odpovídající podmínce.

```javascript
db.movies.updateOne(
  { title: "Samotáři" },
  {
    $push: { tags: "czech" },
    $addToSet: { genres: "comedy" },
    $pull: { ratings: { $lt: 2 } }
  }
)
```

`upsert` znamená "update or insert". Pokud dotaz najde dokument, provede se update. Pokud žádný dokument nenajde a je zapnuté `{ upsert: true }`, MongoDB vytvoří nový dokument.

```javascript
db.movies.updateOne(
  { title: "Tmavomodrý svět", year: { $gt: 2000 } },
  {
    $set: {
      director: { firstname: "Jan", lastname: "Svěrák" },
      year: 2001
    },
    $inc: { rating: 2 }
  },
  { upsert: true }
)
```

V update režimu nový dokument vznikne z hodnot v query, které jsou obyčejné rovnosti, a z výsledku update operátorů. U složitějších podmínek jako `{ year: { $gt: 2000 } }` nelze hodnotu přímo převzít, takže se použije hodnota z `$set`.

> **Upsert:** Operace, která aktualizuje existující dokument, nebo vloží nový, pokud žádný dokument neodpovídá dotazu.

## Delete

`deleteOne()` smaže první dokument odpovídající dotazu. `deleteMany()` smaže všechny odpovídající dokumenty:

```javascript
db.movies.deleteOne({ _id: ObjectId("64b000000000000000000002") })
```

```javascript
db.movies.deleteMany({ year: { $lt: 1950 } })
```

Volby delete operací mohou obsahovat například pravidla pro porovnávání řetězců, tedy collation.

> **Collation:** Pravidla pro porovnávání textu, například citlivost na velikost písmen nebo diakritiku.

## Find, projekce a modifikátory

`find()` vybírá dokumenty z kolekce. První parametr je query, druhý je projection:

```javascript
db.movies.find(
  { year: { $gte: 2005, $lte: 2015 } },
  { _id: 0, title: 1, year: 1 }
)
```

`findOne()` vrátí jeden dokument přímo, ne cursor. Pokud nic nenajde, vrací `null`. Pro nalezení jediného dokumentu je vhodnější než `find().limit(1)`.

Projection určuje, která pole se vrátí. Hodnota `1` nebo `true` pole zahrne, hodnota `0` nebo `false` ho vyloučí. Pozitivní a negativní projekce se nemají míchat; výjimkou je `_id`, které je standardně vrácené, ale můžeme ho vypnout.

```javascript
db.movies.find(
  { year: 2000 },
  { title: 1, _id: 0, actors: { $slice: -2 } }
)
```

> **Projekce:** Výběr polí, která se mají vrátit ve výsledku dotazu. Pomáhá omezit přenesená data a zlepšit čitelnost výsledku.

Na cursor lze řetězit modifikátory `sort()`, `skip()` a `limit()`:

```javascript
db.movies
  .find({ year: { $gt: 2005 } }, { _id: 0, title: 1 })
  .sort({ title: -1 })
  .skip(10)
  .limit(20)
```

Pořadí zápisu v řetězci nemusí být totéž jako logické provedení. Server aplikuje fáze v pořadí `sort -> skip -> limit -> projection`. Bez explicitního `sort()` není pořadí dokumentů definované. Řazení nad větším množstvím dat by mělo být podpořené indexem.

> **Pagination:** Stránkování výsledků. `skip()` a `limit()` jsou jednoduché, ale velké `skip()` může být drahé; pro velké datasety je lepší key-set pagination podle poslední viděné hodnoty.

# Operátory

MongoDB dotazy jsou dokumenty popisující podmínky. Nejběžnější tvar je rovnost:

```javascript
db.movies.find({ year: 2000 })
```

U skalárních hodnot je to intuitivní. U vnořených dokumentů a polí je potřeba být opatrný. Rovnost celého objektu vyžaduje shodu všech vnořených polí včetně jejich názvů, hodnot a pořadí. Tyto dva dotazy proto nejsou ekvivalentní:

```javascript
db.movies.find({ director: { firstname: "Jan", lastname: "Svěrák" } })
db.movies.find({ director: { lastname: "Svěrák", firstname: "Jan" } })
```

Podobně u pole rozhoduje počet a pořadí prvků:

```javascript
db.movies.find({ actors: [ObjectId("a7"), ObjectId("a5")] })
db.movies.find({ actors: [ObjectId("a5"), ObjectId("a7")] })
```

> **Value equality:** Porovnání celé hodnoty. U objektů a polí v MongoDB zahrnuje i vnitřní strukturu a pořadí.

## Dot notation

K vnořeným polím se přistupuje pomocí tečkové notace:

```javascript
db.movies.find({
  "director.firstname": "Jan",
  "director.lastname": "Svěrák"
})
```

Tečková notace funguje i pro pozice v poli:

```javascript
db.movies.find({ "actors.2": ObjectId("64b000000000000000000005") })
```

Pozice polí začínají od nuly. MongoDB dovoluje vnoření až do hluboké úrovně, ale prakticky platí: když dotazy potřebují sahat příliš hluboko a často, je dobré znovu promyslet model dokumentu.

> **Dot notation:** Zápis pro přístup k vnořeným polím, například `"address.city"` nebo `"actors.0"`.

## Porovnávací operátory

Porovnávací operátory se zapisují pod názvem pole:

```javascript
db.movies.find({
  year: { $gte: 2005, $lte: 2015 }
})
```

Základní operátory jsou:

```text
$eq   rovnost
$ne   nerovnost
$lt   menší než
$lte  menší nebo rovno
$gt   větší než
$gte  větší nebo rovno
$in   rovno alespoň jedné hodnotě ze seznamu
$nin  není rovno žádné hodnotě ze seznamu
```

MongoDB při porovnávání bere v úvahu BSON typy. Některé číselné konverze se provedou automaticky, ale obecně je dobré držet typy v kolekci konzistentní. Když je `year` někdy číslo a někdy string, dotazy a indexy se chovají hůř.

> **BSON typ:** Konkrétní datový typ uložený v BSON dokumentu, například string, int, double, date nebo ObjectId.

## Element, evaluation a textové operátory

`$exists` testuje, zda pole existuje:

```javascript
db.movies.find({ rating: { $exists: true } })
```

Je rozdíl mezi polem, které neexistuje, a polem, které existuje s hodnotou `null`. To je častá past u dokumentových databází, protože nevyplněné atributy mohou mít oba tvary.

> **Chybějící pole vs. null:** Chybějící pole v dokumentu vůbec není. `null` je explicitně uložená hodnota. Dotazy na existenci a rovnost je mohou rozlišovat.

`$regex` testuje řetězec regulárním výrazem:

```javascript
db.movies.find({
  "director.lastname": { $regex: /^svěrák$/i }
})
```

`$text` provádí full-textové hledání, ale vyžaduje textový index. MongoDB podporuje i wildcard indexy, které mohou indexovat širší množinu polí, například u méně stabilních dokumentů.

> **Regulární výraz:** Vzor pro porovnávání textu, například "začíná na", "končí na" nebo "obsahuje určitou kombinaci znaků".
>
> **Full-text search:** Vyhledávání v textu podle slov, často s podporou relevance. V MongoDB vyžaduje textový index.
>
> **Wildcard index:** Index definovaný nad proměnlivou množinou polí, typicky pomocí `$**`. Hodí se opatrně u dokumentů s méně stabilní strukturou.

## Logické operátory

`$and` a `$or` kombinují více dotazových výrazů:

```javascript
db.movies.find({
  $or: [
    { year: { $lt: 2000 } },
    { rating: { $gte: 4 } }
  ]
})
```

`$and` je často implicitní. Dotaz:

```javascript
db.movies.find({ year: 2000, rating: { $gt: 1 } })
```

znamená, že musí platit obě podmínky. `$not` neguje jeden query operator, ale nepoužívá se jako top-level negace celého dotazu:

```javascript
db.movies.find({ rating: { $not: { $gte: 4 } } })
```

> **Top-level operátor:** Operátor použitý přímo na nejvyšší úrovni query dokumentu, například `{ $or: [...] }`.

## Pole a array operátory

Dotaz na rovnost vůči poli má dvě různé tváře. Pokud pole porovnáváme s celým polem, musí sedět přesně obsah i pořadí. Pokud ale pole porovnáváme se skalární hodnotou, podmínka platí i tehdy, když alespoň jeden prvek pole odpovídá:

```javascript
db.movies.find({ actors: ObjectId("64b000000000000000000005") })
```

Tento dotaz najde dokument, kde `actors` je přímo daný ObjectId, i dokument, kde je daný ObjectId jedním prvkem pole `actors`.

`$all` ověřuje, že pole obsahuje všechny uvedené prvky, nezávisle na pořadí:

```javascript
db.movies.find({
  actors: {
    $all: [
      ObjectId("64b000000000000000000005"),
      ObjectId("64b000000000000000000007")
    ]
  }
})
```

`$size` testuje velikost pole proti pevnému číslu:

```javascript
db.movies.find({ actors: { $size: 3 } })
```

Neumí přímo rozsah typu "pole má aspoň 3 prvky". Pro takové dotazy je obvykle lepší držet pomocné pole s počtem, nebo použít agregaci.

Nejdůležitější operátor pro pole je `$elemMatch`. Bez něj může dotaz s více podmínkami splnit různé prvky pole:

```javascript
db.movies.find({ ratings: { $gte: 2, $lte: 3 } })
```

Tento dotaz může projít i tehdy, když jedna hodnota v poli splní `$gte: 2` a jiná hodnota splní `$lte: 3`. Pokud chceme jeden prvek, který splní obě podmínky, použijeme:

```javascript
db.movies.find({
  ratings: {
    $elemMatch: { $gte: 2, $lte: 3 }
  }
})
```

> **`$elemMatch`:** Operátor, který vyžaduje, aby jeden konkrétní prvek pole splnil všechny uvedené podmínky.

## Projekční operátory

V projekci lze kromě `1` a `0` použít operátory nad poli. `$slice` vrátí jen část pole:

```javascript
db.movies.find(
  { year: 2000 },
  { title: 1, _id: 0, actors: { $slice: -2 } }
)
```

Tady se vrátí název filmu a poslední dva herci. `$slice` může brát i dvojici `[skip, count]`.

Projekční `$elemMatch` vrátí první prvek pole, který odpovídá podmínce:

```javascript
db.movies.find(
  { title: "Samotáři" },
  { actors: { $elemMatch: { lastname: "Trojan" } } }
)
```

> **Projekční operátor:** Operátor použitý ve druhém parametru `find()`, tedy při výběru polí výsledku, ne při filtrování dokumentů.

# Aggregační framework

Aggregation framework slouží k výpočtům nad dokumenty: filtrování, seskupování, řazení, transformace, práce s poli, spojení s jinou kolekcí a zápis výsledků. MongoDB historicky nabízelo i MapReduce, ale ten je dnes deprecated. Hlavní způsob agregací je **aggregation pipeline**.

> **Deprecated:** Označení pro funkcionalitu, která stále může existovat, ale už se nedoporučuje používat a může být v budoucnu odstraněna.

Pipeline je posloupnost stages. Každá stage vezme dokumenty na vstupu, nějak je transformuje a pošle dál:

```javascript
db.orders.aggregate([
  { $match: { size: "medium" } },
  { $group: { _id: "$name", totalQuantity: { $sum: "$quantity" } } }
])
```

`$match` vyfiltruje dokumenty, `$group` je seskupí podle názvu a spočítá součet množství. Pořadí stages je důležité pro význam i výkon. Dobré pravidlo je filtrovat brzy a těžké operace dělat až po zmenšení dat.

> **Pipeline:** Řetězec kroků, kde výstup jednoho kroku je vstupem dalšího.
>
> **Stage:** Jeden krok aggregation pipeline, například `$match`, `$group` nebo `$sort`.

Aggregation pipeline nemění původní dokumenty, pokud nepoužije zapisovací stage jako `$out` nebo `$merge`. Výstupem agregace je cursor. Některé stages, například `$sort` a `$group`, mají paměťová omezení; při větších datech lze použít `allowDiskUse`, ale to je pojistka, ne náhrada za dobrý návrh pipeline a indexů.

> **Spill na disk:** Situace, kdy operace nevejde do paměti a začne používat disk. Je to pomalejší a mělo by se ověřovat pomocí `explain()`.

## Základní stages

`$match` filtruje dokumenty stejným jazykem jako `find()`:

```javascript
db.orders.aggregate([
  { $match: { size: "medium", price: { $gte: 15 } } }
])
```

Když je `$match` na začátku pipeline a odpovídá indexu, MongoDB může výrazně zmenšit množství zpracovaných dokumentů.

`$group` seskupuje dokumenty podle výrazu a počítá agregace:

```javascript
db.orders.aggregate([
  {
    $group: {
      _id: "$name",
      totalOffers: { $sum: 1 },
      maxQuantity: { $max: "$quantity" }
    }
  }
])
```

> **Akumulátor:** Výraz v `$group`, který z více dokumentů počítá jednu hodnotu, například `$sum`, `$avg`, `$min`, `$max`, `$first` nebo `$last`.

`$count` vytvoří dokument s počtem vstupních dokumentů:

```javascript
db.orders.aggregate([
  { $match: { name: "Pepperoni" } },
  { $count: "totalPepperoniOffers" }
])
```

`$sort`, `$limit` a `$skip` odpovídají řazení a stránkování:

```javascript
db.orders.aggregate([
  { $sort: { price: -1 } },
  { $limit: 3 }
])
```

Kombinace `$sort` a `$limit` je klasický Top-N vzor, například tři nejdražší položky.

> **Top-N dotaz:** Dotaz typu "najdi prvních N položek podle určitého pořadí", například 10 nejprodávanějších produktů.

## Transformace dokumentů

`$project` mění tvar dokumentu. Umí pole zahrnout, odstranit, přejmenovat nebo vytvořit výpočtem:

```javascript
db.orders.aggregate([
  {
    $project: {
      _id: 0,
      name: 1,
      revenue: { $multiply: ["$price", "$quantity"] }
    }
  }
])
```

`$addFields` přidá nová pole a ponechá původní dokument:

```javascript
db.orders.aggregate([
  {
    $addFields: {
      revenue: { $multiply: ["$price", "$quantity"] }
    }
  }
])
```

`$unset` odstraní pole. Je to v agregaci zkratka pro projekci, která dané pole vyloučí:

```javascript
db.users.aggregate([
  { $unset: ["password", "secretKey", "sessionInfo"] }
])
```

`$replaceRoot` a `$replaceWith` nahradí celý aktuální dokument novým objektem. Novější a čitelnější je `$replaceWith`:

```javascript
db.orders.aggregate([
  { $replaceWith: { name: "$name", size: "$size" } }
])
```

`$mergeObjects` spojí více dokumentů dohromady. Pokud se pole překrývají, pozdější hodnota přepíše dřívější:

```javascript
db.orders.aggregate([
  {
    $replaceWith: {
      $mergeObjects: ["$$ROOT", { status: "paid", source: "web" }]
    }
  }
])
```

> **`$$ROOT`:** Systémová proměnná v aggregation pipeline reprezentující celý aktuální dokument.

## Práce s poli: `$unwind`

`$unwind` rozbalí pole tak, že z jednoho dokumentu vytvoří více dokumentů, jeden pro každý prvek pole:

```javascript
db.orders.aggregate([
  { $unwind: "$assortment" },
  {
    $group: {
      _id: "$assortment.size",
      qty: { $sum: "$assortment.quantity" }
    }
  },
  { $sort: { qty: -1 } }
])
```

To je typický postup, když máme uvnitř dokumentu pole položek a chceme agregovat přes jednotlivé položky. Dokumenty, kde pole chybí nebo je prázdné, se běžně zahodí. Pokud je chceme zachovat, použijeme:

```javascript
db.orders.aggregate([
  {
    $unwind: {
      path: "$assortment",
      preserveNullAndEmptyArrays: true
    }
  }
])
```

> **`$unwind`:** Stage, která rozloží pole na více dokumentů. Hodí se, když chceme agregovat přes prvky pole místo přes celé dokumenty.

## Spojování kolekcí: `$lookup`

MongoDB dokumentový model často preferuje vnoření nebo denormalizaci, ale aggregation framework má i `$lookup`, který provádí left outer join s jinou kolekcí ve stejné databázi:

```javascript
db.orders.aggregate([
  {
    $lookup: {
      from: "customers",
      localField: "customerId",
      foreignField: "_id",
      as: "customerInfo"
    }
  }
])
```

Výsledek přidá ke každé objednávce pole `customerInfo` s odpovídajícími dokumenty zákazníků. Je to užitečné, ale není to důvod modelovat MongoDB jako relační databázi. Časté a velké `$lookup` mohou být drahé.

> **Left outer join:** Spojení, které zachová všechny dokumenty z levé strany a přidá odpovídající dokumenty z pravé strany, pokud existují.

Pokročilejší forma `$lookup` používá vnořenou pipeline a `$expr`:

```javascript
db.orders.aggregate([
  {
    $lookup: {
      from: "customers",
      let: { cid: "$customerId" },
      pipeline: [
        { $match: { $expr: { $eq: ["$_id", "$$cid"] } } },
        { $project: { _id: 1, name: 1 } }
      ],
      as: "customer"
    }
  }
])
```

`$expr` dovoluje v podmínkách používat výrazy a porovnávat pole s polem, nejen pole s konstantou.

> **`$expr`:** Operátor, který dovoluje používat agregační výrazy uvnitř dotazu, například porovnat dvě pole jednoho dokumentu.

## Zapisovací stages a výkon agregací

`$out` zapíše výsledek pipeline do kolekce a cílovou kolekci nahradí. `$merge` výsledek zapíše nebo sloučí s existující kolekcí podle identifikátoru:

```javascript
db.orders.aggregate([
  { $match: { status: "paid" } },
  { $out: "paidOrdersSnapshot" }
])
```

```javascript
db.orders.aggregate([
  { $group: { _id: "$customerId", total: { $sum: "$price" } } },
  {
    $merge: {
      into: "customerTotals",
      on: "_id",
      whenMatched: "merge",
      whenNotMatched: "insert"
    }
  }
])
```

`$out` a `$merge` musí být poslední stages, protože zapisují výsledek. `$geoNear` musí být naopak první stage, pokud se použije.

Výkonová pravidla z přednášky jsou praktická: dát `$match` co nejdřív, zúžit dokumenty pomocí `$project`, těžké stages jako `$group`, `$sort`, `$lookup`, `$facet` a window funkce dělat až po redukci dat, řadit podle indexu, používat `$limit` brzy po selektivním sortu, nerozbalovat pole zbytečně, optimalizovat `$lookup` filtrováním a projekcí, velké `skip` nahradit key-set pagination a všechno ověřovat pomocí `explain()`.

> **`explain()`:** Nástroj, který ukáže plán dotazu nebo agregace: zda se použil index, kolik dokumentů se skenovalo a kde vznikaly náklady.
>
> **Window funkce:** Výpočty nad "oknem" okolních řádků nebo dokumentů, například průběžné součty. V MongoDB patří mezi pokročilejší agregační operace.

Užitečné mapování na SQL:

```text
WHERE       -> $match
GROUP BY    -> $group
HAVING      -> $match za $group
SELECT      -> $project
ORDER BY    -> $sort
LIMIT       -> $limit
JOIN        -> $lookup
SELECT INTO -> $out
MERGE INTO  -> $merge
UNION ALL   -> $unionWith
```

# Indexování

Index zrychluje dotazy tím, že databáze nemusí procházet všechny dokumenty. MongoDB používá B-Tree indexy a umí vytvořit index nad libovolným polem dokumentu, včetně vnořených polí a polí uvnitř arrays.

> **B-Tree:** Stromová datová struktura používaná pro indexy. Umožňuje efektivní vyhledávání, rozsahové dotazy a řazení.

Index vytvoříme příkazem `createIndex()`:

```javascript
db.orders.createIndex({ name: 1, size: -1 })
```

Hodnota `1` znamená vzestupné pořadí, `-1` sestupné. U některých typů indexů se místo směru používá textový typ:

```javascript
db.articles.createIndex({ title: "text", body: "text" })
db.users.createIndex({ userId: "hashed" })
db.places.createIndex({ location: "2dsphere" })
```

Index může mít vlastní jméno:

```javascript
db.orders.createIndex(
  { name: 1, size: -1 },
  { name: "query_for_size" }
)
```

Seznam indexů ukáže:

```javascript
db.orders.getIndexes()
```

## Typy indexů

**Single field index** indexuje jedno pole. Hodí se pro časté filtrování nebo řazení podle jednoho atributu:

```javascript
db.movies.createIndex({ year: 1 })
```

**Compound index** indexuje více polí v jednom pořadí:

```javascript
db.movies.createIndex({ year: 1, title: 1 })
```

Pořadí polí je zásadní. Index `{ year: 1, title: 1 }` není totéž jako `{ title: 1, year: 1 }`.

> **Compound index:** Složený index nad více poli. Pomáhá dotazům, které filtrují nebo řadí podle kombinace atributů.

**Multikey index** vzniká nad polem, které obsahuje array. MongoDB indexuje jednotlivé prvky pole, takže lze efektivně hledat dokumenty obsahující konkrétní hodnotu:

```javascript
db.movies.createIndex({ actors: 1 })
```

> **Multikey index:** Index nad polem typu array. Jeden dokument může do indexu přispět více položkami, jednu pro každý prvek pole.

**Hashed index** indexuje hash hodnoty. Hodí se zejména pro rovnoměrné rozdělení dat u hashed shardingu, ale nepodporuje efektivní rozsahové dotazy.

**Text index** podporuje `$text` vyhledávání. **Geospatial index** jako `2d` nebo `2dsphere` slouží pro geografické dotazy. **Clustered index** ovlivňuje fyzické uspořádání dokumentů v kolekci a je specializovanější.

> **Geospatial index:** Index pro prostorová data, například body na mapě nebo geografické oblasti.
>
> **Clustered index:** Index, podle kterého jsou data fyzicky organizovaná. Nejde jen o pomocnou strukturu vedle dat.

## Vlastnosti indexů

`unique` zajistí, že MongoDB nedovolí dvěma dokumentům stejnou indexovanou hodnotu:

```javascript
db.users.createIndex({ email: 1 }, { unique: true })
```

> **Unique index:** Index, který zároveň vynucuje unikátnost hodnoty.

`partial` indexuje jen dokumenty splňující filtr:

```javascript
db.users.createIndex(
  { email: 1 },
  { partialFilterExpression: { active: true } }
)
```

To se hodí, když dotazy často pracují jen s aktivní podmnožinou dat.

> **Partial index:** Index obsahující jen dokumenty, které splňují danou podmínku.

`sparse` vynechá dokumenty, které indexované pole vůbec nemají:

```javascript
db.users.createIndex({ phone: 1 }, { sparse: true })
```

> **Sparse index:** Index, který ignoruje dokumenty bez indexovaného pole. Pozor na rozdíl mezi chybějícím polem a hodnotou `null`.

`TTL` index automaticky odstraňuje dokumenty po uplynutí času:

```javascript
db.sessions.createIndex(
  { createdAt: 1 },
  { expireAfterSeconds: 3600 }
)
```

> **TTL index:** Index, podle kterého MongoDB automaticky maže dokumenty po vypršení časového limitu. Hodí se pro session data, tokeny nebo dočasné záznamy.

Ne všechny kombinace typů, forem a vlastností indexů dávají smysl nebo jsou podporované. Vždy je potřeba ověřit konkrétní kombinaci v dokumentaci a hlavně v plánu dotazu.

## Pravidlo ESR a covered queries

U složených indexů se v přednášce zdůrazňuje pravidlo **ESR**: Equality, Sort, Range. Nejprve do indexu patří pole používaná v rovnostních podmínkách, potom pole používaná pro řazení a nakonec pole používaná pro rozsah.

```javascript
db.movies.find({
  genre: "drama",
  year: { $gte: 2000 }
}).sort({ title: 1 })
```

Pro takový dotaz může dávat smysl index:

```javascript
db.movies.createIndex({ genre: 1, title: 1, year: 1 })
```

`genre` je equality, `title` je sort a `year` je range. Skutečný nejlepší index ale závisí na selektivitě dat a konkrétním dotazu, proto se musí ověřovat přes `explain()`.

> **ESR pravidlo:** Doporučení pro pořadí polí ve složeném indexu: nejdřív rovnosti, potom řazení, nakonec rozsahy.
>
> **Selektivita:** Jak moc podmínka zužuje výsledek. Vysoce selektivní podmínka vrací malou část dokumentů.

**Covered query** je dotaz, který MongoDB dokáže obsloužit přímo z indexu bez načtení zdrojových dokumentů. To je rychlé, ale vyžaduje, aby všechna filtrovaná i vracená pole byla v indexu a aby projekce nevyžadovala nic mimo něj.

> **Covered query:** Dotaz pokrytý indexem. Databáze vrátí výsledek jen z indexu, bez čtení samotných dokumentů.

## Kdy indexy škodí

Index není zadarmo. Zrychluje čtení, ale zpomaluje zápisy, protože každý insert, update nebo delete musí udržovat i indexové struktury. Příliš mnoho indexů znamená vyšší nároky na paměť, větší diskový prostor a pomalejší zápisy.

> **Over-indexing:** Stav, kdy má kolekce příliš mnoho indexů. Dotazy možná nejsou rychlejší, ale zápisy a údržba jsou dražší.

Špatný index je také index nad polem, které se skoro nikdy nepoužívá v dotazech, nebo index, který se nevejde do working setu. Working set je část dat a indexů, se kterými databáze aktivně pracuje a ideálně je drží v RAM. Pokud indexy neustále padají na disk, výkon prudce klesá.

> **Working set:** Aktivní část dat a indexů, ke které databáze často přistupuje. Když se nevejde do RAM, dotazy častěji čekají na disk.

Praktické shrnutí: indexy navrhujeme podle skutečných dotazů, ne podle všech možných polí. U MongoDB platí stejná filozofie jako u ostatních NoSQL databází v předmětu: nejdřív znát přístupové vzory, potom navrhnout dokumenty, sharding key a indexy.

## Jak si MongoDB pamatovat ke státnicím

MongoDB je dokumentová databáze pro BSON dokumenty organizované v kolekcích. Je silná tam, kde data tvoří přirozené agregáty, mají flexibilní strukturu a aplikace potřebuje dotazovat podle polí uvnitř dokumentu. Oproti key-value databázi vidí dovnitř hodnoty. Oproti relační databázi ale nevede návrh přes normalizaci a joiny, ale přes dokumenty, vnoření, reference a indexy podle dotazů.

U zkoušky je dobré držet si v hlavě tři vrstvy. První je datový model: JSON/BSON, `_id`, kolekce, vnoření vs. reference. Druhá je query jazyk: CRUD, dotazové operátory, pole, projekce a cursor modifikátory. Třetí je výkon a distribuce: aggregation pipeline, indexy, ESR, sharding, replica set a read/write concern. Když tyto tři vrstvy spojíme, MongoDB přestane být jen "databáze na JSON" a začne dávat smysl jako konkrétní NoSQL systém pro čtení a zápis dokumentových agregátů ve škálovatelném prostředí.
