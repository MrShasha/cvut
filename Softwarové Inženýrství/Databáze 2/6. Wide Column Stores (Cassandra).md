- **[[6. Wide Column Stores (Cassandra)#Obecné principy|Obecné principy]]:** datový model (keyspace, table/column family, row, column), návrh dle dotazů (partition key + clustering columns).
- **[[6. Wide Column Stores (Cassandra)#Datové typy a hodnoty|Datové typy a hodnoty]]:** primitivní typy, kolekce (list, set, map), tuples, user-defined types (UDT), frozen mode, tombstones, TTL, timestamp.
- **[[6. Wide Column Stores (Cassandra)#Cassandra Query Language (CQL)|Cassandra Query Language (CQL)]]:**
    - DDL: CREATE/DROP KEYSPACE, USE, CREATE/DROP/TRUNCATE TABLE.
    - DML: SELECT (FROM, WHERE, GROUP BY, ORDER BY, LIMIT, DISTINCT, ALLOW FILTERING), INSERT, UPDATE, DELETE.

Wide column databáze patří do rodiny NoSQL systémů, které se hodí pro velká distribuovaná data s vysokým objemem zápisů a předem známými přístupovými vzory. V předmětu je jejich hlavním zástupcem **Apache Cassandra**. Není to "relační databáze s proměnlivým počtem sloupců". Má jiný způsob přemýšlení: tabulky se navrhují podle dotazů, data se záměrně denormalizují a primární klíč neurčuje jen identitu řádku, ale i fyzické rozložení dat v clusteru.

Cassandra je dobrý příklad databáze, u které návrh schématu nejde oddělit od distribuovaného systému pod ní. Když špatně zvolíme partition key, vzniknou hotspoty. Když špatně zvolíme clustering columns, nepůjde efektivně řadit ani filtrovat. Když nepochopíme consistency level, můžeme číst starší data nebo zbytečně platit latencí. To je přesně důvod, proč se u Cassandry ve státnicové otázce musí spojit datový model, architektura a CQL dohromady.

> **Přístupový vzor:** Typický způsob, jakým aplikace čte nebo zapisuje data. Například "najdi posledních 100 měření senzoru za daný den" je přístupový vzor. V Cassandře podle něj často vzniká konkrétní tabulka.

# Obecné principy

Wide column store ukládá data do tabulek, ale tabulka zde neznamená totéž jako v relační databázi. Řádky v jedné tabulce mají být podobné, ale nemusí mít všechny stejné vyplněné sloupce. Sloupce mohou být řídké a některé řádky mohou obsahovat jen malou část možných hodnot. Důležité je, že jeden řádek má sdružovat data, která se typicky čtou společně.

> **Řídká data:** Data, kde většina záznamů nemá vyplněnou většinu možných atributů. Wide-column model s nimi umí pracovat přirozeněji než pevná relační tabulka plná `NULL` hodnot.

Základní hierarchie v Cassandře je:

```text
instance -> keyspaces -> tables -> rows -> columns
```

**Keyspace** je nejvyšší logický kontejner, přibližně podobný databázi nebo schématu v relačním světě. V keyspace se nastavuje replikace, tedy jak a kam se mají ukládat kopie dat. **Table**, dříve často nazývaná **column family**, je kolekce řádků. **Row** je řádek identifikovaný primárním klíčem. **Column** je dvojice název-hodnota s dodatečnými metadaty, například časem zápisu nebo TTL.

> **Column family:** Starší označení pro tabulku ve wide-column databázích. Vyjadřuje, že tabulka sdružuje podobné řádky a jejich sloupce, ale není to relační tabulka s důrazem na joiny a normalizaci.
>
> **Metadata:** Data o datech. U sloupce v Cassandře to může být například čas posledního zápisu nebo informace, kdy má hodnota expirovat.

## Cassandra jako distribuovaná databáze

Apache Cassandra je open-source wide-column databáze původně vyvinutá ve Facebooku a později spravovaná Apache Software Foundation. Je implementovaná v Javě a historicky vznikla pro systémy, které potřebují vysokou dostupnost, lineární horizontální škálování a provoz přes více datacenter.

Její architektura je **decentralizovaná peer-to-peer**. To znamená, že v clusteru není jeden centrální master, který by řídil všechny ostatní uzly. Každý uzel může přijmout požadavek od klienta a stát se pro tento požadavek koordinátorem. Koordinátor podle partition key zjistí, které uzly mají daná data držet, odešle jim čtení nebo zápis a počká na tolik odpovědí, kolik vyžaduje zvolená konzistence.

> **Peer-to-peer architektura:** Architektura, kde uzly mají stejnou roli a nejsou trvale rozdělené na master a slave. Konkrétní uzel může být pro jeden požadavek koordinátorem, ale není centrálním bodem celého systému.
>
> **Koordinátor:** Uzel Cassandry, který přijme klientský požadavek, zjistí odpovědné repliky a shromáždí jejich odpovědi. Koordinátor nemusí být uzel, který data sám ukládá.

Uzly si mezi sebou periodicky vyměňují informace pomocí **gossip protokolu**. Díky tomu cluster bez centrálního správce ví, které uzly jsou živé, které jsou pomalé a jaký je stav ostatních členů. Data jsou logicky rozmístěna do **ringu** podle tokenů. Partition key se zahashuje, hash určí pozici v tokenovém prostoru a podle ní se rozhodne, na který uzel data patří.

> **Gossip protokol:** Mechanismus, kde si uzly průběžně posílají malé zprávy o svém stavu a stavu ostatních uzlů. Název odpovídá principu "šíření klepů": informace se postupně rozšíří celým clusterem.
>
> **Token:** Číselná hodnota v hashovacím prostoru Cassandry. Uzly vlastní rozsahy tokenů a podle hashe partition key se rozhoduje, do kterého rozsahu záznam spadne.

Cassandra používá **consistent hashing** a často také **virtual nodes**. Místo aby jeden fyzický server vlastnil jeden souvislý rozsah tokenů, může vlastnit mnoho menších rozsahů. To usnadňuje přidávání a odebírání uzlů, protože při změně clusteru se přesouvají menší části dat.

> **Consistent hashing:** Technika rozdělení dat mezi uzly tak, aby při přidání nebo odebrání uzlu nebylo nutné přepočítat a přesunout všechna data, ale jen část tokenového prostoru.
>
> **Virtual node (vnode):** Virtuální rozsah tokenů přiřazený fyzickému uzlu. Jeden fyzický uzel může držet mnoho vnodes, což pomáhá rovnoměrněji rozložit data a zjednodušit rebalancing.

## Replikace a dostupnost

Cassandra replikuje data na více uzlů. Počet kopií určuje **replication factor**. Pokud je RF = 3, každý záznam má tři repliky. Replikace zvyšuje dostupnost, protože výpadek jednoho uzlu nemusí znamenat nedostupnost dat. Zároveň ale stojí více místa a komplikuje konzistenci.

> **Replication factor (RF):** Počet kopií každého záznamu v clusteru. RF = 3 znamená, že Cassandra ukládá tři repliky.

Strategie replikace se nastavuje na keyspace. **SimpleStrategy** se hodí hlavně pro vývoj nebo jednoduchý single-datacenter cluster. Repliky umisťuje na sousední uzly v ringu. **NetworkTopologyStrategy** se používá pro produkční nasazení a více datacenter, protože umí nastavit replikační faktor pro každé datacenter zvlášť a rozkládat repliky přes různé racky.

> **Datacenter:** Logická skupina uzlů v Cassandře, typicky odpovídající fyzickému datacentru nebo cloud regionu.
>
> **Rack-aware replikace:** Rozmístění replik tak, aby neležely všechny ve stejném racku. Když vypadne rack, data stále mohou být dostupná z jiné části infrastruktury.

Z hlediska CAP teorému je Cassandra typicky vnímaná jako **AP systém**: snaží se zachovat dostupnost a toleranci k síťovým partition. Konzistence je laditelná. To neznamená, že Cassandra neumí konzistentní čtení, ale že konzistenci nastavujeme pro konkrétní operace pomocí consistency level.

> **CAP teorém:** V distribuovaném systému při síťovém rozdělení nelze současně garantovat plnou konzistenci i dostupnost. Systém si musí vybrat, zda raději odmítne část požadavků, nebo riskuje čtení starších dat.
>
> **Síťová partition:** Stav, kdy se části clusteru navzájem nevidí kvůli problému v síti. Uzly mohou běžet, ale nemohou spolu spolehlivě komunikovat.

## Partition key a clustering columns

Nejdůležitější část návrhu tabulky v Cassandře je primární klíč. Ten se skládá ze dvou částí:

```text
PRIMARY KEY ((partition_key_part_1, partition_key_part_2), clustering_column_1, clustering_column_2)
```

**Partition key** rozhoduje, do které partition záznam patří a na kterých uzlech bude uložen. Může být jednoduchý, například `user_id`, nebo složený, například `(sensor_id, day)`. Cassandra z partition key spočítá hash a podle něj vybere tokenový rozsah.

**Clustering columns** určují pořadí řádků uvnitř jedné partition. Jsou lokálním uspořádáním dat, ne mechanismem pro rozdělení dat mezi uzly. Typický příklad je časová řada: partition key je `(sensor_id, day)` a clustering column je `timestamp`. Tím dostanu všechna měření jednoho senzoru za jeden den pohromadě a uvnitř partition seřazená podle času.

> **Partition:** Skupina řádků se stejným partition key. Cassandra ji používá jako jednotku umístění v clusteru.
>
> **Clustering column:** Sloupec, který určuje pořadí záznamů uvnitř partition. Díky němu lze efektivně číst rozsahy uvnitř jedné partition, například časový interval.

Tady je klíčový rozdíl proti relační databázi. V relačním modelu navrhujeme tabulky primárně podle entit a normalizace. V Cassandře navrhujeme tabulky podle dotazů. Chci-li číst data podle `user_id`, potřebuji tabulku s `user_id` v partition key. Chci-li číst podle `movie_id`, často vznikne jiná tabulka. Duplicitní data nejsou chyba, ale běžná cena za rychlé čtení bez joinů.

> **Denormalizace:** Záměrné ukládání stejné informace na více místech. V relačních databázích se jí často vyhýbáme kvůli konzistenci; v Cassandře je běžná, protože nahrazuje joiny a podporuje dotazy podle konkrétních přístupových vzorů.
>
> **Join:** Operace, která spojuje řádky z více tabulek podle vztahu. Cassandra CQL joiny nepodporuje, takže data musí být připravena ve tvaru, ve kterém je aplikace čte.

Dobrý partition key má dvě vlastnosti. Za prvé rozkládá data rovnoměrně, aby nevznikly hotspoty. Za druhé odpovídá dotazům, aby aplikace neprocházela velké části tabulky. Špatný partition key může být například hodnota s nízkou kardinalitou, jako `country`, pokud většina dat leží v jedné zemi. Taková partition by byla obrovská a jeden uzel by nesl nepřiměřenou zátěž.

> **Hotspot:** Uzel nebo partition, na kterou chodí výrazně víc požadavků než na ostatní. Hotspot ruší výhodu horizontálního škálování.
>
> **Kardinalita:** Počet různých hodnot atributu. `gender` má typicky nízkou kardinalitu, `user_id` vysokou.

## Laditelná konzistence

Cassandra dovoluje nastavovat konzistenci po jednotlivých operacích. Mezi běžné consistency levels patří `ANY`, `ONE`, `QUORUM`, `LOCAL_QUORUM`, `EACH_QUORUM` a `ALL`. Čím vyšší level, tím více replik musí odpovědět. Vyšší konzistence snižuje riziko čtení starých dat, ale zvyšuje latenci a snižuje dostupnost při výpadcích.

> **Consistency level:** Počet a typ replik, od kterých Cassandra vyžaduje potvrzení operace. Není to globální vlastnost tabulky, ale parametr čtení nebo zápisu.

Typický princip quora je jednoduchý. Pokud je replikační faktor 3 a zápis i čtení používají `QUORUM`, zápis čeká na 2 repliky a čtení také na 2 repliky. Tyto množiny se musí alespoň v jedné replice překrýt, takže čtení má šanci narazit na aktuální verzi.

> **Quorum:** Nadpoloviční většina replik. U RF = 3 je quorum 2, u RF = 5 je quorum 3.
>
> **Stale read:** Čtení starší hodnoty, protože některá replika ještě neobsahuje nejnovější zápis.

Když jsou repliky dočasně nekonzistentní, Cassandra má několik opravných mechanismů. **Read repair** opravuje rozdíly zjištěné při čtení. **Hinted handoff** dočasně ukládá zápisy určené pro nedostupný uzel a po jeho návratu mu je doručí. **Anti-entropy repair** je plánovaná synchronizace replik, která porovnává data a dorovnává rozdíly.

> **Read repair:** Oprava nekonzistence mezi replikami při čtení. Pokud Cassandra zjistí, že některá replika má starší hodnotu, může ji opravit.
>
> **Hinted handoff:** Mechanismus, kdy jiný uzel dočasně drží "nápovědu" o zápisu pro nedostupnou repliku a později jí zápis předá.
>
> **Anti-entropy repair:** Proces porovnávání a synchronizace replik, aby se dlouhodobě odstranily rozdíly mezi kopiemi dat.

Konflikty zápisů Cassandra řeší pomocí timestampů a strategie **last write wins**. Každá hodnota má čas zápisu s mikrosekundovou přesností. Když se objeví dvě různé hodnoty pro stejný sloupec, vyhraje ta s novějším timestampem. Je to rychlé a jednoduché, ale vyžaduje opatrnost: špatně synchronizované hodiny nebo ručně nastavený timestamp mohou způsobit, že "novější" zápis ve skutečnosti přepíše správnější hodnotu.

> **Last write wins:** Strategie řešení konfliktů, kde se za vítěznou považuje hodnota s nejnovějším timestampem.
>
> **Timestamp:** Časová značka zápisu. V Cassandře ji systém obvykle přiřazuje automaticky, ale lze ji zadat i ručně v CQL.

# Datové typy a hodnoty

Cassandra má schéma tabulky, ale je flexibilnější než klasická relační tabulka. Každá tabulka má definované sloupce a typy, primární klíč je povinný, ale řádky nemusí mít vyplněné všechny sloupce. Hodnoty mohou být atomické, kolekce, tuple nebo user-defined type.

## Nativní typy

Mezi číselné typy patří `tinyint`, `smallint`, `int`, `bigint`, `varint`, `decimal`, `float` a `double`. Běžný textový typ je `text`, případně `varchar`, což je v Cassandře totéž. Text je UTF-8 a literály se píší do jednoduchých uvozovek:

```sql
'Samotáři'
'Jiří Macháček'
```

Jednoduchá uvozovka uvnitř textu se escapuje zdvojením:

```sql
'It''s fine'
```

> **UTF-8:** Kódování textu, které umí reprezentovat Unicode znaky. Pro české znaky je důležité, aby klient, driver i databáze pracovaly konzistentně v UTF-8.

Pro datum a čas existují typy `date`, `time` a `timestamp`. `timestamp` reprezentuje absolutní okamžik v čase, interně v UTC. Dále existuje `boolean`, `blob` pro libovolné bajty, `inet` pro IPv4/IPv6 adresy a další specializované typy.

> **UTC:** Koordinovaný světový čas. Používá se jako neutrální časová reference bez lokální časové zóny.
>
> **Blob:** Binary Large Object. Libovolná binární data, například serializovaný soubor nebo obsah, kterému databáze sama nerozumí.

Speciální typ je `counter`. Je to 8bajtové celé číslo, které lze pouze inkrementovat nebo dekrementovat. Nelze ho nastavit na libovolnou hodnotu. Má omezení: nesmí být součástí primárního klíče, nepodporuje TTL, nepodporuje secondary indexes a tabulka nemůže volně míchat counter sloupce s běžnými neklíčovými sloupci.

> **Counter:** Sloupec určený pro počítadlo. Hodí se pro případy typu počet zobrazení, ale kvůli distribuovanému chování má v Cassandře zvláštní pravidla.

## Kolekce: list, set, map

Cassandra podporuje tři základní kolekce. `LIST<T>` je uspořádaný seznam hodnot, kde se hodnoty mohou opakovat. `SET<T>` je množina unikátních hodnot. `MAP<K,V>` je kolekce dvojic klíč-hodnota, kde klíče musí být unikátní.

```sql
genres LIST<TEXT>
countries SET<TEXT>
actors MAP<TEXT, TEXT>
```

Literály vypadají takto:

```sql
['comedy', 'drama']
{'CZ', 'SK'}
{'trojan': 'Ivan', 'machacek': 'Jiří'}
```

Seznam udržuje pořadí podle pozic, ale práce s indexy seznamu je v distribuovaném prostředí citlivá. Přednáška výslovně upozorňuje, že update typu `genres[1] = 'comedy'` není kvůli konzistenčním problémům dobrý nápad. Bezpečnější je používat operace přidání a odebrání.

> **Kolekce:** Datový typ, který v jednom sloupci drží více hodnot. U Cassandry se hodí pro malé související množiny hodnot, ne jako náhrada za neomezeně rostoucí podtabulku.

## Tuple a UDT

**Tuple** je uspořádaná n-tice anonymních polí. Každé pole může mít jiný typ. Hodí se pro malé pevné struktury, kde názvy polí nejsou důležité.

```sql
name TUPLE<TEXT, TEXT>
```

Hodnota tuple se zapisuje kulatými závorkami:

```sql
('Jiří', 'Macháček')
```

**User-defined type** je vlastní složený typ s pojmenovanými poli. Například detail filmu může obsahovat délku a anotaci:

```sql
CREATE TYPE details (
  length SMALLINT,
  annotation TEXT
);
```

Potom se typ použije ve sloupci:

```sql
properties details
```

A hodnota se zapisuje jako objekt:

```sql
{length: 100, annotation: 'Krátký popis'}
```

> **Tuple:** Pevná n-tice hodnot bez názvů polí. Význam hodnot je dán pořadím.
>
> **UDT (user-defined type):** Uživatelsky definovaný typ s pojmenovanými poli. Pomáhá ukládat malé složené hodnoty uvnitř řádku.

## Frozen mode

Vnořené kolekce, tuple nebo UDT mohou být v Cassandře ukládány ve **frozen** režimu. Frozen hodnota se z pohledu databáze chová jako jeden celek. Cassandra neumí efektivně měnit její vnitřní část, ale musí přepsat celou hodnotu. To je jednoduché, ale při velkých hodnotách nákladné.

> **Frozen mode:** Režim u složených hodnot, kdy Cassandra ukládá celou hodnotu jako nedělitelný celek. Nelze pohodlně aktualizovat jen jednu vnořenou část bez přepsání celé hodnoty.

Praktické pravidlo je: kolekce a UDT mají být malé a přirozeně patřit k jednomu řádku. Pokud hodnota může růst bez omezení, například nekonečný seznam komentářů, je lepší vytvořit samostatnou tabulku navrženou pro konkrétní dotaz.

## TTL, timestamp a tombstones

Cassandra ukládá k hodnotám dodatečná metadata. **TTL** říká, za kolik sekund má hodnota automaticky expirovat. Lze ho použít u `INSERT` nebo `UPDATE`:

```sql
INSERT INTO movies (id, title)
VALUES ('stesti', 'Štěstí')
USING TTL 86400;
```

Po uplynutí TTL hodnota nezmizí okamžitě jako fyzicky smazaná buňka. Cassandra ji označí tak, aby se už neměla vracet, a později ji odstraní při kompaktaci. Tady vstupují do hry **tombstones**.

> **TTL (time-to-live):** Doba života hodnoty v sekundách. Po jejím uplynutí se hodnota považuje za expirovanou.
>
> **Kompaktace:** Interní proces, při kterém Cassandra slučuje datové soubory na disku, odstraňuje staré verze a čistí tombstones, když je bezpečné je zahodit.

**Tombstone** je značka smazání. V distribuované databázi nestačí hodnotu prostě smazat z jednoho uzlu, protože jiná replika může být zrovna nedostupná a později by starou hodnotu znovu rozšířila. Tombstone říká: "tato hodnota byla smazána". Repliky se tak při opravách dozvědí, že nemají starou hodnotu obnovovat.

> **Tombstone:** Záznam označující smazání hodnoty. Cassandra ho potřebuje kvůli replikaci a eventual consistency, aby se smazání správně rozšířilo na všechny repliky.

Každá hodnota má také **writetime**, tedy timestamp poslední modifikace. CQL umí vracet `WRITETIME(column)` a `TTL(column)`, i když s omezeními u kolekcí a jejich prvků.

```sql
SELECT WRITETIME(title), TTL(title)
FROM movies
WHERE id = 'stesti';
```

# Cassandra Query Language (CQL)

CQL je deklarativní jazyk inspirovaný SQL, ale nesmí se číst jako plnohodnotné SQL. Umí vytvářet keyspaces a tabulky, vkládat, měnit, mazat a vybírat data. Neumí joiny a jeho `WHERE` je silně omezené tím, jak je tabulka fyzicky navržená.

> **Deklarativní jazyk:** Jazyk, kde říkáme, jaký výsledek chceme, ne přesný postup výpočtu. SQL i CQL jsou deklarativní, ale CQL má kvůli distribuovanému modelu podstatně přísnější omezení.

S Cassandrou se dá pracovat přes `cqlsh`, interaktivní příkazovou řádku, nebo přes klientské drivery pro jazyky jako Java, Python, Ruby, PHP, C++, Scala či Erlang.

## DDL: keyspaces

Keyspace vytvoříme příkazem `CREATE KEYSPACE`. Replikační nastavení je povinné:

```sql
CREATE KEYSPACE moviedb
WITH replication = {
  'class': 'SimpleStrategy',
  'replication_factor': 3
};
```

V produkci s více datacentery se typicky použije `NetworkTopologyStrategy`:

```sql
CREATE KEYSPACE moviedb
WITH replication = {
  'class': 'NetworkTopologyStrategy',
  'dc1': 3,
  'dc2': 2
};
```

Aktuální keyspace se nastaví příkazem:

```sql
USE moviedb;
```

Keyspace lze odstranit:

```sql
DROP KEYSPACE IF EXISTS moviedb;
```

V přednášce se objevuje i `ALTER KEYSPACE`, kterým lze měnit nastavení existujícího keyspace, například replikační strategii. U takové změny je ale potřeba počítat s tím, že samotná změna definice nestačí na okamžité fyzické přeskupení všech dat; v praxi se musí řešit i opravy a provozní dopady.

## DDL: tabulky

Tabulka se vytváří příkazem `CREATE TABLE`. Každá tabulka musí mít právě jeden primární klíč. Primární klíč může být jednoduchý:

```sql
CREATE TABLE IF NOT EXISTS actors (
  id TEXT PRIMARY KEY,
  name TUPLE<TEXT, TEXT>,
  year SMALLINT,
  movies SET<TEXT>
);
```

Nebo může být složený z partition key a clustering columns:

```sql
CREATE TABLE IF NOT EXISTS sensor_readings (
  sensor_id TEXT,
  day DATE,
  measured_at TIMESTAMP,
  value DOUBLE,
  PRIMARY KEY ((sensor_id, day), measured_at)
);
```

V druhém příkladu `sensor_id` a `day` tvoří složený partition key. Sloupec `measured_at` je clustering column. Dotaz "dej mi měření senzoru za daný den seřazená podle času" je díky tomu přirozený a efektivní.

Tabulku lze odstranit:

```sql
DROP TABLE IF EXISTS sensor_readings;
```

`TRUNCATE TABLE` zachová strukturu tabulky, ale smaže její data:

```sql
TRUNCATE TABLE sensor_readings;
```

> **TRUNCATE:** Operace, která odstraní všechna data z tabulky, ale ponechá její definici. Je to něco jiného než `DROP TABLE`, který odstraní i tabulku jako objekt.

Přednáška zmiňuje také `ALTER TABLE`, které umožňuje přidávat nebo odebírat sloupce a měnit některé vlastnosti tabulky. V Cassandře ale změna schématu není náhrada za dobrý návrh podle dotazů. Když se změní přístupové vzory, často je lepší přidat novou query-specific tabulku než se snažit zachránit jednu univerzální tabulku.

## DML: SELECT

`SELECT` vybírá řádky z jedné tabulky. Cassandra nepodporuje joiny, takže `FROM` ukazuje vždy na jednu tabulku:

```sql
SELECT id, title, actors
FROM movies
WHERE id = 'medvidek';
```

Tohle je přirozený dotaz, pokud `id` je partition key. Cassandra ví, na kterou partition sáhnout, a nemusí procházet celou tabulku.

Největší rozdíl proti SQL je `WHERE`. Cassandra dovoluje efektivně filtrovat hlavně podle primárního klíče. Partition key se typicky musí určit rovností nebo `IN`. Clustering columns lze filtrovat v pořadí, ve kterém jsou definované, a podporují i rozsahové podmínky. Ne-klíčové sloupce jsou problém, pokud nad nimi není index nebo pokud nepoužijeme `ALLOW FILTERING`.

```sql
SELECT measured_at, value
FROM sensor_readings
WHERE sensor_id = 's1'
  AND day = '2026-05-26'
  AND measured_at >= '2026-05-26 08:00:00+0000'
  AND measured_at <  '2026-05-26 09:00:00+0000';
```

> **Secondary index:** Dodatečný index nad neklíčovým sloupcem. V Cassandře není univerzální náhrada za relační indexy; musí se používat opatrně a podle kardinality a dotazů.

Pro kolekce existují operátory `CONTAINS` a `CONTAINS KEY`. `CONTAINS` testuje, jestli list, set nebo map obsahuje hodnotu. `CONTAINS KEY` testuje, jestli map obsahuje daný klíč. Efektivní použití obvykle vyžaduje index a stále platí, že to není náhrada za promyšlený datový model.

```sql
SELECT id, title
FROM movies
WHERE genres CONTAINS 'comedy'
ALLOW FILTERING;
```

Tento dotaz je didakticky užitečný, ale produkčně podezřelý. Cassandra varuje, že `ALLOW FILTERING` může znamenat procházení mnoha řádků, ze kterých se vrátí jen malá část. Výkon pak závisí na množství přečtených dat, ne na množství vrácených dat.

> **ALLOW FILTERING:** Povolení dotazu, který Cassandra neumí provést jako předvídatelný lookup podle klíče nebo indexu. Může vést k full table scanu a v produkci je většinou signálem špatného modelu.
>
> **Full table scan:** Procházení celé tabulky nebo její velké části. V distribuované databázi s obrovskými daty je to často nepřijatelně drahé.

`SELECT` podporuje projekci sloupců, aliasy pomocí `AS`, `DISTINCT`, agregace a funkce jako `WRITETIME` a `TTL`.

```sql
SELECT DISTINCT sensor_id
FROM sensor_readings
WHERE sensor_id IN ('s1', 's2');
```

`DISTINCT` v Cassandře funguje jen omezeně, typicky nad partition key columns, a může být paměťově náročný. Agregace jako `COUNT`, `MIN`, `MAX`, `SUM` a `AVG` existují, ale nejsou pozvánkou k libovolné analytice nad celým clusterem. Cassandra je optimalizovaná pro předvídatelné dotazy, ne pro ad-hoc analytické SQL.

> **Agregace:** Výpočet souhrnné hodnoty z více řádků, například počet, součet nebo průměr.
>
> **Ad-hoc dotaz:** Dotaz vytvořený bez předchozího návrhu datového modelu pro daný přístupový vzor. Cassandra je na takové dotazy výrazně méně vhodná než relační databáze nebo analytický engine.

`ORDER BY` lze použít jen pro clustering columns a musí respektovat clustering order definovaný v tabulce. Nelze libovolně řadit podle jakéhokoli sloupce jako v SQL:

```sql
SELECT measured_at, value
FROM sensor_readings
WHERE sensor_id = 's1'
  AND day = '2026-05-26'
ORDER BY measured_at DESC
LIMIT 100;
```

`LIMIT` omezuje počet vrácených řádků. Je užitečný například pro dotaz "posledních 100 událostí", pokud jsou data správně seřazená clustering column.

`GROUP BY` je také omezený. Cassandra dovoluje jen seskupení odvozené z primárního klíče. Pokud se v `SELECT` objeví neagregovaný sloupec, který není součástí grouping logiky, Cassandra vrátí první narazenou hodnotu, což je potřeba chápat velmi opatrně.

## DML: INSERT

`INSERT` vkládá nový řádek do tabulky. Pokud řádek se stejným primárním klíčem už existuje, Cassandra ho v podstatě aktualizuje. Musí být uvedeny alespoň sloupce primárního klíče a názvy sloupců se vždy explicitně vypisují:

```sql
INSERT INTO movies (id, title, director, year, actors, genres)
VALUES (
  'stesti',
  'Štěstí',
  ('Bohdan', 'Sláma'),
  2005,
  {'vilhelmova': 'Monika', 'liska': 'Toník'},
  ['comedy', 'drama']
)
USING TTL 86400;
```

V CQL lze použít `IF NOT EXISTS`, což z insertu udělá podmíněný zápis. Ten je ale dražší než obyčejný zápis, protože vyžaduje koordinaci navíc.

> **Podmíněný zápis:** Zápis, který proběhne jen tehdy, když je splněna podmínka, například "řádek ještě neexistuje". V Cassandře se takové operace realizují jako lehké transakce a jsou dražší než běžné zápisy.

## DML: UPDATE

`UPDATE` mění sloupce existujícího řádku. Pokud řádek s daným primárním klíčem neexistuje, Cassandra ho vytvoří. To je další rozdíl proti běžné intuici z relační databáze: zápisy v Cassandře jsou často **upsert**.

```sql
UPDATE movies
SET year = 2006,
    director = ('Jan', 'Svěrák'),
    actors = {'machacek': 'Robert Landa', 'sverak': 'Josef Tkaloun'},
    genres = ['comedy']
WHERE id = 'vratnelahve';
```

> **Upsert:** Operace, která vloží nový záznam, pokud neexistuje, nebo aktualizuje existující záznam, pokud už existuje.

U `UPDATE` musí `WHERE` obsahovat potřebné části primárního klíče. Cassandra nemá provádět update typu "najdi všechny filmy z roku 2006 a změň jim atribut" přes celou tabulku. Takový styl práce patří spíš do relační databáze nebo analytického zpracování.

Kolekce lze měnit celé nebo po částech:

```sql
UPDATE movies
SET actors = actors + {'vilhelmova': 'Helenka'},
    genres = ['drama'] + genres,
    countries = countries + {'SK'}
WHERE id = 'vratnelahve';
```

Odebírání z kolekcí:

```sql
UPDATE movies
SET actors = actors - {'vilhelmova', 'landovsky'},
    genres = genres - ['drama', 'sci-fi'],
    countries = countries - {'SK'}
WHERE id = 'vratnelahve';
```

`UPDATE` i `INSERT` podporují `USING TTL` a `USING TIMESTAMP`:

```sql
UPDATE sessions
USING TTL 3600
SET user_id = 'u123'
WHERE session_id = 'abc';
```

`TTL` ovlivní jen nově zapsané nebo změněné hodnoty. Staré hodnoty, kterých se update nedotkne, tím automaticky nedostanou nový čas života.

## DML: DELETE

`DELETE` může odstranit celý řádek, vybrané sloupce, prvky kolekcí nebo pole UDT hodnoty:

```sql
DELETE FROM movies
WHERE id = 'stesti';
```

Odstranění jednoho sloupce:

```sql
DELETE title
FROM movies
WHERE id = 'stesti';
```

Odstranění prvku mapy:

```sql
DELETE actors['vilhelmova']
FROM movies
WHERE id = 'stesti';
```

Kvůli replikaci se delete uvnitř Cassandry projeví jako tombstone. Proto je nebezpečné navrhnout model, který vytváří obrovské množství mazání nebo krátkých TTL hodnot v rychlém tempu. Systém pak musí při čtení přeskakovat spoustu tombstones a později je čistit kompaktací.

## Jak si Cassandru pamatovat ke státnicím

Cassandra je wide-column databáze pro známé dotazy, vysoký objem dat, rychlé zápisy a distribuovaný provoz. Její hlavní síla je horizontální škálování, vysoká dostupnost, peer-to-peer architektura, replikace a laditelná konzistence. Hlavní cena je omezený dotazovací model, absence joinů, nutnost denormalizace a velmi pečlivý návrh partition key a clustering columns.

Nejkratší mentální model je tento: **tabulka v Cassandře není univerzální sklad entit, ale materializovaný přístupový vzor**. Pokud znám dotaz, navrhnu partition key tak, aby Cassandra rychle našla správnou partition, a clustering columns tak, aby data uvnitř partition byla ve správném pořadí. Pokud dotaz neodpovídá primárnímu klíči, Cassandra mě donutí buď vytvořit další tabulku, použít index s rozmyslem, nebo sáhnout po `ALLOW FILTERING`, což je většinou varování, že model není pro daný dotaz dobrý.
