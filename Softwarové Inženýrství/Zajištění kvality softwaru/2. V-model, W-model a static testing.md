## 1. V-model

**V-model** propojuje fáze vývoje s odpovídajícími úrovněmi testování. 

> [!NOTE]
> Jako test basis slouží business model, requirements, functional design, technical design a implementation, proti nim stojí development tests, integration tests, system tests a user acceptance tests.

![[Assets/ZKS/Pasted image 20260530130158.png]]

Typická vazba:

| Levá strana                   | Pravá strana             |
| ----------------------------- | ------------------------ |
| business model / requirements | user acceptance tests    |
| functional design             | system tests             |
| technical design              | integration tests        |
| implementation                | development / unit tests |

Hlavní myšlenka:

> Každá úroveň testování má svůj test basis na levé straně modelu.

Výhody:
- přehledná struktura;
- traceability mezi artefakty a testy;
- jasné test levels;
- vhodné pro plánování testování.

![[Assets/ZKS/Pasted image 20260530130254.png|660]]

Nevýhody:
- při lineárním použití se chyby v requirements často najdou až pozdě;
- pozdní opravy jsou drahé;
- model sám o sobě neznamená, že artefakty na levé straně jsou průběžně kontrolované.

Dobrá formulace:

> V-model dobře mapuje vztah mezi vývojovými artefakty a testovacími úrovněmi, ale jeho slabinou je riziko pozdního odhalení chyb vzniklých v requirements nebo designu.

## 2. W-model

**W-model** rozšiřuje V-model o průběžné ověřování artefaktů na levé straně životního cyklu.

![[Assets/ZKS/Pasted image 20260530130345.png|674]]

Nejde jen o to, že testy existují na konci. Každá fáze přípravy systému má vlastní kontrolní nebo testovací aktivitu.

Příklady:
- quality aspects jsou součástí business modelu;
- revision of requirements;
- revision of functional design;
- revision of technical design;
- static testing před implementací a během ní.

Výhody:
- chyby se zachytí dříve;
- podporuje shift-left;
- snižuje riziko drahých pozdních oprav;
- zvyšuje kvalitu test basis;
- lépe odpovídá Boehmovu principu ceny chyby.

Nevýhody:
- vyžaduje vyšší počáteční investici;
- potřebuje zapojení testerů/QA už od začátku;
- v praxi není vždy používán důsledně.

Krátké srovnání:

| Kritérium                         | V-model                             | W-model                                            |
| --------------------------------- | ----------------------------------- | -------------------------------------------------- |
| Hlavní princip                    | mapuje vývojové fáze na test levels | ke každé fázi přidává kontrolní/testovací aktivitu |
| Levá strana modelu                | slouží jako test basis              | aktivně se ověřuje                                 |
| Zachycení requirement/design chyb | spíše později                       | dříve                                              |
| Investice                         | nižší počáteční                     | vyšší počáteční                                    |
| Přínos                            | struktura testování                 | shift-left a prevence defektů                      |

## 3. Static testing

**Static testing** je testování specifikace, návrhu nebo implementace **bez spuštění softwaru**, například pomocí reviews nebo static analysis. Je uvedeno, že probíhá na levé straně V nebo W modelu, bývá samostatnou úrovní testování a z teorie testování jde o velmi efektivní metodu detekce defektů.

Do static testing patří hlavně:
- reviews;
- inspections;
- walkthroughs;
- static code analysis;
- kontrola dokumentace;
- quality gates.

> [!NOTE]
> Static testing vyžaduje počáteční investici a ROI se hůř prokazuje u unikátních projektů. U změn pouze části systému může být problém s doménovou znalostí a důsledky změny.

> [!IMPORTANT]
> **static analysis a dynamic testing jsou komplementární** - jedno nenahrazuje druhé. Static analysis hledá problémy před spuštěním, ale neověří například splnění všech funkčních požadavků při běhu.

## 4. Role static testing ve W-modelu

Ve W-modelu je static testing přirozenou součástí levé strany procesu.

![[Assets/ZKS/Pasted image 20260530130904.png]]

Pomáhá ověřit:
- requirements proti business modelu;
- functional design proti requirements;
- technical design proti functional designu;
- source code proti coding standards, security rules a quality rules.

Smysl:
```text
najít defekt předtím, než se dostane do implementace a dynamických testů
```

To snižuje cenu opravy, protože se opravuje artefakt včas.

## 5. Metody static testingu

### Requirements review

Kontrola požadavků na:
- úplnost;
- jednoznačnost;
- testovatelnost;
- konzistenci;
- konflikt mezi požadavky;
- business správnost.

> [!NOTE]
> Příklad problému: požadavek „systém má být rychlý“ není testovatelný, pokud není určena konkrétní metrika.
### Design review

Kontrola návrhu systému proti požadavkům a quality goals.

Typicky se řeší:
- realizovatelnost návrhu;
- škálovatelnost;
- bezpečnost;
- integrace;
- rizikové části architektury;
- konzistence s requirements.
### Code review

Ruční nebo částečně podporovaná kontrola zdrojového kódu.

Hledá:
- logické chyby;
- bezpečnostní slabiny;
- porušení coding standards;
- špatnou čitelnost;
- duplicity;
- příliš složité metody;
- potenciální runtime problémy.
### Walkthrough

Autor artefaktu provází ostatní účastníky dokumentem, návrhem nebo kódem.

Cíl:
- sdílet pochopení;
- najít nejasnosti;
- získat feedback;
- méně formální než inspection.
### Inspection

Formálnější metoda review.

Typicky má:
- role;
- checklisty;
- přípravu před meetingem;
- záznam nalezených defektů;
- follow-up.

Je vhodná tam, kde je vysoký požadavek na kvalitu nebo auditovatelnost.
### Static code analysis

Automatizovaná analýza zdrojového kódu bez spuštění programu.

Hledá například:
- chyby v kódu;
- security vulnerabilities;
- porušení standardů;
- smell patterns – podezřelé vzory v kódu, které nejsou nutně chybou, ale naznačují špatný design nebo horší udržovatelnost;
- vysokou cyclomatic complexity;
- dead code;
- technický dluh.
### Quality gates

Automatické kontroly typicky v CI/CD.

Příklady:
- linting;
- static analysis;
- test coverage threshold;
- security scan;
- dependency scan;
- kontrola complexity;
- build must pass.

## 6. Výhody a nevýhody static testingu

Výhody:
- zachytí defekty velmi brzy;
- levnější oprava;
- lze testovat i dokumenty a návrh;
- pomáhá se standardy a compliance;
- vhodné pro shift-left.

Nevýhody:
- vyžaduje počáteční investici;
- ROI (**Return on Investment**) může být hůř prokazatelné;
- reviews vyžadují čas lidí;
- static analysis může dávat false positives;
- nenahrazuje dynamické testování.
