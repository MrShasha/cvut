## 1. Co je E2E testing

**End-to-end testing** testuje celý software nebo business flow od začátku do konce. Ověřuje **celý průchod systémem od začátku do konce**, tedy realistický workflow nebo transakci. Slidy to rozdělují na **horizontal E2E testing** a **vertical E2E testing**.

Cíl:
- ověřit, že integrované části systému spolupracují;
- simulovat reálný uživatelský scénář;
- ověřit datovou integritu;
- najít chyby, které se projeví až v celé integraci.

Přínosy E2E testů:
- ověřují systém z uživatelského pohledu;
- zachytí integrační chyby;
- zabraňují regresím v klíčových flows;
- ověřují business scénáře;
- dávají managementu srozumitelný signál o dopadu selhání.

Příklad E2E scénáře:

```text
uživatel se přihlásí → přidá zboží do košíku → zaplatí → systém vytvoří objednávku a fakturu
```

E2E testy jsou špička pyramidy:

```text
mnoho unit testů
méně integration/service tests
málo E2E testů
```

Důvod:
- E2E testy jsou pomalé;
- drahé na údržbu;
- náchylné k flaky failures;
- horší diagnostika příčiny chyby.

> [!NOTE]
> > E2E tests are important, but they should cover key user journeys, not replace unit and integration tests.

## 4. Horizontal E2E testing

**Horizontal testing** ověřuje, zda se uživatel dokáže systémem pohybovat a zda celý workflow nebo transakce proběhne správně napříč jednotlivými aplikacemi či subsystémy. Je zaměřené z pohledu uživatele a ověřuje end-to-end business proces. Výhody uvedené ve slidech:

Příklad:

```text
open browser → login → create order → pay → receive confirmation
```

Výhody:
- user perspective;
- business flow coverage;
- prevence problémů v produkci.

Nevýhody:
- složité test environment;
- pomalejší;
- křehčí při změnách UI;
- náročnější diagnostika.

---
## 5. Vertical E2E testing

**Vertical testing** je více architekturní. Probíhá sekvenčně po vrstvách systému a často se používá pro **kritické komponenty komplexních systémů**, které obvykle nezahrnují uživatele nebo UI. 

Příklad:

```text
API request → service logic → database → response
```

Výhody:
- rychlejší než full horizontal UI test;
- vyšší code coverage;
- vhodné pro backend nebo safety-critical komponenty;
- méně závislé na UI.

## 6. Selenium

**Selenium** automatizuje webové prohlížeče.

Umí:
- otevřít URL;
- najít prvky na stránce;
- kliknout;
- vyplnit formulář;
- pracovat s cookies;
- navigovat mezi stránkami;
- spustit JavaScript;
- ověřit text nebo stav prvků.

Lokátory:
- ID;
- name;
- XPath;
- CSS selector;
- link text;
- class name;
- tag name.

---
## 7. Problém jednoduchých Selenium testů

Naivní testy jsou často:
- plné Selenium detailů;
- špatně čitelné;
- duplicitní;
- citlivé na změny DOM;
- obtížné na údržbu.

Příklad problému:

```text
když se změní CSS selector, musí se měnit mnoho testů
```

## 8. Page Object Model (POM)

**Page Object Model (POM)** je design pattern pro UI testy, ve kterém má každá stránka aplikace vlastní **page class**. 

Tato třída obsahuje lokátory UI prvků a metody pro práci s nimi, například vyplnění formuláře, kliknutí na tlačítko nebo přihlášení uživatele. 

Samotný test potom nepoužívá nízkoúrovňové Selenium detaily, ale volá srozumitelné business-level metody, například `loginAs(user)` místo ručního hledání inputů a klikání na tlačítka.

Příklad rozdílu:

```text
špatně: findElement(By.cssSelector("#username")).sendKeys("user")
lépe: loginPage.loginAs("user", "password")
```

Výhody:
- vyšší čitelnost;
- méně copy-paste;
- lepší údržba;
- Selenium kód je skrytý;
- změna UI se často opraví v jedné page class.

Nevýhody:
- vyšší setup cost;
- page classes mohou být velké;
- zaměření na stránky místo user stories;
- možné porušení SRP/OCP;
- ne vždy vhodné pro single-page apps.

## 9. BDD

**Behavior Driven Development (BDD)** je přístup k vývoji a testování, který se zaměřuje na **chování systému z pohledu uživatele nebo businessu**. Hlavní myšlenka je, že systém se nepopisuje jen technicky, ale pomocí srozumitelných scénářů, kterým rozumí vývojáři, testeři, analytici i business role.

BDD pomáhá zmenšit komunikační mezeru mezi business a technickým týmem. Místo nejasného požadavku typu „systém má podporovat objednávky“ se popíše konkrétní příklad chování: co je počáteční stav, co uživatel udělá a jaký výsledek se očekává.

Typická struktura:

```gherkin
Given initial state
When action happens
Then expected outcome
```

Příklad:

```gherkin
Feature: Add new todo item
Scenario: Adding first item
  Given James has an empty todo list
  When he adds "Buy milk"
  Then the todo list should contain "Buy milk"
```

Výhody:
- čitelné pro netechnické lidi;
- snižuje misunderstanding requirements;
- zaměřuje se na behavior;
- vhodné pro acceptance tests.

## 10. Screenplay pattern

**Screenplay pattern** je návrhový vzor pro automatizované acceptance/UI testy, který se nezaměřuje primárně na stránky aplikace, ale na **uživatelské cíle, úkoly a chování systému**.

Na rozdíl od Page Object Modelu, kde často modelujeme jednotlivé stránky, Screenplay pattern modeluje test jako příběh: určitý uživatel chce dosáhnout určitého cíle, má k tomu určité schopnosti, provádí úkoly a potom ověřuje výsledek.

Základní pojmy:
- **Actor** — uživatel nebo role, která používá systém, například `James`, `Admin`, `Customer`;
- **Ability** — schopnost actora, například používat webový prohlížeč, volat API nebo pracovat s aplikací;
- **Task** — úkol, který actor provádí, například přihlásit se, přidat produkt do košíku nebo dokončit objednávku;
- **Question** — otázka na stav systému, například zda je uživatel přihlášen nebo zda košík obsahuje produkt;
- **Goal** — business cíl, kterého chce actor dosáhnout.

Jednoduchý příklad:

```
Actor: Customer
Ability: browse the web
Task: add product to cart
Question: does the cart contain the product?
Goal: buy a product
```

Výhody:
- story-focused;
- dobrá čitelnost;
- lepší modularita;
- composition over inheritance;
- vhodné pro BDD;
- lepší maintainability u větších test suites.

Nevýhody:
- steep learning curve;
- vyžaduje změnu myšlení;
- více abstrakce;
- může být zbytečné pro malé projekty.

## 11. API E2E testing

**E2E testing nemusí znamenat pouze UI testing.** End-to-end test může být proveden i přes API, pokud API pokrývá celý business flow systému.

API E2E test nekliká v uživatelském rozhraní, ale posílá požadavky přímo na API endpointy. Přesto může ověřovat celý scénář od začátku do konce, například vytvoření objednávky, platbu, změnu stavu objednávky a ověření uložených dat.

API E2E test může ověřovat například:
- status code;
- response body;
- content length;
- response time;
- data persistence;
- celý business flow přes API.

Nástroje:
- Postman;
- Newman;
- REST Assured;
- SoapUI;
- JMeter.

## 12. Flaky tests

**Flaky test** je test, který někdy projde a někdy selže, i když se samotný kód aplikace nezměnil. Takový test je problém, protože snižuje důvěru v testovací sadu. Vývojáři potom nevědí, zda test selhal kvůli skutečné chybě v aplikaci, nebo kvůli nestabilitě testu.

Příčiny:
- špatná izolace;
- sdílená test data;
- nestabilní externí služby;
- testování proti produkčnímu backendu;
- špatné timeouty;
- async/AJAX čekání;
- nestabilní infrastruktura.

Řešení:
- izolovat testy;
- používat mocky/stuby tam, kde dává smysl;
- kontrolovaná testovací data;
- explicit waits;
- vyhnout se fixním sleepům;
- logy, screenshoty, trace;
- testovat méně přes UI, více přes nižší vrstvy.
