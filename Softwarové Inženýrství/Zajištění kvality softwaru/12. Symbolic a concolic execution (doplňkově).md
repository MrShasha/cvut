
Toto téma není hlavní jádro okruhu o V/W modelu, MBT, CIT a path-based testingu, ale patří do širšího kontextu automatického generování testů a analýzy programů.

## 1. Problém manuálního a náhodného testování

### Manuální testování

Tester ručně vytváří vstupy.

Výhody:
- lidská intuice;
- corner cases;
- jednoduché použití.

Nevýhody:
- drahé;
- omezené pokrytí;
- těžké najít vzácné chyby;
- závisí na zkušenosti.
### Random testing

Stroj generuje mnoho náhodných vstupů.

Výhody:
- snadno automatizovatelné;
- levné generování vstupů.

Nevýhody:
- špatně trefuje corner cases;
- mnoho vstupů pokrývá stejnou cestu;
- nízká pravděpodobnost pro specifické vzácné kombinace.

---
## 2. Symbolic execution

**Symbolic execution** spouští program se symbolickými hodnotami místo konkrétních vstupů.

Příklad:

```text
místo x = 5 použijeme x = α
```

Když program narazí na větev závislou na symbolické hodnotě, execution se rozdělí:

```text
if (x > 0)
```

vzniknou dvě větve:

```text
α > 0
α <= 0
```

Každá cesta má svůj **path condition**.

---
## 3. Path condition

**Path condition** je logická podmínka, kterou musí vstupy splnit, aby program prošel danou cestou.

Příklad:

```text
α > 0 ∧ β < 5 ∧ γ != 0
```

Solver pak hledá konkrétní hodnoty, které podmínku splňují.

Výsledek:

```text
α = 1, β = 4, γ = 2
```

Tyto hodnoty lze použít jako konkrétní test case.

---
## 4. Výhody symbolic execution

- systematicky prochází cesty;
- jedna symbolická cesta reprezentuje mnoho konkrétních běhů;
- umí generovat konkrétní failing input;
- dobré pro bug detection;
- užitečné pro automatické generování testů.

Důležité:

> Symbolic execution is powerful for finding bugs, but in general it cannot prove absence of all errors.

---
## 5. Limity symbolic execution

- path explosion;
- loops;
- recursion;
- pointers;
- heap modeling;
- arrays;
- memory aliasing;
- floating point constraints;
- string constraints;
- system calls;
- environment interaction;
- concurrency;
- cryptographic code.

---
## 6. Path explosion

Každý `if` může zdvojnásobit počet cest.

```text
n větvení → až 2^n cest
```

Smyčky jsou ještě horší.

Řešení:
- bounded loop unrolling;
- search heuristics;
- DFS/BFS;
- random search;
- coverage-guided search;
- prioritization of unexplored code.

---
## 7. SAT a SMT solvery

**Symbolic execution** potřebuje solver, protože při průchodu programem vznikají **path conditions**. Solver potom ověřuje, zda existují konkrétní vstupní hodnoty, které danou podmínku splní.

Jinými slovy: symbolic execution vytvoří logickou podmínku a solver řekne, jestli je možné ji splnit. Pokud ano, může vygenerovat konkrétní hodnoty pro test case.

### SAT

**SAT** znamená **Boolean Satisfiability Problem**. Řeší otázku, zda existuje přiřazení hodnot `true/false`, které udělá booleovskou formuli pravdivou.

Příklad:

```text
A ∧ ¬B
```

je splnitelné například pro:

```text
A = true, B = false
```

Naopak výraz:

```text
A ∧ ¬A
```

splnitelný není, protože `A` nemůže být zároveň `true` i `false`.

SAT pracuje hlavně s booleovskými proměnnými a logickými operátory jako `AND`, `OR`, `NOT`.

### SMT

**SMT** znamená **Satisfiability Modulo Theories**. Je to rozšíření SAT, které kromě booleovské logiky umí pracovat i s různými typy dat a matematickými teoriemi.

SMT solver tedy neřeší jen `true/false`, ale také například:
- celočíselnou aritmetiku;
- reálnou aritmetiku;
- pole;
- bitové vektory;
- seznamy;
- řetězce;
- uninterpreted functions.

Příklad SMT podmínky:

```text
x > 10 ∧ y = x + 5
```

Solver může najít například:

```text
x = 11, y = 16
```

To je užitečné pro symbolic execution, protože podmínky v programu často nejsou jen booleovské, ale obsahují čísla, porovnání, výpočty, pole nebo řetězce.

Příklad známého SMT solveru:

- **Z3**.

Krátce:

> SAT řeší splnitelnost booleovských formulí. SMT řeší splnitelnost logických formulí rozšířených o teorie, například aritmetiku, pole, bitové vektory nebo řetězce.

## 8. Concolic testing

**Concolic testing** kombinuje **concrete execution** a **symbolic execution**. Název vzniká ze slov **concrete + symbolic**.

Princip je takový, že program se nejprve spustí s konkrétními vstupními hodnotami. Během tohoto běhu se zároveň sleduje, jaké podmínky musely být splněny, aby program prošel právě danou cestou. Tyto podmínky se nazývají **path constraints**.

Na konci běhu se některá dosud neprozkoumaná branch condition neguje a solver se pokusí najít nové konkrétní vstupní hodnoty, které program navedou na jinou cestu. Program se potom znovu spustí s těmito novými vstupy. Tím se postupně prozkoumávají různé execution paths.

Základní princip:
1. program se spustí s konkrétním vstupem;
2. zároveň se sledují symbolické path constraints;
3. na konci cesty se neguje některá branch condition;
4. solver vygeneruje nový konkrétní vstup;
5. program se spustí znovu;
6. postup se opakuje.

Výhoda concolic testingu je v tom, že používá skutečný běh programu, takže lépe zvládá některé praktické problémy čistě symbolického provádění, například system calls, knihovní funkce nebo příliš složité výrazy. Zároveň ale díky symbolické části umí systematicky hledat nové vstupy pro nepokryté cesty.

Krátce:

> Concolic testing spouští program s konkrétními vstupy, ale zároveň sbírá symbolické podmínky cesty a pomocí solveru generuje nové vstupy pro další cesty.

---
## 9. Concolic testing algorithm

Zjednodušený algoritmus:
1. klasifikovat vstupy jako symbolic/concrete;
2. instrumentovat program, aby bylo možné sledovat path constraints;
3. vybrat počáteční konkrétní vstup;
4. spustit program;
5. zaznamenat path constraints pro prošlou cestu;
6. negovat ještě neprozkoumanou branch condition;
7. použít solver pro vygenerování nového konkrétního vstupu;
8. spustit program znovu;
9. opakovat, dokud nejsou pokryty požadované cesty nebo dokud už solver nenajde nové vstupy.

Jednoduchý příklad:

```text
if (x > 10) {
    if (y == x + 1) {
        error();
    }
}
```

První běh může začít například s hodnotami:

```text
x = 5, y = 0
```

Program projde větví:

```text
x <= 10
```

Concolic engine potom neguje tuto podmínku a chce projít druhou větev:

```text
x > 10
```

Solver může vygenerovat například:

```text
x = 11, y = 0
```

Další běh projde do vnitřního `if`, kde vznikne podmínka:

```text
y != x + 1
```

Negací této podmínky dostaneme:

```text
y == x + 1
```

Solver může vygenerovat:

```text
x = 11, y = 12
```

Tento vstup už dovede program do `error()`.

## 10. Nástroje

Mezi známé nástroje a technologie patří:

- **KLEE** — symbolic execution nástroj pro C, postavený nad LLVM;
- **SAGE** — nástroj od Microsoftu pro symbolic execution na úrovni x86, používaný hlavně pro bezpečnostní testování;
- **Z3** — SMT solver, který se používá pro řešení path constraints;
- **Java PathFinder (JPF)** — explicit-state model checker pro Java bytecode;
- **JDart** — nástroj pro concolic execution v Javě, založený na JPF;
- **Pex / jCUTE / Cloud9** — další nástroje pro symbolic nebo concolic testing.
