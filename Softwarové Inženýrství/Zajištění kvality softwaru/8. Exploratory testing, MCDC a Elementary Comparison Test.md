## 1. Exploratory testing

**Exploratory testing** je přístup, při kterém probíhá současně:
- učení se o systému;
- návrh testů;
- provádění testů;
- vyhodnocování výsledků.

Není to úplný chaos ani náhodné klikání. Tester aktivně zkoumá systém, sleduje jeho chování a nové poznatky hned používá k návrhu dalších testů. Test design tedy nevzniká celý předem, ale postupně během samotného testování.

---
## 2. Exploratory testing vs free testing

### Free testing
- tester volně zkouší systém;
- málo struktury;
- často se nezaznamenává historie;
- není jasné coverage.
### Exploratory testing
- adaptivní, ale disciplinovanější;
- tester si zaznamenává průběh a nálezy;
- používá zkušenosti a hypotézy;
- návrh testů vzniká během testování.

---
## 3. Kdy se exploratory testing hodí

Hodí se, když:
- chybí test basis;
- dokumentace je slabá;
- systém je nový;
- není čas na plnou přípravu testů;
- chceme rychle najít kritické problémy;
- testujeme usability nebo UI chování.

Nevhodné jako jediný přístup pro:
- safety-critical systémy;
- audity vyžadující prokazatelné coverage;
- situace, kde musí být testy plně reprodukovatelné;
- dlouhodobé regression suites.

---
## 4. Výhody a nevýhody exploratory testingu

Výhody:
- rychlé;
- flexibilní;
- dobré pro neznámý systém;
- najde netriviální problémy;
- využívá zkušenosti testerů.

Nevýhody:
- slabší coverage proof;
- závisí na zkušenosti testera;
- může být hůř reprodukovatelné;
- horší reporting;
- není náhrada systematických coverage kritérií.

## 5. MC/DC

**MC/DC** znamená **Modified Condition/Decision Coverage**. Je to technika pokrytí rozhodovací logiky, která se používá hlavně u důležitých nebo kritických systémů.

Základní princip je:

> Každá jednoduchá podmínka v rozhodnutí musí být otestována tak, aby bylo vidět, že sama nezávisle ovlivňuje výsledek celého rozhodnutí.

Nestačí tedy pouze to, že každá podmínka byla někdy `true` a někdy `false`. To by bylo jen slabší condition coverage. U MC/DC musí být ukázáno, že když změníme jednu konkrétní podmínku a ostatní podmínky necháme stejné, změní se také výsledek celého rozhodnutí.

Například pro rozhodnutí:

```text
A AND B
```

musíme ukázat, že:
- změna `A` může změnit výsledek celého výrazu;
- změna `B` může změnit výsledek celého výrazu.

Typicky se MC/DC snaží dosáhnout dobrého pokrytí rozhodovací logiky s menším počtem testů než úplné testování všech kombinací. Pro `n` jednoduchých podmínek se často uvádí minimálně přibližně `n + 1` testů, zatímco exhaustive testing by mělo `2^n` kombinací.

---

## 6. Neutral value

MC/DC používá koncept **neutral value**. Neutral value je hodnota ostatních podmínek, která „neblokuje“ vliv testované podmínky na výsledek celého výrazu.

Pro výraz:

```text
A AND B
```

je neutral value pro druhou podmínku `true`, protože:

```text
A AND true = A
```

To znamená, že pokud chceme testovat vliv podmínky `A`, nastavíme `B = true`. Potom změna `A` z `false` na `true` opravdu změní výsledek celého výrazu.

Příklad:

```text
A = false, B = true  → výsledek false
A = true,  B = true  → výsledek true
```

Tady je vidět, že právě `A` nezávisle ovlivňuje výsledek.

Pro výraz:

```text
A OR B
```

je neutral value `false`, protože:

```text
A OR false = A
```

Pokud chceme testovat vliv podmínky `A`, nastavíme `B = false`. Potom změna `A` z `false` na `true` změní výsledek celého výrazu.

Příklad:

```text
A = false, B = false → výsledek false
A = true,  B = false → výsledek true
```

Shrnutí:

```text
AND → neutral value je true
OR  → neutral value je false
```

Neutral value tedy pomáhá izolovat jednu konkrétní podmínku a ukázat, že právě ona má vliv na výsledek rozhodnutí.

## 7. MC/DC vs jiné coverage

| Kritérium                   | Co zajišťuje                              | Slabina                                              |
| --------------------------- | ----------------------------------------- | ---------------------------------------------------- |
| Condition coverage          | každá podmínka true/false                 | nemusí ovlivnit decision                             |
| Decision coverage           | decision má true/false výsledek           | nemusí testovat každou podmínku                      |
| Multiple condition coverage | kontrola všechn kombinací podmínek        | exponenciálně mnoho testů                            |
| MC/DC                       | každá podmínka nezávisle ovlivní decision | složitější návrh (hlavně u větších logických výrazů) |

> [!IMPORTANT]
> Počet testů:
> 
> ```text
> n podmínek → typicky n + 1 testů
> ```
> 
> Exhaustive testing:
> 
> ```text
> n podmínek → 2^n testů
> ```
> 

> [!NOTE]
> Kde se MC/DC používá
> 
> MC/DC je důležité pro:
> - avioniku;
> - zdravotnictví;
> - automotive;
> - finanční systémy;
> - safety-critical software;
> - složitou decision logic.
> 
> Ve slidech je zmiňován kontext DO-178B.

## 8. Elementary Comparison Test (ECT)

**Elementary Comparison Test (ECT)** je technika pro testování složitější rozhodovací logiky v procesu nebo workflow. Používá se tehdy, když systém neobsahuje jen jedno izolované rozhodnutí, ale více navazujících **decision points**, jejichž výsledky určují další cestu systémem.

ECT kombinuje dvě věci:
1. **testování rozhodnutí v jednotlivých decision points** — například pomocí MC/DC nebo jiné techniky coverage;
2. **path-based / workflow testing** — tedy skládání těchto rozhodnutí do celkových cest procesem.

Jinými slovy, ECT nejprve řeší, jak dobře otestovat jednotlivá rozhodnutí, a potom řeší, jak tato rozhodnutí poskládat do smysluplných testovacích scénářů přes celý proces.

### Jak ECT funguje

Nejprve identifikujeme všechny decision points v procesu. Pro každý decision point vytvoříme sadu lokálních test situations, například podle MC/DC. Každá **test situation** říká, jaké vstupní podmínky vedou k určitému výsledku rozhodnutí.

Potom určujeme, kam výsledek každého rozhodnutí vede ve workflow. Některý výsledek může pokračovat do dalšího decision pointu, jiný může proces ukončit. Nakonec se lokální test situations kombinují do celkových test cases tak, aby vznikly realizovatelné cesty systémem.

Důležité je také odstranit vzájemně vylučující se kombinace. Některé test situations totiž nemohou nastat ve stejném test case, protože si odporují ve vstupních datech nebo protože jedna větev workflow vůbec nevede k danému dalšímu rozhodnutí.

### Jednoduchý příklad

Mějme proces se dvěma rozhodnutími:

```text
A: Může uživatel pokračovat?
   true  → B
   false → End

B: Má uživatel nárok na bonus?
   true  → Bonus granted → End
   false → Bonus denied → End
```

Pro decision point **A** vytvoříme test situations, které pokryjí rozhodnutí, kdy uživatel může nebo nemůže pokračovat. Pro decision point **B** vytvoříme test situations, které pokryjí rozhodnutí o bonusu.

ECT potom tyto situace spojí do celkových test cases:

```text
Test 1: A = true  → B = true  → Bonus granted
Test 2: A = true  → B = false → Bonus denied
Test 3: A = false → End
```

Situace `A = false` se nekombinuje s rozhodnutím `B`, protože pokud A skončí false, workflow do bodu B vůbec nepokračuje.
