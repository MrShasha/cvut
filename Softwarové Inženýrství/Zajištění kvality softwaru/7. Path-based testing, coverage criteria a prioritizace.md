## 1. Princip path-based testingu

**Path-based testing** testuje systém pomocí cest přes model.

Cesta může být:
- cesta workflowem;
- cesta business procesem;
- cesta přes stavy systému;
- cesta control flow graph v kódu.

Hlavní myšlenka:

> Test case odpovídá cestě od startu do některého koncového bodu modelu.

---
## 2. Proč path-based testing

Manuální návrh testů pro větší workflow je obtížný:
- mohou vznikat duplicity;
- coverage není zaručená;
- některé cesty se snadno přehlédnou;
- při změně modelu se musí testy předělat;
- u mission-critical systémů je čistě manuální přístup slabý.

Path-based testing umožňuje:
- definovat model;
- zvolit coverage criterion;
- generovat test requirements;
- optimalizovat test set;
- systematicky řídit intenzitu testování.

---
## 3. Formální model systému

Základní varianta:

```text
G = (N, E)
```

kde:
- `N` = množina uzlů, typicky actions nebo decision points;
- `E` = množina hran, tedy transitions v SUT;
- `E ⊆ N × N`;
- existuje start node `ns ∈ N`;
- existuje množina end nodes `Ne ⊆ N`, `Ne ≠ ∅`.

Alternativní varianta z pohledu workflow:

```text
G = (D, A)
```

kde:
- `D` = decision points;
- `A` = actions mezi decision points;
- `A ⊆ D × D`;
- existuje start decision point;
- existuje neprázdná množina end decision points.

---
## 4. Test case jako cesta

Test case je sekvence uzlů a hran:

```text
d1, d2, ..., dn
```

s akcemi:

```text
a1, a2, ..., an-1
```

kde:

```text
ai = (di, di+1)
ai ∈ A
di ∈ D
d1 = start
dn ∈ end nodes
```

V test case se mohou opakovat:
- uzly;
- hrany;
- části cesty.

To je důležité u cyklů a workflow návratů.

---
## 5. Test coverage level

**Test coverage** vyjadřuje, do jaké míry test suite pokrývá zvolené coverage items.

Coverage items mohou být:
- uzly;
- hrany;
- dvojice hran;
- prime paths;
- test requirements;
- rozhodnutí;
- stavy.

> [!IMPORTANT]
> Coverage level určuje intenzitu testování.
> 
> Důležité:
> 
> > Intenzita testů je manažerské a rizikové rozhodnutí. Ne vždy je ekonomicky smysluplné chtít maximální coverage.
> 

---
## 6. Node coverage

**Node coverage** znamená, že každý uzel grafu je navštíven alespoň jednou.

Formálně:

```text
∀ n ∈ N: n se vyskytne v alespoň jednom test case
```

Výhoda:
- jednoduché;
- nízký počet testů;
- vhodné jako minimum.

Nevýhoda:
- slabé coverage;
- nepokrývá všechny přechody;
- může minout chyby v transitions.

---
## 7. Edge coverage

**Edge coverage** znamená, že každá hrana je vykonána alespoň jednou.

Formálně:

```text
∀ e ∈ E: e se vyskytne v alespoň jednom test case
```

Ve slidech odpovídá přibližně:

```text
TDL = 1
```

Výhoda:
- silnější než node coverage;
- pokrývá všechny přechody;
- vhodné pro smoke nebo low-priority actions.

Nevýhoda:
- nepokrývá kombinace předchozího a následujícího přechodu;
- některé sekvenční chyby zůstanou skryté.

---
## 8. Edge-pair coverage

**Edge-pair coverage** znamená, že každá dvojice po sobě jdoucích hran je pokryta alespoň jednou.

Jinak řečeno:

```text
incoming edge + outgoing edge přes decision point
```

Ve slidech odpovídá:

```text
TDL = 2
```

Výhoda:
- testuje lokální sekvence;
- dobrý kompromis intenzity;
- zachytí chyby závislé na předchozím kroku.

Nevýhoda:
- stále nemusí zachytit dlouhé závislosti v business workflow.

---
## 9. Test Depth Level (TDL)

TDL definuje hloubku lokálních cest, které musí být pokryty.

![[Assets/ZKS/Pasted image 20260530180201.png]]

### TDL = 1

Každá akce/hrana se vyskytne alespoň jednou v test setu.

```text
TDL=1 ≈ edge coverage
```

### TDL = 2

Pro každý decision point jsou pokryty všechny kombinace:

```text
příchozí hrana + následující odchozí hrana
```

```text
TDL=2 ≈ edge-pair coverage
```

### TDL = 3

Pro každý decision point jsou pokryty cesty:

```text
příchozí hrana + dvě následující hrany
```

Vyšší TDL znamená vyšší intenzitu, ale také více testů.

---
## 10. Prime path coverage

**Simple path** je cesta bez opakování uzlů, s výjimkou jednoduchého cyklu, kde první a poslední uzel mohou být stejné.

**Prime path** je simple path, která není vlastní podcestou žádné jiné simple path.

**Prime-path coverage** znamená:

> Každá prime path grafu musí být covered/toured alespoň jedním test case.

Důležité upřesnění:

> Ne každá prime path musí začínat a končit ve stejném uzlu. To platí pouze pro některé cykly. Obecná definice je: maximal simple path, která není proper subpath jiné simple path.

Výhody:
- silnější než edge-pair;
- praktičtější než all-paths;
- dobře pokrývá významné jednoduché cesty.

Nevýhody:
- může generovat více test requirements;
- u velkých grafů složitější.

---
## 11. All paths coverage

**All paths coverage** znamená pokrýt všechny možné cesty grafem.

Teoreticky nejsilnější, ale prakticky často nemožné.

Důvody:
- cykly;
- větvení;
- path explosion;
- vysoký počet testů;
- náklady.

> [!NOTE]
> Dobrá věta:
> 
> > All-paths coverage is theoretically strongest, but usually infeasible in realistic systems because of loops and path explosion.
> 

---
## 12. Test requirements

**Test requirements** jsou konkrétní prvky, které musí test set pokrýt.

Formálně:

```text
R = {r1, r2, ..., rn}
```

kde `ri` může být například:
- uzel;
- hrana;
- edge pair;
- prime path;
- konkrétní subpath.

Výhoda:
- sjednocuje problém generování testů;
- lze použít pro coverage i prioritizaci;
- podporuje divide & conquer.

---
## 13. Test set generation jako optimalizační problém

Cíl není jen splnit coverage, ale udělat to efektivně.

Optimalizační kritéria mohou být:
- počet test cases `|T|`;
- průměrná délka test case;
- celková délka test setu;
- rozptyl délek test cases;
- duplicity;
- priorita cest;
- proveditelnost v SUT.

Možné strategie:
- brute force;
- set-covering based solution;
- matching-based prefix graph solution;
- heuristiky.

---
## 14. Prioritizace workflow aktivit

V praktickém modelu lze použít **directed multigraph**, kde:
- paralelní hrany jsou povolené;
- hrany nebo uzly mohou mít prioritu;
- priority se často modelují barvením.

Typické priority:

```text
low / medium / high
```

Příklad:
- payment processing = high;
- invoice generation = high;
- optional newsletter step = low;
- profile update = medium.

---
## 15. Jak se priorita používá při generování testů

Generátor nemusí testovat všechny části stejně intenzivně.

Možný přístup:

| Priorita | Coverage                  |
| -------- | ------------------------- |
| low      | node nebo edge coverage   |
| medium   | edge-pair / TDL=2         |
| high     | prime-path nebo vyšší TDL |

Smysl:
- high-priority části dostanou silnější coverage;
- low-priority části dostanou slabší coverage;
- test effort se soustředí na riziko;
- odpovídá to risk-based testingu.

Dobrá formulace:

> Prioritization is modeled by assigning priorities or colors to graph nodes or edges. During test generation, high-priority workflow parts can require stronger coverage, while low-priority parts can be covered with weaker criteria.
