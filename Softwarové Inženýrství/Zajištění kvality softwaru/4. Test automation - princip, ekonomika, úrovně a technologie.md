## 1. Princip automatizovaného testování

**Test automation** znamená, že určité části testovacího procesu provádí nástroj nebo skript místo člověka.

Automatizovat lze například:
- přípravu test data;
- spuštění testů;
- ověření výsledků;
- reporting;
- regression testing;
- smoke testing;
- měření coverage;
- deployment validation.

Důležitá věta:

> Automatizace není cíl sama o sobě. Má smysl jen tehdy, když snižuje riziko, čas nebo náklady v porovnání s manuálním postupem.

---
## 2. Kdy se automatizace vyplatí

Automatizace se vyplatí hlavně, když:
- testy se opakují vícekrát;
- je potřeba rychlý regression feedback;
- testuje se více platforem;
- je potřeba smoke testing po každém buildu;
- jde o kritické části systému;
- existuje velké množství kombinací;
- test je stabilní a dobře specifikovaný;
- výsledek lze automaticky ověřit.

Typické pravidlo z přednášek:

> Test automation starts to make sense when tests are repeated several times, for example 3–5 times, but only if maintenance remains economically reasonable.

---
## 3. Ekonomika automatizace

![[Assets/ZKS/Pasted image 20260530134752.png|607]]

Automatizovaný test má obvykle:
- vyšší cenu přípravy;
- nižší cenu opakovaného spuštění;
- dodatečné náklady na údržbu.

Teorie:
```text
vyšší počáteční investice → levné opakované běhy → ROI
```

Realita:
```text
vyšší počáteční investice + maintenance cost → ROI není zaručené
```

Aby se automatizace vyplatila, je potřeba:
- snížit cenu tvorby automatizovaných testů;
- snížit cenu jejich údržby;
- neautomatizovat zbytečně všechno;
- vybrat stabilní a opakovatelné scénáře;
- testovat na nejnižší vhodné vrstvě.

> [!IMPORTANT]
> ## Co neautomatizovat slepě
> 
> Opatrnost je potřeba u testů, které jsou:
> - jednorázové;
> - často se měnící;
> - silně závislé na UI detailech;
> - obtížně vyhodnotitelné automatickým oraclem;
> - založené na lidském posouzení použitelnosti;
> - exploratory;
> - závislé na nestabilním prostředí.
>

## 4. Test automation pyramid

Zdravá strategie automatizace obvykle odpovídá test pyramid:

![[Assets/ZKS/Pasted image 20260530135049.png|285]]

```text
          UI / E2E tests        málo, drahé, pomalé
       integration/service      střední množství
          unit tests            hodně, rychlé, levné
```

V testovací pyramidě tvoří největší část **unit tests**. Jsou rychlé, levné, stabilní a mohou běžet velmi často, například při každém buildu nebo commitu. Testují malé jednotky systému izolovaně, typicky jednotlivé funkce, metody nebo třídy.

Nad nimi jsou **integration / service tests**, které ověřují spolupráci více komponent. Patří sem například API testy, testy integrace s databází, messaging, contract testing nebo ověření chování služby jako celku. Jsou pomalejší a dražší než unit testy, ale lépe ověřují, zda spolu jednotlivé části systému opravdu správně komunikují.

Na vrcholu pyramidy jsou **UI / E2E tests**. Těch má být nejméně, protože jsou nejpomalejší, nejdražší na údržbu a často křehčí vůči změnám uživatelského rozhraní nebo testovacího prostředí. Měly by proto pokrývat hlavně klíčové business scénáře, například přihlášení, vytvoření objednávky nebo dokončení platby.

Dobrá věta:

> Cílem není automatizovat všechno přes UI. Business logiku je lepší testovat na nejnižší vrstvě, která ještě pokrývá dané riziko.

## 5. Úrovně automatizace podle test levelu

Testy lze automatizovat na různých úrovních:

| Úroveň        | Co ověřuje                     | Příklady nástrojů / technologií                   |
| ------------- | ------------------------------ | ------------------------------------------------- |
| Unit          | metody, třídy, funkce          | JUnit, pytest, NUnit, Jest                        |
| Component     | komponenta izolovaně           | mocks, stubs, component test frameworks           |
| Integration   | spolupráce komponent           | Testcontainers, docker-compose, integration tests |
| API / Service | endpointy, kontrakty, odpovědi | Postman, Newman, REST Assured, SoapUI             |
| UI / FE       | uživatelské rozhraní           | Selenium, Playwright, Cypress, TestCafe           |
| E2E           | celý workflow                  | Playwright, Selenium, Cypress, Robot Framework    |
| Performance   | výkon, load, stress            | JMeter, Gatling, k6                               |
| Security      | zranitelnosti                  | SAST/DAST tools, OWASP ZAP                        |
| CI/CD gates   | build quality                  | linters, SonarQube, dependency scanners           |

> [!NOTE]
> ## Příklady nástrojů
> 
> ### UI testing
> - Selenium;
> - Playwright;
> - Cypress;
> - TestCafe;
> - Puppeteer;
> - Robot Framework.
> ### API testing
> - Postman;
> - Newman;
> - REST Assured;
> - SoapUI;
> - JMeter.
> ### Unit / integration
> - JUnit;
> - pytest;
> - Jest;
> - Testcontainers;
> - docker-compose test environments.
> ### Static analysis / quality gates
> - SonarQube;
> - PMD;
> - SpotBugs;
> - ESLint;
> - CodeQL;
> - Infer.

## 6. Přístupy k FE/UI automatizaci

| Přístup                         | Jednoduše řečeno                                         | Výhody                                                                    | Nevýhody                                                                                | Kdy se hodí                                                      |
| ------------------------------- | -------------------------------------------------------- | ------------------------------------------------------------------------- | --------------------------------------------------------------------------------------- | ---------------------------------------------------------------- |
| **Clicking automaton**          | Nástroj „kliká“ na obrazovce podobně jako člověk.        | Jednoduché na pochopení, použitelné i pro web i desktop.                  | Často nespolehlivé, citlivé na změny UI, rozlišení obrazovky nebo pozici prvků.         | Spíš pro jednoduché scénáře nebo starší aplikace.                |
| **Captured URL / HTTP traffic** | Test nepoužívá UI, ale opakuje zachycené HTTP požadavky. | Stabilnější a rychlejší než klikání přes UI.                              | Neověřuje skutečné GUI, jen komunikaci na síťové/API úrovni.                            | API/service testing, ověření request/response logiky.            |
| **JavaScript injection**        | Do stránky se vloží nebo spustí vlastní JavaScript kód.  | Použitelné pro web, umožňuje přímo pracovat se stránkou.                  | Může být citlivé na rozdíly mezi prohlížeči a méně odpovídá reálnému chování uživatele. | Speciální webové kontroly nebo pomocné automatizační akce.       |
| **Browser automation**          | Nástroj ovládá reálný nebo headless prohlížeč.           | Dobře simuluje reálné webové chování, vhodné pro E2E testy.               | Pomalejší než API testy, vyžaduje stabilní lokátory, správné čekání a izolované testy.  | Klíčové webové scénáře, například login, checkout, payment flow. |
| **OCR-based testing**           | Test čte text nebo prvky z obrazovky jako obrázek.       | Funguje i tam, kde není přístup k DOM, API nebo interním prvkům aplikace. | Velmi křehké, složité na údržbu, citlivé na vzhled, fonty a rozlišení.                  | Nouzová varianta, když nejde použít lepší technický přístup.     |

## 7. Test oracle

**Test oracle** je mechanismus, který určuje, zda test prošel nebo selhal.

Může být:
- očekávaný výsledek v test case;
- assertion v testu;
- porovnání s referenčním systémem;
- business pravidlo;
- snapshot;
- property-based assertion.

Problém:

> U velkého množství automatických testů může být náročné určit expected result pro každý test. Proto není automatizace vždy levná, i když samotné spuštění testu je automatické.
