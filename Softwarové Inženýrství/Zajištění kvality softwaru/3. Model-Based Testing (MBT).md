## 1. Princip MBT

**Model-Based Testing (MBT)** je přístup, ve kterém se testy negenerují ručně přímo z dokumentace, ale z modelu systému pod testem.

Základní proces:

```text
test basis → SUT model → test scenarios/test cases → test execution
```

![[Assets/ZKS/Pasted image 20260530132528.png|697]]

> [!NOTE]
> **SUT = System Under Test** - systém, komponenta nebo aplikace, která je právě předmětem testování.
> 
> Jinými slovy, SUT je ta část softwaru, u které ověřujeme, zda se chová správně podle požadavků. Může to být celý systém, jedna aplikace, konkrétní modul, API, služba nebo jen jedna funkce.

Důležité:

> MBT automatizuje hlavně test design / test generation. Není to totéž jako pouze automatické spouštění testů.

Test execution může být:
- manuální;
- automatizované;
- částečně automatizované.

> [!NOTE]
> ### Příklad MBT: checkout v e-shopu
> 
> Představme si, že testujeme objednávkový proces v e-shopu.
> 
> **Test basis** může být například požadavek:
> 
> > Uživatel může vložit produkt do košíku, zvolit dopravu, zvolit platbu a dokončit objednávku. Pokud platba selže, objednávka nesmí být potvrzena.
> 
> Z tohoto požadavku vytvoříme **SUT model**, například jednoduchý stavový / workflow model:
> 
> ```text
> Start
>   ↓
> Product selected
>   ↓
> Cart opened
>   ↓
> Shipping selected
>   ↓
> Payment selected
>   ↓
> Payment result?
>   ├── success → Order confirmed → End
>   └── failure → Payment error shown → End
> ```
> 
> Z tohoto modelu může MBT nástroj automaticky vygenerovat test scenarios například podle zvoleného coverage kritéria.
> 
> Například pro pokrytí všech hran modelu vzniknou testy:
> 
> ```text
> Test 1:
> Start → Product selected → Cart opened → Shipping selected → Payment selected
> → Payment success → Order confirmed → End
> 
> Test 2:
> Start → Product selected → Cart opened → Shipping selected → Payment selected
> → Payment failure → Payment error shown → End
> ```
> 
> Tyto testy potom můžeme provést:
> 
> - **manuálně** — tester projde kroky v aplikaci podle vygenerovaného scénáře;
>     
> - **automatizovaně** — scénář se převede například na Selenium / Playwright test;
>     
> - **částečně automatizovaně** — MBT nástroj vygeneruje test cases, ale jejich vykonání provede tester ručně.
>     
> 
> Důležité je, že tester nevymýšlí všechny scénáře ručně. Vytvoří model systému a test cases se generují systematicky podle modelu a coverage kritéria.

## 2. Manuální přístup vs. MBT

### Manuální přístup

![[Assets/ZKS/Pasted image 20260530132856.png|571]]

Proces:
1. přečíst dokumentaci;
2. zeptat se analytiků/vývojářů;
3. ručně vytvořit test scenarios/test cases;
4. ručně nebo automatizovaně je provést.

Výhody:
- nízké vstupní náklady;
- flexibilita;
- vhodné pro malé nebo rychle se měnící systémy;
- vhodné pro exploratory testing.

Nevýhody:
- pomalé škálování;
- vyšší riziko lidské chyby;
- duplicity;
- horší kontrola coverage;
- náročná údržba při změnách.
### MBT přístup

Proces:
1. přečíst test basis;
2. vytvořit model systému;
3. zvolit coverage criterion;
4. generovat testy z modelu;
5. provést testy.

Výhody:
- lepší kontrola coverage;
- méně duplicit;
- rychlejší generování test cases;
- částečně automatizovatelná impact analysis;
- vhodné pro intenzivní testování;
- změna coverage může vést k regenerování testů.

Nevýhody:
- nutná investice do modelu;
- nutná údržba modelu;
- model může být chybný nebo zastaralý;
- vyžaduje zkušenosti;
- nemusí se vyplatit pro malé nebo jednorázové projekty.

## 3. Porovnání MBT a manuálního návrhu testů

| Aspekt | MBT | Manuální test design |
|---|---|---|
| Vytvoření modelu | nutné | není nutné |
| Údržba | model + testy | hlavně testy |
| Generování testů | automatizované | ruční |
| Coverage control | silnější | omezená |
| Duplicity | lze redukovat algoritmicky | často vznikají ručně |
| Impact analysis | částečně automatizovatelná | většinou ruční |
| Vhodnost pro intenzivní testování | vysoká | slabší |
| Vstupní náklady | vyšší | nižší |

## 4. Motivace pro MBT

MBT se zavádí hlavně proto, že test manager chce vědět:
- co testujeme;
- jak intenzivně to testujeme;
- co netestujeme;
- zda jsou testy duplicitní;
- zda coverage odpovídá riziku;
- jak rychle upravit testy při změně systému.

Typická věta:

> MBT dává test coverage více pod kontrolu a umožňuje systematicky generovat test cases z modelu podle definovaných coverage kritérií.

---
## 5. Příklady modelů použitelných v MBT

MBT může používat různé modely:
- state machine model;
- workflow model;
- graph-based model;
- UML activity diagram;
- UML state diagram;
- BPMN process model;
- decision table;
- control flow graph;
- finite automaton;
- API model;
- business process model.

Pro zkoušku stačí říct:

> In MBT, the model can be for example a state machine, workflow graph, UML activity diagram, BPMN process model or control flow graph.

## 6. Vztah MBT a test automation

MBT a test automation spolu souvisí, ale nejsou totéž.
### MBT

Automatizuje nebo podporuje:
- návrh testů;
- generování test scenarios;
- výběr test paths;
- coverage calculation;
- optimalizaci test setu.
### Test automation

Automatizuje hlavně:
- spuštění testů;
- porovnání actual vs expected result;
- reporting;
- regression testing;
- CI/CD feedback.

Možné kombinace:

| Situace                     | Příklad                              |
| --------------------------- | ------------------------------------ |
| MBT bez automatické exekuce | model vygeneruje manuální test cases |
| automatizace bez MBT        | ručně napsané Selenium testy v CI    |
| MBT + automatizace          | model vygeneruje executable tests    |

---
## 7. Kdy je MBT vhodné

MBT se hodí hlavně pro:
- mission-critical systémy;
- důležité části systému;
- komplexní workflow;
- dlouhodobě udržované produkty;
- systémy s vysokým požadavkem na coverage;
- systémy s dobře modelovatelným chováním;
- regresní testování.

Méně vhodné je pro:
- malé jednorázové projekty;
- velmi rychle se měnící systém bez stabilního modelu;
- projekty bez času na modelování;
- nekritické funkce s nízkým rizikem.

---
## 8. Typická rizika MBT

- Model neodpovídá realitě.
- Model je příliš detailní a drahý na údržbu.
- Model je příliš abstraktní a negeneruje užitečné testy.
- Generované testy mohou být formálně správné, ale businessově málo hodnotné.
- Tým se může soustředit na model a zapomenout na realitu uživatelského chování.
