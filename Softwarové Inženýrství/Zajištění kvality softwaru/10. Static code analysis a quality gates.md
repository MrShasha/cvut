## 1. Co je static code analysis

**Static code analysis** je analýza zdrojového kódu bez jeho spuštění.

Cíl:
- najít chyby před runtime;
- najít security weaknesses;
- ověřit coding standards;
- měřit kvalitu kódu;
- snížit technický dluh;
- zlepšit kvalitu před unit testingem.

Je to specifická podmnožina static testingu.

---
## 2. Static analysis vs dynamic testing

| Kritérium | Static analysis | Dynamic testing |
|---|---|---|
| Spuštění programu | ne | ano |
| Vstupní data | nejsou nutná | nutná |
| Co hledá | potenciální chyby ve struktuře/kódu | skutečné chování při běhu |
| Výstup | warnings, findings, metrics | passed/failed tests, failures |
| Limity | false positives, neověří funkční chování | pokrývá jen vykonané cesty |

> [!NOTE]
> 
> Důležité:
> 
> > Static analysis and dynamic testing are complementary. Static analysis does not replace functional testing.

## 3. Co static analysis umí najít

Příklady:
- null pointer risks;
- resource leaks;
- SQL injection patterns;
- unsafe casts;
- dead code;
- duplicate code;
- unreachable code;
- high cyclomatic complexity;
- coding standard violations;
- security vulnerabilities;
- dependency risks;
- suspicious control flow;
- chybějící closing resources;
- špatné exception handling.

---
## 4. Jak static analyzers fungují

Typický pipeline:

```text
source code → parser/lexer → AST → internal representation → analysis rules → report
```
### AST

**Abstract Syntax Tree** reprezentuje syntaktickou strukturu kódu.

Vhodné pro:
- hledání syntaktických patternů;
- naming rules;
- coding style;
- jednoduchá pravidla.
### CFG

**Control Flow Graph** reprezentuje možné cesty provedení programu.

Vhodné pro:
- analýzu větvení;
- resource handling;
- unreachable code;
- path-sensitive pravidla;
- složitější tokové chyby.
### Data-flow analysis

Sleduje, jak se hodnoty šíří programem.

Použití:
- taint analysis;
- security;
- nullness analysis;
- použití neinicializovaných proměnných.

---
## 5. Rules a findings

Static analyzers pracují s pravidly.

Příklad pravidla:

```text
Nepoužívej prázdný catch blok.
```

Finding obvykle obsahuje:
- soubor;
- řádek;
- popis problému;
- severity;
- doporučený fix;
- rule ID;
- někdy flow trace.

---
## 6. False positives a false negatives

### False positive

Nástroj hlásí problém, ale ve skutečnosti nejde o defekt.

Problém:
- zvyšuje náklady na triage;
- může vést k ignorování nástroje;
- zvlášť u velkých codebases.
### False negative

Nástroj problém nenajde, i když v kódu existuje.

Důvod:
- pravidlo není implementované;
- příliš složitý tok;
- dynamické vlastnosti jazyka;
- nedostatečný model prostředí.

---
## 7. Nástroje

Příklady:

- **PMD** — Java a další jazyky, custom rules;
- **SpotBugs** — bug patterns v Java bytecode;
- **SonarQube** — quality gates, technical debt, security, reliability;
- **ESLint** — JavaScript/TypeScript linting;
- **Infer** — Facebook/Meta static analyzer;
- **CodeQL** — query-based code analysis;
- **IDE inspections** — IntelliJ, Visual Studio, etc.

---
## 8. Code metrics

Static analysis často měří:
- cyclomatic complexity;
- maintainability index;
- lines of code;
- coupling;
- cohesion;
- depth of inheritance;
- duplication;
- test coverage integration;
- technical debt.
### Cyclomatic complexity

Vyjadřuje počet nezávislých cest v metodě/funkci.

Vyšší hodnota znamená:
- těžší testování;
- horší čitelnost;
- vyšší riziko chyb;
- vhodný kandidát na refactoring.

---
## 9. Quality gates

**Quality gate** je sada podmínek, které musí projekt splnit, aby mohl pokračovat v pipeline.

Příklady gate conditions:
- build passes;
- unit tests pass;
- coverage >= threshold;
- no critical vulnerabilities;
- no blocker findings;
- complexity below threshold;
- no new critical code smells;
- dependency scan clean.

Smysl:

> Automatizovaně zabránit tomu, aby se špatná kvalita dostala do dalších fází.

---
## 10. Static analysis v CI/CD

Typický workflow:

```text
commit / merge request
→ build
→ unit tests
→ static analysis
→ security scan
→ quality gate
→ merge/deploy allowed or blocked
```

Výhody:
- rychlá zpětná vazba;
- jednotná pravidla;
- méně ruční kontroly;
- auditovatelný proces;
- podpora shift-left.

---
## 11. Limity static code analysis

Static analysis neumí plně ověřit:
- zda software splňuje business požadavky;
- reálné runtime chování;
- uživatelskou použitelnost;
- integraci se skutečným prostředím;
- všechny performance problémy;
- všechny chyby závislé na datech.

Proto se kombinuje s:
- unit tests;
- integration tests;
- E2E tests;
- reviews;
- security testing;
- monitoringem.
