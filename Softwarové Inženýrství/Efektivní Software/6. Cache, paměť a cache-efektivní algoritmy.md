# Proč je DRAM pomalá

## SRAM vs. DRAM — fyzikální trade-off

**SRAM (Static RAM):**
- 6 transistorů na bit → drahé, velké, ale rychlé.
- Neztratí hodnotu bez obnovy.
- Latence: < 1 ns.
- Použití: registry, L1/L2/L3 cache, TLBs.

**DRAM (Dynamic RAM):**
- 1 transistor + 1 kondenzátor na bit → levné, husté, ale pomalé.
- **Čtení ničí hodnotu** → po každém čtení musí refresh (write-after-read).
- Nabíjení/vybíjení kondenzátoru není instantní → fyzikální limit.
- Latence: ~60–100 ns (200 cyklů na 3 GHz CPU).
- Použití: hlavní paměť.

**Fyzikální analogie:** SRAM je jako světelný přepínač — okamžitá odezva. DRAM je jako water balloon — musíš ho naplnit a pak ho "zmáčknout". Naplňování trvá.

## DRAM latence — konkrétní čísla

```
Přístup do L1 cache:    4 cykly     (~1 ns)
Přístup do L2 cache:    12 cyklů    (~4 ns)
Přístup do L3 cache:    40 cyklů    (~14 ns)
Přístup do DRAM:        200 cyklů   (~67 ns)
```

CPU při cache miss čeká. Moderní CPU je tak rychlé, že tato čekání jsou absolutní bottleneck pro většinu programů.

# Architektura cache a typy cache miss

## Cache line — základní přenosová jednotka

Cache nepracuje s jednotlivými bajty — pracuje s **cache lines** o velikosti typicky **64 bajtů**. Při cache miss se z DRAM načte celých 64 bajtů najednou.

**Implikace pro programátora:**
- Přístup k `array[0]` načte do cache `array[0]` až `array[15]` (pro int32). Přístup k `array[1]` je pak z cache → zdarma.
- Proto je **sequential access** (array traversal) tak rychlý.
- Proto je **random access** (linked list traversal, hash table) tak pomalý.

## Associativita cache

Cache je organizována do **setů** a **způů** (ways):
- **Direct-mapped** (1-way): každá adresa mapuje na právě jeden cache slot → vysoký výskyt conflict miss.
- **N-way set-associative** (typicky 4–16 way): adresa může být v N různých slotech v daném setu → méně conflict miss.
- **Fully associative**: libovolný slot → optimální hit rate, ale drahé hardware lookup.

Moderní CPU: L1 typicky 8-way, L2 typicky 16-way, L3 typicky 12–24 way.

## Tři typy cache miss

### Cold miss (compulsory miss)
Přistupuješ na data, která **nikdy nebyla v cache**. Nevyhnutelné při prvním přístupu.
**Řešení:** Prefetching (hardware nebo software) — načti data do cache předem, než je potřebuješ.

### Capacity miss
Cache je **příliš malá** pro tvoje working set. Data jsou opakovaně vyháněna (eviction) před dalším přístupem.
**Řešení:** Tiling/blocking (viz níže) — rozděl výpočet na bloky, které se vejdou do cache.

### Conflict miss
Dvě různé adresy mapují na **stejný cache set**, i když je v cache volné místo jinde. Způsobeno nevhodnou associativitou.
**Řešení:** Padding (zarovnání datových struktur) nebo změna přístupového vzoru.

# Datové struktury a dynamická alokace

## Linked list vs. array — proč linked list zabíjí cache

```
Linked list:
[data|ptr] → [data|ptr] → [data|ptr] → ...
  (náhodné adresy v paměti — každý uzel = cache miss)

Array:
[d0][d1][d2][d3][d4][d5][d6][d7]...
  (kontinuální → jeden cache miss načte 16 prvků najednou)
```

**Linked list traversal:** každý uzel = potenciální cache miss = ~200 cyklů.
**Array traversal:** jeden cache miss na 16 prvků (int32) = ~200/16 = 12 cyklů amortizovaně.

**Dopad:** Linked list traversal je typicky 3–10× pomalejší než array traversal.

## Data-oriented design (DOD)

Místo OOP přístupu (každý objekt = jeden heap allocation):

```cpp
// AOS (Array of Structures) — špatné pro cache:
struct Entity { float x, y, z; int health; float velocity; };
Entity entities[10000];  // → x, y, z, health, velocity prokládány

// SOA (Structure of Arrays) — dobré pro cache (pokud zpracováváš jen x,y,z):
float x[10000], y[10000], z[10000];
int health[10000];
float velocity[10000];
// → při zpracování pozic: čistý sequential access přes x[], y[], z[]
```

**Kdy SOA pomáhá:** SIMD vektorizace, zpracování jen části atributů (fyzikální engine zpracovává jen pozice a rychlosti, ne health).

## Dynamická alokace — fragmentace a GC pressure

- `malloc`/`free` fragmentuje haldu → vzdálené kusy paměti → cache miss.
- Preferuj **pool allocator** (arena) — alokuj objekty z kontinuálního bloku.
- Preferuj **value types** (stack allocation, inline structs) nad heap allocation, kde to jde.

# Příklad: násobení matic

## Naivní implementace — ignoruje cache

```c
for (int i = 0; i < N; i++)
  for (int j = 0; j < N; j++)
    for (int k = 0; k < N; k++)
      C[i][j] += A[i][k] * B[k][j];  // B[k][j] = sloupcový přístup!
```

**Problém:** `B[k][j]` přistupuje po sloupcích (column-major), zatímco C/C++ ukládá matice po řádcích (row-major). Každý přístup `B[k][j]` pro různá `k` je na jiné cache line → capacity miss storm.

## Tiled (blocked) implementace — cache-efektivní

```c
int BLOCK = 64;  // velikost bloku: BLOCK × BLOCK floatů × 4 B = 16 KB (vejde se do L1)
for (int ii = 0; ii < N; ii += BLOCK)
  for (int jj = 0; jj < N; jj += BLOCK)
    for (int kk = 0; kk < N; kk += BLOCK)
      for (int i = ii; i < min(ii+BLOCK, N); i++)
        for (int j = jj; j < min(jj+BLOCK, N); j++)
          for (int k = kk; k < min(kk+BLOCK, N); k++)
            C[i][j] += A[i][k] * B[k][j];
```

**Proč to funguje:** Blok BLOCK×BLOCK se vejde do L1 cache. Vnitřní smyčky pracují výhradně s daty v cache → žádné capacity missy.

**Fyzikální analogie:** Místo abys prohlížel celou knihovnu stránku po stránce z různých knih najednou, vezmeš si pět knih na stůl (cache) a přečteš je celé, pak je vyměníš za dalších pět. Mnohem méně cest k policím.

**Výkon:** Tiled násobení matic je typicky **3–10× rychlejší** než naivní implementace pro velké N.

# Multi-core cache: coherency, false sharing, NUMA

## Cache coherency — problém sdílených dat

Každé jádro má **vlastní L1 a L2 cache**. Pokud dvě jádra modifikují stejná data:
- Jádro A zapíše hodnotu do své L1 → svá L1 je aktualizovaná.
- Jádro B stále vidí starou hodnotu ve své L1 → **nekoherence**!

**MESI protokol** — hardware řeší koherenci pomocí stavů cache line:
- **M** (Modified) — linka je jen v tomto jádru a je modifikovaná.
- **E** (Exclusive) — linka je jen v tomto jádru a je čistá.
- **S** (Shared) — linka je v více jádrech, čistá.
- **I** (Invalid) — linka je neplatná, musí se znovu načíst.

**Overhead:** Přechod stavu (M→I pro jiné jádro) = invalidace + reload = ~200+ ns.

## False sharing — zákeřný bug výkonu

**False sharing** nastane, když dvě vlákna modifikují **různé** proměnné, které leží na **stejné cache line**.

```c
struct {
    int counter_a;  // vlákno 1 modifikuje
    int counter_b;  // vlákno 2 modifikuje
    // obě jsou na stejné cache line (64 B)!
} shared;
```

I přestože vlákna nesoupeří o data logicky, MESI protokol posílá linku neustále tam a zpět → dramatické zpomalení.

**Řešení:** Padding na hranici cache line:
```c
struct {
    int counter_a;
    char padding[60];  // zarovnání na 64 B
    int counter_b;
};
```

Nebo v C++ `alignas(64)`.

**Fyzikální analogie:** Dvě osoby sedí u stejného stolu, ale každá pracuje na jiném papíru. Kdykoliv jeden papír potřebuje přemístit, celý stůl musí být vyklizen. Řešení: každý dostane vlastní stůl.

## NUMA — Non-Uniform Memory Access

Na serverech s více sockety (multi-socket systems) má každý socket vlastní RAM. Přístup k RAM vlastního socketu je rychlý (local), přístup k RAM jiného socketu je pomalý (remote, ~2× latence).

**Implikace:**
- Alokace paměti by měla být na stejném uzlu jako zpracovávající vlákno.
- `numactl --localalloc ./program` — alokuje paměť na lokálním NUMA uzlu.
- `perf stat -e NUMA-miss` — detekce NUMA remote přístupů.

**V kódu:**
- Pinni vlákna na konkrétní jádra (`pthread_setaffinity_np`).
- Alokuj paměť blízko vláknu, které ji bude používat.

> Ke státnicím: Cache line = 64 B (základní přenosová jednotka). Cold/capacity/conflict miss — různé příčiny, různá řešení. Linked list = cache miss hell. SOA > AOS pro SIMD. Tiling = výpočet v blocích vejdoucích se do L1. MESI = hardware koherence. False sharing = dvě vlákna na jedné cache line → řeš paddingem. NUMA = vzdálená paměť = 2× latence → alokuj lokálně.
