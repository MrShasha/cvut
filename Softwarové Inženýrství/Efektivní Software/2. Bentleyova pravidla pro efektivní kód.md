# Optimalizace datových struktur

## Čtyři základní strategie

Datová struktura je víc než "kde jsou data uložena" — je to **kontrakt mezi algoritmem a hardwarem** o tom, jak data přistupovat. Špatná volba struktury může způsobit 10–100× zpomalení bez jediné špatné algoritme.

### 1. Reduce Bits (Packing) — šetři místo

**Princip:** Zmenši data tak, aby se jich vešlo více do jedné cache line (64 B).

- **Encoding:** Data zapiš s méně bity (datum jako `int32` místo `string`).
- **Packing:** Ulož více malých hodnot do jednoho strojového slova (bit fields, packed structs).

**Fyzikální analogie:** Cache line je kontejner. Pokud má každý balík půl kontejneru volného prostoru, můžeš vozit dvojnásobek balíků najednou, nebo platit za půl kontejnerů.

**Kdy použít:** Read-heavy workload, velká pole dat, network packet processing.

**Kdy ne:** Write-heavy workload, kde packing přidává overhead extrakce bitů.

### 2. Spend Memory (Augmentation) — přidej redundanci

**Princip:** Přidej redundantní data, aby byly běžné operace rychlejší.

**Příklad:** Linked list bez pole `size` → počítání délky je O(N). Přidej `size` pole → O(1). Zaplatíš 4–8 B navíc, ušetříš traversal.

**Další příklady:**
- Skip list — víceúrovňové propojení pro rychlejší search.
- B-tree vs. binary tree — víc klíčů v uzlu = méně úrovní = méně cache miss.
- Denormalizace v databázi — duplikovaná data pro rychlejší čtení.

**Fyzikální analogie:** Navigační mapa s vyznačenými dálnicemi — redundantní informace (cesty jsou i v detailní mapě), ale orientace je rychlejší.

### 3. Spend Time (Precomputation) — předpočítej výsledky

**Princip:** Spočítej výsledky dopředu a ulož je — lookup table místo výpočtu za běhu.

**Příklady:**
- Lookup table pro sinus/cosinus (přesné hodnoty ve stupních).
- Počítání bitů (popcount) — lookup table pro každý bajt, kombinuj pro 64-bit číslo.
- Prefix sum array — předpočítané kumulativní součty pro O(1) range queries.

**Trade-off:** Startup čas + paměť vs. runtime výkon. Kritické u real-time systémů, kde latence spuštění nezáleží, ale odpovědní latence ano.

### 4. Avoid Work (Sparsity/Caching) — nepočítej zbytečně

**Princip:** Neskladuj co tam není, neřpočítej co už znáš.

- **Sparsity:** Compressed Row Storage (CSR) pro řídké matice — místo 1000×1000 matice s 1 % nenulových prvků ulož jen nenulové.
- **Caching/Memoization:** Ulož výsledky drahých výpočtů (LRU cache, memoize rekurzivní funkce).
- **Lazy evaluation:** Proveď výpočet jen pokud je výsledek skutečně potřeba.

> [!note] Ke státnicím
> Čtyři strategie: Packing (méně bitů, víc do cache line), Augmentation (redundantní data pro rychlejší operace), Precomputation (lookup tables), Sparsity/Caching (nepočítej zbytečně). Vždy trade-off paměť vs. čas.

# Optimalizace logiky

## Short-circuit evaluation

V `if (A && B)` se B nevyhodnotí, pokud A je false. Pokud je A levný test a B drahý výpočet → dej A na první místo.

**Pravidlo:** Levné podmínky (integer comparison) dávej před drahé (volání funkce, přístup do paměti).

## Reordering conditions

Pokud máš sérii nezávislých podmínek, přeuspořádej je podle kombinace:
- **Pravděpodobnosti selekce** — podmínky, které nejčastěji selžou, dávej první (short-circuit).
- **Ceny vyhodnocení** — levné podmínky dávej první.

**Příklad:** Validace vstupu — nejdřív ověř, zda je číslo v rozsahu (levné), pak teprve zda prvočíslo (drahé).

## Loop unswitching

Místo:
```c
for (int i = 0; i < N; i++) {
    if (condition) { /* jedna větev */ }
    else { /* druhá větev */ }
}
```
Přepiš na:
```c
if (condition) {
    for (int i = 0; i < N; i++) { /* jedna větev */ }
} else {
    for (int i = 0; i < N; i++) { /* druhá větev */ }
}
```
**Proč:** Podmínka se vyhodnocuje N-krát místo 1×. Branch predictor nemusí hádat. LLVM to někdy udělá automaticky, ale ne vždy.

# Optimalizace smyček

## Loop unrolling

Místo:
```c
for (int i = 0; i < N; i++) { a[i] += b[i]; }
```
Přepiš na:
```c
for (int i = 0; i < N; i += 4) {
    a[i]   += b[i];
    a[i+1] += b[i+1];
    a[i+2] += b[i+2];
    a[i+3] += b[i+3];
}
```

**Proč pomáhá:**
- Méně testů podmínky smyčky a méně inkrementací (`i++`).
- Větší instrukční okno pro out-of-order execution → lepší ILP.
- Překladač (LLVM) toto dělá automaticky (`-O2`, `-O3`). Manuální unrolling je jen pro případy, kde překladač selhá.

## Loop fusion

Spoj dvě smyčky přes stejná data do jedné:

```c
// Před (2 průchody přes pole):
for (int i = 0; i < N; i++) { a[i] = f(i); }
for (int i = 0; i < N; i++) { b[i] = g(a[i]); }

// Po (1 průchod):
for (int i = 0; i < N; i++) { a[i] = f(i); b[i] = g(a[i]); }
```

**Proč:** Data zůstanou v cache mezi výpočty `f` a `g`. Jinak druhá smyčka znovu načte celé pole z DRAM (eviction).

## Loop tiling / blocking

Pro 2D operace (matice, obrázky) rozděl smyčku na bloky, které se vejdou do cache:

```c
// Klasicky (špatné pro cache):
for (int i = 0; i < N; i++)
    for (int j = 0; j < N; j++)
        C[i][j] += A[i][k] * B[k][j];

// Tiled (cache-friendly):
for (int ii = 0; ii < N; ii += BLOCK)
  for (int jj = 0; jj < N; jj += BLOCK)
    for (int kk = 0; kk < N; kk += BLOCK)
      for (int i = ii; i < ii+BLOCK; i++)
        for (int j = jj; j < jj+BLOCK; j++)
          for (int k = kk; k < kk+BLOCK; k++)
            C[i][j] += A[i][k] * B[k][j];
```

**Detaily** jsou v kapitole o cache → [[6. Cache, paměť a cache-efektivní algoritmy#Příklad: násobení matic|Příklad: násobení matic]].

# Optimalizace funkcí

## Function inlining

Volání funkce má overhead: uložení registrů, skok na adresu, nastavení stack frame, návrat. Pro malé funkce může být overhead větší než samotný kód.

**Inlining** = překladač zkopíruje tělo funkce přímo na místo volání. Eliminuje overhead + umožňuje další optimalizace (constant folding, dead code elimination v kontextu volání).

```c
// Překladač může inlinovat:
inline int max(int a, int b) { return a > b ? a : b; }
```

**Kdy je inlining nebezpečný:** Velké funkce volané na mnoha místech → code bloat → větší binárka → instrukční cache miss.

**LTO (Link-Time Optimization):** Moderní překladače inlinují přes hranice kompilačních jednotek díky LTO.

## Tail call optimization

Rekurzivní volání na konci funkce (tail call) lze přeložit jako skok místo nového stack frame → O(N) stack → O(1) stack.

```c
// Tail recursive factorial (překladač optimalizuje):
int fact(int n, int acc) {
    if (n == 0) return acc;
    return fact(n - 1, n * acc); // tail call
}
```

## Coarsening (granularity)

Rozděl práci na hrubší granuly — méně volání funkcí, méně synchronizace (v paralelním kontextu). Příklad: místo rekurzivního mergesort do hloubky 1 přepni na insertion sort pro malá pole.

> Ke státnicím: Bentleyova pravidla = systematický toolbox. Datové struktury: pack, augment, precompute, avoid work. Smyčky: unroll, fuse, tile. Funkce: inline, tail call. Vždy měř — překladač dělá hodně automaticky.
