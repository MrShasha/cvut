# Motivace a definice efektivního softwaru

**Efektivní software** = software, který využívá dostupné zdroje (CPU, paměť, energie, síť) efektivně — tedy vytěží z hardwaru co nejvíce při co nejnižších nákladech.

Proč na tom záleží? Dva pohledy:

**Výkon (Performance):**
- **Latence** (response time) — jak rychle dostanu odpověď? Záleží u real-time systémů, her, lékařských přístrojů.
- **Propustnost** (throughput) — kolik práce udělám za jednotku času? Záleží u serverů, datových pipeline, dávkového zpracování.

**Náklady (Cost):**
- CPU čas, paměť, šířka pásma, energie = **peníze ve velkém měřítku**. Amazon snížil latenci o 100 ms a tržby vzrostly o 1 %. Google zjistil, že zpomalení o 500 ms sníží traffic o 20 %.
- Na embedded a mobilních zařízeních je energie fyzické omezení, ne jen cena.

**Omezení (Constraints):**
- Vestavěné systémy, mobilní zařízení, real-time deadlines — musíš se vejít do limitovaného hardwaru.

> [!note] Inženýrský pohled
> Optimalizace není o eleganci — je o měřitelném výsledku. Nikdy neoptimalizuj bez benchmarku. Nikdy neutrácej čas na kód, který není bottleneck.

# Optimalizační přístup a technologický stack

## Iterativní cyklus výkonnostní optimalizace

Jediný správný přístup je empirický a iterativní:

```
1. MĚŘ    → najdi co je pomalé (profiler, benchmarky)
2. ROZUMĚJ → zjisti PROČ je to pomalé (cache miss? lock contention? I/O?)
3. ZMĚŇ   → oprav jednu věc najednou
4. OVĚŘ   → změřil jsi zlepšení? Nezavedl jsi regresi?
```

**Proč opravovat jen jednu věc najednou?** Protože jinak nevíš, která změna co způsobila. Vědecká metoda platí i v softwarovém inženýrství.

## Technologický zásobník

Každá vrstva zásobníku může být bottleneck:

```
Tvoje aplikace
  ↓ Knihovny
  ↓ Překladač (C/C++) nebo JIT (JVM)
  ↓ Runtime / GC / JIT kompilátor
  ↓ Operační systém (scheduler, virtual memory, I/O)
  ↓ Virtualizace (KVM, Docker)
  ↓ Hardware (pipeline, caches, superscalar, SIMD)
```

**Lidský překlad:** Než začneš optimalizovat svůj kód, ujisti se, že problém není v jiné vrstvě. Pomalá databáze? Špatné indexy? GC pause v JVM? Špatná konfigurace OS? Virtualizační overhead?

# Hardware: procesory a paměťová hierarchie

## Paměťová hierarchie — fyzická realita

Paměť není flat array. Je to pyramida rychlosti vs. kapacity:

| Vrstva | Typická latence | Kapacita |
|---|---|---|
| Registry | < 1 ns | desítky B |
| L1 cache | ~4 cykly (~1–2 ns) | 32–64 KB |
| L2 cache | ~12 cykly (~3–5 ns) | 256 KB – 1 MB |
| L3 cache | ~40 cykly (~10–20 ns) | 8–64 MB |
| DRAM | ~200 cykly (~60–100 ns) | GBs |
| SSD | ~100 000 ns | TBs |
| HDD | ~10 000 000 ns | TBs |

**Fyzikální analogie:** Představ si, že programátor stojí v centru Prahy (CPU). Registry jsou věci v kapse — dosáhneš okamžitě. L1 cache je v místnosti. L2 je v budově. L3 je v sousední ulici. DRAM je na druhém konci města. Disk je v jiném státě.

**Kritická implikace:** Přístup do DRAM je 200x pomalejší než L1 cache. Pokud tvůj algoritmus "mine" do DRAM, CPU čeká. Moderní CPU je tak rychlé, že je paměť absolutní bottleneck pro většinu neoptimalizovaných programů.

## Cache line — základní jednotka přenosu

Cache pracuje v blocích nazývaných **cache line** (typicky **64 bajtů**). Při čtení jednoho bajtu z DRAM se do cache načte celých 64 bajtů.

**Implikace pro programátora:**
- Data, která jsou v paměti blízko sebe, jsou přečtena najednou → spatial locality.
- Data, ke kterým přistupuješ opakovaně, zůstanou v cache → temporal locality.
- Linked list = katastrofa pro cache (každý uzel je jinde v paměti).
- Array = ráj pro cache (kontinuální data, sequential access pattern).

# Instruction-Level Parallelism (ILP)

## Superskalar a pipeline

Moderní CPU nevykonává instrukce sekvenčně — vykonává jich **více najednou** (superscalar) a překrývá jejich fáze (pipeline).

**Pipeline** — instrukce prochází fázemi (Fetch → Decode → Execute → Write-back). Zatímco první instrukce je v Decode, druhá může být ve Fetch.

**Superscalar** — CPU má více výpočetních jednotek (ALU, FPU, load/store). Může vykonávat 4–6 instrukcí za takt.

## Out-of-order execution

CPU přehazuje pořadí instrukcí za účelem maximálního vytížení výpočetních jednotek. Výsledky jsou ale doručeny ve správném pořadí (reorder buffer).

**Implikace:** Závislosti mezi instrukcemi (data hazards) způsobují stalls. Kód s dlouhými dependency chains je pomalý i přes OOO.

## Branch prediction

Podmíněné skoky (if/else, smyčky) způsobují problém pipeline — CPU neví, jakou větev provést. **Branch predictor** odhaduje výsledek a spekulativně vykonává instrukce.

**Misprediction** = zahozeni spekulativně vykonaných instrukcí = ztráta ~15–20 cyklů.

**Implikace:** Kód s neprediktabilnými větvemi (např. náhodná data) bude pomalý. Branchless programming a SIMD jsou nástroje, jak se branch prediction problémům vyhnout.

# Task parallelism a vlákna

## Kdy pomáhají vlákna

Vlákna pomáhají, pokud je bottleneck **CPU-bound** a problém lze rozdělit na nezávislé části.

**Amdahlův zákon** — pokud je $p$ část kódu paralelizovatelná, maximální zrychlení s $n$ procesory je:
$$S(n) = \frac{1}{(1-p) + \frac{p}{n}}$$

**Lidský překlad:** Pokud 10 % kódu je sekvenční, maximální zrychlení je 10×, bez ohledu na počet jader. Sekvenční bottleneck (inicializace, I/O, logging) limituje vše.

## Kdy vlákna nepomáhají (nebo škodí)

- **Memory-bound kód** — přidáváš jádra, ale všechna čekají na DRAM. Přidáním jader problém s pamětí nevyřešíš.
- **I/O-bound kód** — čekáš na síť nebo disk. Asynchronní I/O (non-blocking) je lepší řešení než vlákna.
- **Lock contention** — příliš jemná synchronizace → vlákna si překáží → výkon klesá s přidáváním jader.

> Ke státnicím: Efektivní software = latence + throughput + náklady. Paměťová hierarchie je klíčový bottleneck. ILP (superskalar, pipeline, OOO, branch prediction) je důvod, proč moderní CPU jsou tak rychlé — a proč je těžké je vytížit. Amdahlův zákon říká, proč paralelismus není stříbrná kulka.
