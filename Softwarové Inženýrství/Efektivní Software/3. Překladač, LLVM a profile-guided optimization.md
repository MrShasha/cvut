Znalost těchto hranic ti umožní psát kód, který překladač dokáže plně optimalizovat — a vědět, kdy musíš optimalizovat ručně.

# Struktura překladače a AST

## CPU jako hardware interpreter

CPU je v podstatě hardwarový interpret — přijímá binárně zakódované instrukce a vykonává je. Překlad ze zdrojového kódu na strojový kód probíhá ve fázích:

```
Zdrojový kód (.c, .cpp)
    ↓ Lexer (tokenizace)
    ↓ Parser → AST (Abstract Syntax Tree)
    ↓ Semantic analysis (typová kontrola)
    ↓ IR generování (Intermediate Representation)
    ↓ Optimalizační průchody (LLVM passes)
    ↓ Code generation (backend, ISA specifický)
    ↓ Strojový kód / objektový soubor
```

## Abstract Syntax Tree (AST)

AST je stromová reprezentace struktury programu — nezávislá na syntaxi, ale zachycující sémantiku. Umožňuje překladači snadno provádět analýzu a transformace.

**Příklad:** výraz `a + b * c` se zobrazí jako strom `+(a, *(b, c))` — priorita operátorů je zachycena strukturou stromu, ne závorkami.

## Intermediate Representation (IR)

LLVM IR je **jazyk na úrovni mezi C a assemblerem**. Klíčové vlastnosti:
- **SSA forma** (Static Single Assignment) — každá proměnná přiřazena právě jednou → zjednodušuje analýzu závislostí.
- **Typovaná** — operace mají explicitní typy (i32, f64, ptr).
- **Platform-agnostic** — optimalizace na IR úrovni platí pro všechny cílové architektury.

**Fyzikální analogie:** IR je jako technický výkres stavby — nezávislý na tom, zda ji postavíš z cihel nebo betonu. Optimalizace na výkrese (přesunutí nosné zdi) je lepší než optimalizace při stavbě.

# LLVM a optimalizační průchody

## Přehled LLVM architektury

LLVM (Low Level Virtual Machine) je modulární překladačový framework. Clang je C/C++ frontend pro LLVM. Rust, Julia a mnoho dalších jazyků používá LLVM backend.

**Historie a realita:** LLVM (původně _Low Level Virtual Machine_, ale dnes už ta zkratka neznamená nic) vytvořil kolem roku 2000 inženýr Chris Lattner (později v Applu vytvořil jazyk Swift). Není to žádná virtuální mašina jako Java JVM. **Fyzicky je to obrovská knihovna napsaná v C++.** Žije ve formě zkompilovaných `.dll` nebo `.so` souborů, které si stahují tvůrci kompilátorů, aby nemuseli psát optimalizace od nuly.

**Proč to vzniklo:** Před LLVM musel každý tvůrce jazyka (C, C++, Pascal) napsat celý překladač od začátku až do konce. Musel napsat analyzátor syntaxe a pak musel vymyslet, jak to optimálně přeložit pro procesory Intel, AMD, ARM atd. To je brutální duplikace práce a gigantický technický dluh.

LLVM zavedlo **Modulární architekturu překladače**. Rozseklo proces na tři izolované části (Frontend, Optimizer, Backend), které spolu mluví přes společný jazyk zvaný **LLVM IR (Intermediate Representation)**.

**Fyzikální analogie: Továrna s univerzálním jazykem** Představ si, že máš inženýry z Česka (C++), Francie (Rust) a Japonska (Julia). Místo aby se každý učil, jak obsluhovat 10 různých fréz (procesory x86, ARM, M1), všichni přeloží své plány do **Esperanta (LLVM IR)**. Tento plán v Esperantu se pošle do optimalizační továrny (LLVM Optimizer). Továrna vezme plán v Esperantu, proškrtá hlouposti, zefektivní ho a pak ho pošle do specifických oddělení, která z Esperanta vygenerují přesné instrukce pro konkrétní frézu.

Optimalizace probíhají na **dvou úrovních:**
### **Úroveň 1: High-level LLVM passes (Operace nad LLVM IR)**

Tyto optimalizace ("průchody" - passes) se dějí právě nad tím "Esperantem" (LLVM IR). Jsou nezávislé na tom, jestli to poběží na Intelu nebo mobilním ARMu. Dělají to, co bys mohl teoreticky napsat ty sám do kódu, ale LLVM to udělá za tebe, aby tvůj kód zůstal čistý.

#### **1. Constant folding (Skládání konstant)**

- **Co to je:** Spočítání matematiky během kompilace.
    
- **Ukázka (Tvůj kód):** `int secondsInDay = 60 * 60 * 24;`
    
- **Co udělá LLVM:** Do binárky zapíše rovnou `int secondsInDay = 86400;`.
    
- **Lidský překlad / Pod kapotou:** Procesor nikdy neuvidí to násobení. Nestojí tě to ani jeden hodinový takt na produkci. Proč to nepíšeme rovnou my? Protože `60 * 60 * 24` je pro člověka mnohem čitelnější a udržitelnější než magické číslo `86400`. LLVM nám dovoluje psát čistý kód bez obětování výkonu.
    

#### **2. Dead code elimination (DCE - Odstranění mrtvého kódu)**

- **Co to je:** Odstřihnutí větví, které se nikdy neprovedou.
    
- **Ukázka:** `const bool debug = false; if (debug) { Log("Error"); }`
    
- **Co udělá LLVM:** Celý blok `if` včetně volání funkce `Log` z výsledného kódu úplně vymaže.
    
- **Pod kapotou:** Zmenšuje to velikost binárky (méně paměti, rychlejší načítání do CPU Cache). Často se to řetězí s dalšími passes – jiná optimalizace něco zjednoduší na `false` a DCE to pak přijde "zamést".
    

#### **3. Inlining (Vkládání funkcí)**

- **Co to je:** Vezme tělo malé funkce a "vleze" s ním přímo na místo, kde se funkce volá.
    
- **Ukázka (Před):** `int Add(int a, int b) { return a + b; } ... int x = Add(5, 10);`
    
- **Ukázka (Po):** `... int x = 5 + 10;` (A to pak Constant folding změní na `15`).
    
- **Kde se to rozbije (Edge cases):** Skok do funkce a návrat z ní (tzv. _call overhead_) stojí procesor čas (uložení kontextu na zásobník). Inlining to odstraní. Ale pozor: pokud obrovskou funkci inlinuješ na 1000 místech, brutálně nafoukneš velikost binárky a zničíš si Instruction Cache v procesoru (tzv. _Code Bloat_). LLVM má složitou heuristiku, která přesně váží, kdy se to vyplatí a kdy už ne.
    

#### **4. Loop unrolling & Loop vectorization**

- **Co to je:** Minimalizace režie `for` cyklů a využití paralelních instrukcí procesoru (SIMD).
    
- **Fyzikální analogie (Unrolling):** Místo abys 10x řekl "Vezmi cihlu, polož cihlu, zkontroluj, jestli jich je 10", řekneš rovnou 1x "Polož 10 cihel". Zmizí režie neustálého kontrolování podmínky `i < 10`.
    
- **Pod kapotou (Vectorization):** LLVM zjistí, že děláš stejnou operaci nad polem čísel. Místo zpracování po jednom nařídí procesoru přes speciální instrukce (např. AVX2), ať naloží 8 čísel naráz, sečte je jedním tikem procesoru a rovnou je vrátí. (Úzce to souvisí s Data-Oriented Designem v herních enginech, jako je Unity a jeho Burst Compiler, který na LLVM přímo stojí).
    

#### **5. Alias analysis & Escape analysis (Kritická paměťová detektivka)**

- **Alias analysis:** Detektivka, která zjišťuje: _"Ukazují tyto dva pointery do stejného místa v paměti?"_
    
    - **Proč je to nutné:** Pokud čteš přes pointer `A` a zapisuješ přes pointer `B`, kompilátor nesmí použít vektorizaci (SIMD), protože bys mohl přepsat data, která teprve budeš číst. Pokud LLVM dokáže matematicky zaručit, že se data nepřekrývají (tzv. _noalias_), odemkne ty nejagresivnější výkonnostní triky.
        
- **Escape analysis:** Detektivka, která zjišťuje: _"Opustí tento objekt někdy tuto funkci?"_
    
    - **Pod kapotou:** Alokace na haldě (Heap s klíčovým slovem `new`) je drahá a vyžaduje Garbage Collector nebo ruční čištění. Pokud LLVM zjistí, že proměnnou použiješ jen lokálně a nikdy z funkce "neunikne" (nevrátíš ji ani nepředáš asynchronnímu vláknu), **tiše ji přealokuje na Zásobník (Stack)**. Zásobník je extrémně rychlý, lokální v Cache a smaže se sám s nulovým zdržením po konci funkce.
        

#### **6. Common subexpression elimination (CSE)**

- **Co to je:** Hledání duplicit ve výpočtech.
    
- **Ukázka:** `x = (A * B) + C; y = (A * B) + D;`
    
- **Co udělá LLVM:** LLVM si řekne: "Počkat, `A * B` už mám spočítané." Změní to na: `temp = A * B; x = temp + C; y = temp + D;`. Šetří instrukce procesoru na úkor jednoho dočasného registru navíc.
    

### **Úroveň 2: Low-level passes (Příprava pro křemík)**

Zde už opouštíme bezpečné Esperanto (IR) a generujeme reálný assembly kód (jedničky a nuly) pro konkrétní čip v počítači.

#### **1. Instruction scheduling (Plánování výroby v potrubí)**

- **Co to je:** Přehazování pořadí řádků kódu (instrukcí), které na sobě nezávisí, aby procesor neměl prostoje.
    
- **Fyzikální analogie:** Procesor je jako montážní linka (Pipeline). Než se z RAM načte pomalá hodnota do registru, trvá to několik taktů. Místo aby celá linka stála a čekala na načtení hodnoty A (tzv. _Pipeline Stall_), LLVM prohodí instrukce tak, aby procesor mezitím spočítal nesouvisející proměnnou B. Navenek tvůj kód dělá to samé, ale uvnitř se věci dějí na přeskáčku, aby CPU jelo na 100 %.
    

#### **2. Register allocation (Kuchařovo prkénko)**

- **Co to je:** Rozdělení tvých stovek proměnných do velmi omezeného počtu hardwarových registrů procesoru (např. pouhých 16 registrů na starších architekturách).
    
- **Fyzikální analogie:** Registr je kuchařovo prkénko. Cache/RAM je lednice. Chceš krájet cibuli (počítat), musí být na prkénku. Problém je, že máš jen 16 prkének, ale 100 ingrediencí.
    
- **Co dělá LLVM:** Řeší matematický problém zvaný _Barvení grafu_. Snaží se zjistit životní cyklus proměnných. Když proměnná `i` umře, její registr se okamžitě recykluje pro proměnnou `j`.
    
- **Edge case (Spilling):** Pokud máš zaráz aktivních více než 16 proměnných, prkénka dojdou. LLVM musí vygenerovat kód, který nějakou hodnotu dočasně "odlije" (Spilling) zpět do RAM, uvolní registr, provede výpočet a pak ji načte zpět. To je brutální zabiják výkonu. Zkušení systémoví vývojáři proto omezují počet lokálních proměnných ve vnitřních smyčkách.
    

#### **3. Peephole optimization (Optimalizace klíčovou dírkou)**

- **Co to je:** Úplně na konci vezme LLVM "lupu", dívá se vždy jen na blok 3-4 po sobě jdoucích strojových instrukcí (klíčová dírka) a zkouší je nahradit nějakým obskurním hw "hackem".
    
- **Ukázka:** V kódu máš `x = x * 2`. Násobení je na úrovni hardwaru docela drahá operace. Peephole pass to uvidí a nahradí to za tzv. Bitový posun doleva (`x = x << 1`). Výsledek je stejný, ale bitový posun zabere na procesoru 1 jediný takt.
    

### **Big Picture (Celkový obrázek v architektuře softwaru)**

Pokud navrhuješ robustní systém, je důležité vědět, že ty **nejsi ten, kdo má za úkol provádět tyto mikro-optimalizace ve svém C# kódu.** Pokud se o to budeš snažit (např. ručně rozvíjet `for` cykly místo kompilátoru), zaneseš si do codebase šílený technický dluh a s největší pravděpodobností ten kompilátor (zde RyuJIT v .NETu nebo LLVM v C++) jenom spleteš a výsledek bude _pomalejší_.

Tvým inženýrským úkolem je psát kód tak čistě, predikovatelně a strukturovaně (krátké funkce, lokální proměnné, minimalizace globálních stavů a pointerových aliasingů), abys LLVM umožnil dělat jeho práci. Ty navrhuješ dálnici, LLVM staví závodní auto.

## Praktické tipy — co překladači pomáhá

- **`-O2` / `-O3`**: aktivuje většinu optimalizačních průchodů.
- **`-march=native`**: překladač může použít instrukce specifické pro aktuální CPU (AVX2, BMI2 atd.).
- **`__builtin_expect(x, 1)`**: hint překladači o pravděpodobnosti větve → lepší branch prediction layout.
- **`restrict`**: říká překladači, že pointer neukazuje na stejná data jako jiný → umožňuje vektorizaci.
- **`const` správně**: pomáhá analýze aliasů.

# Co zůstává na programátorovi

## Limity překladače

Překladač optimalizuje to, co **může bezpečně dokázat**. Nemůže optimalizovat věci, kde by optimalizace mohla změnit chování:

### Problémy s aliasem (pointer aliasing)

```c
void add(int *a, int *b, int *c, int n) {
    for (int i = 0; i < n; i++) c[i] = a[i] + b[i];
}
```
Překladač neví, zda `c` neukazuje na stejná data jako `a` nebo `b` → nemůže vektorizovat. Řešení: přidej `restrict`.

### Nepředvídatelné vstupy za runtime

Překladač nezná hodnoty za runtime → nemůže spekulovat o větvích. Příklad: podmínka závisí na vstupu uživatele.

### Volatilní paměť a side effects

Memory-mapped I/O, atomic operace — překladač musí zachovat pořadí přístupů.

### Cross-function optimization bez LTO

Bez LTO překladač nevidí definici funkce z jiné kompilační jednotky → nemůže inlinovat.

## Profile-Guided Optimization (PGO) — data říkají překladači, co optimalizovat

**PGO** je třífázový proces:
1. **Instrumentovaný překlad** — přelož program s `-fprofile-generate`.
2. **Profilování** — spusť program na reprezentativních vstupech. Binárka sbírá profil (počty skoků, branch profily).
3. **Optimalizovaný překlad** — přelož znovu s `-fprofile-use=<profil>`.

**Co PGO zlepší:**
- Branch prediction layout — horké větve jsou "blíže" v kódu → instrukční cache.
- Inlining rozhodnutí — inlinuje horké funkce, neinlinuje studené.
- Loop unrolling intenzita — agresivnější pro horké smyčky.

> [!note] Reálné využití PGO
> Chromium, V8, LLVM sám, Firefox — všechny používají PGO a dosahují ~5–15 % zlepšení výkonu bez změny zdrojového kódu.

# Profiling v C/C++

## Hardware Performance Counters

Moderní CPUs mají **Performance Monitoring Unit (PMU)** — hardware čítače, které počítají nízkoúrovňové události:
- `cache-misses` — počet L1/L2/L3 cache miss
- `branch-misses` — počet mispredicted branch
- `instructions` / `cycles` — IPC (Instructions Per Cycle)
- `page-faults` — TLB miss → OS stránkování

**Nástroj:** `perf stat ./myprog` (Linux) — zobrazí souhrnné statistiky.
**Nástroj:** `perf record` + `perf report` — sampling profiler na úrovni instrukcí.

## Sampling vs. instrumentace

| | Sampling | Instrumentace |
|---|---|---|
| Princip | Periodické přerušení → zaznamená PC | Přidá kód do každé funkce |
| Overhead | Nízký (~1–5 %) | Vysoký (10–100 %) |
| Přesnost | Statistická (šum) | Přesná |
| Vhodné pro | Rychlý přehled horkých míst | Přesné měření konkrétní funkce |

**Nástroj (sampling):** `perf`, `gprof`, `Valgrind Callgrind`
**Nástroj (HW counters):** `perf stat`, `Intel VTune`

## Self-evicting code

Při profilování si dávej pozor na **self-evicting code** — profilovací instrukce samy zahlcují instrukční cache, čímž zkreslují měření. Sampling profiler to minimalizuje.

> Ke státnicím: Překladač = frontend (AST) + IR optimalizace + backend. LLVM passes = constant folding, DCE, inlining, vectorization, alias analysis. Překladač selže u aliasů, runtime dat, cross-module bez LTO. PGO = tři fáze (instrument→profiluj→recompile) → +5–15 %. Hardware counters → perf → IPC, cache miss, branch miss.
