# Historie a motivace virtualizace

## Proč virtualizace vznikla

**Historická motivace (1960s, IBM):**
- Mainframy byly extrémně drahé → sdílení jednoho stroje více uživateli.
- CP-40 / CP-67 → první "virtual machine monitor".

**Moderní motivace:**
- **Konsolidace serverů:** Místo 20 fyzických serverů (každý vytížen 10 %) → 1 server s 20 VM (celkové vytížení 80 %). Úspora hardware, energie, prostoru.
- **Izolace:** Chyba v jedné VM neovlivní ostatní. Bezpečnostní sandbox.
- **Snapshot & Live migration:** Zastavit VM, uložit stav, přenést na jiný server, spustit znovu.
- **Cloud computing:** AWS, Azure, GCP — vše je virtualizace. Zákazník dostane VM, ne fyzický server.
- **Testování:** Vývojáři testují na VM s přesnou kopií produkce.

## Typy virtualizace

| Typ | Popis | Příklady |
|---|---|---|
| **Full virtualization** | VM neví, že je VM. Spuštění unmodified OS | VMware, VirtualBox, KVM |
| **Paravirtualizace** | VM ví, že je VM — speciální API pro hypervisor | Xen (PV mode), Virtio |
| **OS-level (kontejnerizace)** | Sdílené kernel, izolace přes namespaces/cgroups | Docker, LXC |
| **Hardware-assisted** | CPU instrukce pro efektivní full virtualizaci | Intel VT-x, AMD-V |

# Základy virtualizace: VMM a hypervisor

## Virtual Machine Monitor (VMM / Hypervisor)

**Hypervisor** = software, který vytváří a spravuje virtuální stroje. VM je izolovaný "kontejner" s virtuálním CPU, RAM, disky a sítí.

**Požadavky na VMM (Popek & Goldberg, 1974):**
1. **Fidelity:** VM se chová identicky jako fyzický stroj (kromě výkonu).
2. **Safety:** VMM má plnou kontrolu nad hardware — VM nemůže obejít VMM.
3. **Efficiency:** Velká část instrukcí VM běží přímo na hardware bez VMM zprostředkování.

## Typ 1 vs. Typ 2 hypervisor

**Typ 1 (bare-metal):** Hypervisor běží přímo na hardware (bez host OS):
```
Hardware
    ↓
Hypervisor (Type 1)   ← přímý přístup k hardware
  ↓         ↓
 VM1       VM2
(Guest OS) (Guest OS)
```
**Příklady:** VMware ESXi, Microsoft Hyper-V, Xen, KVM (Linux kernel je hypervisor).
**Výhody:** Nižší overhead, přímá kontrola hardware, lepší výkon.

**Typ 2 (hosted):** Hypervisor běží jako aplikace nad host OS:
```
Hardware
    ↓
Host OS (Windows/Linux)
    ↓
Hypervisor (Type 2)   ← musí přes host OS
  ↓         ↓
 VM1       VM2
```
**Příklady:** VirtualBox, VMware Workstation, Parallels.
**Výhody:** Jednodušší instalace, vývoj a testování.
**Nevýhody:** Vyšší overhead (host OS vrstva navíc).

## Privilege rings a VM izolace

**x86 privilege rings:**
```
Ring 0 (nejprivilegovanější) — Kernel mode (OS kernel, device drivers)
Ring 1 ─ nepoužíváno v moderních OS
Ring 2 ─ nepoužíváno v moderních OS
Ring 3 (nejméně privilegovaný) — User mode (aplikace)
```

**Problém virtualizace:** Guest OS chce běžet v Ring 0, ale to je Ring 0 hypervisoru. Nemůžeme mít dva "šéfy".

**Klasické řešení (x86 bez VT-x):**
- Guest OS je přesunut do Ring 1 (nebo Ring 3).
- Privilegované instrukce Guest OS způsobí **trap** → zachyceny hypervisorem → emulace.
- Overhead: každá privilegovaná instrukce = trap + emulace (drahé!).

**Problém x86 — nevirtualizovatelné instrukce:** Některé x86 instrukce chovají se jinak v Ring 0 vs. Ring 1 bez generování trapu → nelze zachytit. (Např. `POPF` mění IF flag v Ring 0, ale tiše ignoruje v Ring 1.) Řešení: binary translation (VMware) nebo VT-x.

# Hardware-asistovaná virtualizace

## Intel VT-x a AMD-V

**VT-x/AMD-V** přidávají dva nové módy CPU:
- **VMX root mode** (Ring -1) — pro hypervisor. Nad Ring 0.
- **VMX non-root mode** — pro VM. Vlastní Ring 0–3.

```
VMX root mode (Ring -1): Hypervisor
  ↓ VM entry (VMLAUNCH/VMRESUME)
VMX non-root mode: Guest OS (Ring 0) + Guest Apps (Ring 3)
  ↓ VM exit (privilegovaná instrukce, interrupt)
VMX root mode: Hypervisor zachytí, zpracuje, vrátí
```

**Výhody VT-x:**
- Guest OS běží v Ring 0 (non-root) → žádná změna Guest OS.
- Privilegované instrukce automaticky způsobují VM exit → hypervisor je zachytí.
- Žádný binary translation overhead.

**VMCS (Virtual Machine Control Structure):** Datová struktura v paměti, která ukládá stav VM při VM exit a obnovuje ho při VM entry. Hypervisor ji používá pro konfiguraci VM behavior.

## Nested Page Tables — EPT (Intel) / NPT (AMD)

**Problém paměťové virtualizace:**
- Guest OS spravuje vlastní stránkovací tabulky: Guest Virtual Address → Guest Physical Address.
- Hypervisor musí překládat: Guest Physical Address → Host Physical Address.
- Bez hardware podpory: Shadow page tables v hypervisoru → komplexní, pomalé (sync při každé změně Guest PT).

**Řešení — Extended Page Tables (EPT):**
- Dva-úrovňová stránkovací tabulka: Guest PT (GVA→GPA) + EPT (GPA→HPA).
- CPU hardware automaticky provede dvoustupňový překlad.
- Guest OS mění svou PT svobodně — hypervisor jen udržuje EPT.

```
GVA → [Guest PT] → GPA → [EPT (hardware)] → HPA
               ↑ guest OS spravuje    ↑ hypervisor spravuje
```

**TLB a VPID:** TLB cache překladů je sdílená → při VM exit/entry musí být flushována (overhead). VPID (Virtual Processor ID) tag TLB záznamy per-VM → eliminuje TLB flush.

# I/O virtualizace

## NIC emulace (Full emulation)

**Nejjednodušší přístup:** Hypervisor emuluje standardní síťovou kartu (např. Intel e1000) v software.
- Guest OS použije standardní driver → žádná modifikace.
- Každý I/O požadavek → VM exit → hypervisor → SW emulace → host NIC.
- **Overhead:** Velký! Každý paket = minimálně 2 VM exit/entry + SW emulace.

## Paravirtualizace — Virtio

**Virtio** = standardní paravirtualizovaný I/O framework. Guest OS ví, že je virtualizovaný a komunikuje přímo s hypervisorem přes efektivní protokol.

**Architektura Virtio:**
```
Guest OS (Virtio driver) ← sdílená paměť (Virtqueue) → Hypervisor (Virtio backend)
```

- **Virtqueue** = kruhový buffer v sdílené paměti → žádné kopírování dat.
- Méně VM exitů než při plné emulaci.
- Vyžaduje Virtio driver v Guest OS (Linux kernel má Virtio nativně).

**Výkon Virtio vs. emulace:** Virtio je typicky 2–5× rychlejší než SW emulace.

## PCI Pass-through (VFIO)

**Přidělení fyzické NIC přímo VM:**
- Fyzická PCI karta je "odpojena" od host OS a přidělena jedné VM.
- Guest OS má přímý přístup k hardware → žádný hypervisor overhead pro I/O.
- IOMMU (Intel VT-d, AMD-Vi) → DMA izolace (NIC nemůže přistupovat k paměti jiných VM).

**Výhody:** Near-native výkon sítě. Vhodné pro latency-sensitive workloady.
**Nevýhody:** NIC nelze sdílet mezi více VM. Méně flexibilní.

## SR-IOV (Single Root I/O Virtualization)

**SR-IOV** = hardware standard, kdy jedna fyzická NIC (Physical Function — PF) může vytvořit desítky **Virtual Functions (VF)** — virtuálních NIC:

```
Fyzická NIC (Physical Function — PF)
  ├── Virtual Function 1  → VM1 (přímý přístup)
  ├── Virtual Function 2  → VM2 (přímý přístup)
  ├── Virtual Function 3  → VM3 (přímý přístup)
  └── ...
```

- Každá VF je oddělená PCI funkce s vlastní přerušovací linkou a DMA engine.
- VM dostane VF jako "vlastní NIC" → přímý přístup bez hypervisoru.
- Datová cesta: Near-native výkon (hardware switching mezi VF bez software).

**Výhody SR-IOV:** Sdílení fyzické NIC mezi VM při zachování výkonu blízkého pass-through. Desítky VM mohou sdílet jednu fyzickou 100 GbE NIC.

# Efektivní inter-VM networking

## Hardware offloading

Moderní síťové karty (SmartNIC) přebírají část práce od CPU:
- **TSO (TCP Segmentation Offload):** NIC sama segmentuje velké TCP segmenty → CPU posílá velké bloky, NIC segmentuje.
- **LRO (Large Receive Offload):** NIC slučuje příchozí segmenty → méně přerušení CPU.
- **RSS (Receive Side Scaling):** NIC distribuuje příchozí pakety na více CPU front (multi-queue NIC).
- **Checksum offload:** NIC počítá IP/TCP checksumy → CPU ušetřen.

## DPDK (Data Plane Development Kit)

**Problém Linux network stacku:** Každý příchozí paket = přerušení CPU → kernel zpracování → context switch → kopírování do userspace → cca 1000+ CPU cyklů na paket.

**DPDK** = framework pro přímý přístup k NIC z userspace, bypass Linux kernel:
- **PMD (Poll Mode Driver):** Vlákno aktivně polluje NIC (žádná přerušení) → nulový interrupt overhead.
- **Hugepages:** Velké stránky (2 MB/1 GB) → méně TLB miss při DMA.
- **Zero-copy:** Data přenesena přímo do userspace paměti (NIC → userspace DMA).

**Výkon:** Kernel network stack → ~10 Mpps (megapaketů/s). DPDK → 80–100 Mpps na jednom jádru.

**Použití:** Telekomunikace (5G), high-frequency trading, DPI (deep packet inspection), virtuální routery (OVS-DPDK).

> Ke státnicím: Typ 1 hypervisor = bare-metal (KVM, Xen, ESXi). Typ 2 = hosted (VirtualBox). Privilege rings — Guest OS chtěl Ring 0 → problém → VT-x zavádí VMX root/non-root mode (Ring -1). EPT = hardware 2-úrovňový překlad GVA→GPA→HPA (žádné shadow PT). Virtio = paravirtualizovaný I/O přes virtqueue (sdílená paměť, méně VM exit). SR-IOV = fyzická NIC vytvoří N Virtual Functions → přímý přístup VM bez hypervisoru. DPDK = polling userspace driver, bypass kernel → 80+ Mpps.
