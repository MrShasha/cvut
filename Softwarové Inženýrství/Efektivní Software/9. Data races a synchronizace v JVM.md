# Data races a superscalar execution

## Proč CPU přehazuje instrukce

Moderní CPU je **out-of-order execution (OOO)** procesor — přehazuje pořadí instrukcí pro maximální využití výpočetních jednotek. Výsledky jsou doručeny ve správném pořadí (reorder buffer), ale **paměťové přístupy** mohou být pozorovány jiným vláknem v jiném pořadí.

**Příklad:**
```java
// Thread 1:
data = 42;
ready = true;

// Thread 2:
if (ready) {
    use(data);  // Může vidět ready=true, ale data=0!
}
```

CPU může přehodit zápisy v Thread 1: `ready = true` se propíše do cache před `data = 42`. Thread 2 pak vidí nekonzistentní stav.

**Fyzikální analogie:** Dva poštovní bratři posílají balíky. Starší bratr pošle balík (data) a pak potvrzení (ready). Poštovní systém může doručit potvrzení dřív než balík — příjemce si myslí, že balík přišel, ale ještě nepřišel.

## Java Memory Model (JMM) a data race

**Data race** = dva přístupy ke stejné proměnné z různých vláken bez synchronizace, přičemž alespoň jeden je zápis.

**Java Memory Model** definuje, kdy jeden thread "vidí" změny druhého threadu přes **happens-before** vztah:
- Zápis do `volatile` proměnné happens-before čtení z ní.
- `lock()` happens-before `unlock()`.
- Vše v thread happens-before cokoli po `thread.join()`.

Bez happens-before = data race = undefined behavior v JVM kontextu (ne tak explicitně jako v C++, ale výsledky jsou nepředvídatelné).

# Memory barriers a volatile proměnné

## Co jsou memory barriers

**Memory barrier (memory fence)** = instrukce procesoru, která zabrání přeuspořádání paměťových přístupů přes tuto bariéru.

Typy bariér:
- **StoreStore barrier** — zabrání přeuspořádání zápisů (garantuje pořadí zápisů).
- **LoadLoad barrier** — zabrání přeuspořádání čtení.
- **StoreLoad barrier** — nejsilnější, zabrání přeuspořádání zápisů a čtení.

## `volatile` v Javě — kdy stačí

`volatile` zajistí:
1. **Viditelnost** — zápis do volatile proměnné je okamžitě viditelný všem vláknům (flush z cache do main memory).
2. **Pořadí** — zápisy/čtení volatile proměnné nejsou přeuspořádány přes memory barrier.

**`volatile` NESTAČÍ pro:**
```java
volatile int counter = 0;

// Thread 1 & Thread 2:
counter++;  // ← RACE CONDITION! counter++ = read + increment + write (ne atomické)
```

`volatile` garantuje viditelnost, ale ne atomičnost složených operací (read-modify-write).

**`volatile` STAČÍ pro:**
- Jeden writer, více readerů (single-writer pattern).
- Flagy (stop flag pro vlákno, inicializační signál).
- Publikace immutable objektů (after construction, before publish).

## Příklad: Double-checked locking — klasická past

```java
// ŠPATNĚ (pre-Java 5, bez volatile):
if (instance == null) {
    synchronized (this) {
        if (instance == null) {
            instance = new Singleton(); // ← přiřazení nemusí být vidět správně!
        }
    }
}

// SPRÁVNĚ:
private volatile static Singleton instance;  // volatile!
```

# Synchronizace: thin, fat a biased locking

## JVM adaptivní locking — tři úrovně

JVM nepoužívá vždy těžký OS mutex — optimalizuje na základě pozorování:

### 1. Biased locking (předpojatý zámek)

**Předpoklad:** Pokud objekt odemkla vždy jediné vlákno, JVM "předzamkne" objekt tímto vláknem.

**Jak funguje:** Mark word objektu obsahuje ID vlákna. Lock/unlock = žádná atomická operace — jen kontrola, zda jsem to já.

**Overhead:** Prakticky nulový pro nekonkurentní přístup.

**Revocation:** Pokud jiné vlákno chce zamknout → nutná safepoint → revocation (drahá!). Proto biased locking byl v Java 15 deprecated a v Java 21 odstraněn (většina aplikací má dost konkurence, aby revocation převažoval benefity).

### 2. Thin locking (tenký zámek)

**Předpoklad:** Lock je zamknutý, ale bez čekání (rychlé locky).

**Jak funguje:** CAS (Compare-And-Set) instrukce atomicky nastaví ID vlákna v mark wordu. Pokud CAS uspěje → zamčeno. Pokud ne → thread spin.

**Overhead:** 1 CAS instrukce (~20–30 ns).

### 3. Fat locking (tlustý zámek — heavy-weight monitor)

**Předpoklad:** Vlákna soupeří o zámek (contention).

**Jak funguje:** Vytvoří se OS mutex (`pthread_mutex_t`). Vlákno jde do sleep → OS wakeup. Monitor obsahuje wait/notify queue.

**Overhead:** Syscall → ~1000–5000 ns. Ale vlákno nespotřebovává CPU ve čekání.

## Reentrant locks

**`ReentrantLock`** v Javě = explicitní lock s rozšířenými vlastnostmi:
- Trylock (neblokující pokus o zamknutí).
- Timeout (`lock(timeout, unit)`).
- Interruptible locking.
- `Condition` (replacement pro wait/notify).

```java
ReentrantLock lock = new ReentrantLock();
lock.lock();
try {
    // kritická sekce
} finally {
    lock.unlock(); // vždy v finally!
}
```

# Atomické operace a compare-and-set

## CAS (Compare-And-Set) instrukce

**CAS** je hardware instrukce (`CMPXCHG` na x86):
```
CAS(adresa, expected, new_value):
  if (*adresa == expected:
    *adresa = new_value
    return true
  else:
    return false
```
Tato operace je **atomická** — garantovaná hardware. Na ní stojí celé lock-free programování.

## Java atomic třídy

```java
AtomicInteger counter = new AtomicInteger(0);
counter.incrementAndGet();  // atomické — bezpečné z více vláken

// Implementace pod kapotou (zjednodušeně):
public int incrementAndGet() {
    int current, next;
    do {
        current = get();           // volatile čtení
        next = current + 1;
        // Pokud se current změnil, CAS selže a opakujeme (spin)
    } while (!compareAndSet(current, next));
    return next;
}
```

## Atomic field updaters

Pokud potřebuješ atomicky operovat s polem existující třídy (bez přepsání):

```java
AtomicIntegerFieldUpdater<Node> countUpdater =
    AtomicIntegerFieldUpdater.newUpdater(Node.class, "count");

countUpdater.incrementAndGet(nodeInstance);  // atomicky inkrementuje node.count
```

**Výhoda:** Méně paměťového overhead než `AtomicInteger` jako pole (pole je `volatile int`, ne objekt).

# Non-blocking algoritmy

## Wait-free vs. lock-free

| | Wait-free | Lock-free |
|---|---|---|
| Definice | Každé vlákno dokončí operaci v konečném počtu kroků | Alespoň jedno vlákno vždy pokročí |
| Záruka | Žádné vlákno nemůže stárnout | Starvation je teoreticky možný |
| Složitost | Velmi obtížné implementovat | Obtížné implementovat |
| Příklady | Read-side v RCU | Michael-Scott queue, TreibER stack |

## Non-blocking stack (LIFO) — TreibER stack

```java
public class NonBlockingStack<T> {
    private final AtomicReference<Node<T>> top = new AtomicReference<>(null);

    public void push(T value) {
        Node<T> newNode = new Node<>(value);
        Node<T> currentTop;
        do {
            currentTop = top.get();
            newNode.next = currentTop;
        } while (!top.compareAndSet(currentTop, newNode)); // CAS smyčka
    }

    public T pop() {
        Node<T> currentTop;
        do {
            currentTop = top.get();
            if (currentTop == null) return null;
        } while (!top.compareAndSet(currentTop, currentTop.next)); // CAS smyčka
        return currentTop.value;
    }
}
```

**ABA problém v CAS smyčce:** `top` bylo A, pak B, pak zpět A → CAS uspěje, ale stav se změnil. Řešení: `AtomicStampedReference` (přidá verzi/stamp).

## ConcurrentHashMap — non-blocking implementace

Java `ConcurrentHashMap` je jednou z nejpropracovanějších non-blocking datových struktur:
- Čtení je **lock-free** (volatile přístup k entry).
- Zápis používá **CAS** pro přidání prvního prvku do bucketu.
- Pro kolize používá `synchronized` na hlavičce bucketu (fine-grained locking).
- Výsledek: paralelní čtení i zápis bez globálního locku → škáluje s počtem jader.

> Ke státnicím: Data race = přístup ke sdílené proměnné bez synchronizace → nedefinovaný výsledek. JMM = happens-before vztahy (volatile write HB volatile read, unlock HB lock). `volatile` = viditelnost + pořadí, ale ne atomičnost. Locking: biased (žádný overhead, revocation drahá), thin (CAS, ~30 ns), fat (OS mutex, ~1000+ ns). CAS = hardware atomická operace (základ lock-free). Non-blocking LIFO = CAS smyčka + ABA problém.
