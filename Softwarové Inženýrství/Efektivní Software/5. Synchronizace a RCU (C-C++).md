Synchronizace je nejzákeřnější zdroj výkonnostních problémů v multi-threaded systémech. Špatná synchronizace nezpůsobí jen pomalý kód — způsobí nesprávný kód. Tato kapitola ti ukáže, proč naivní přístupy selhávají, jaké mechanismy existují (a co se za nimi skrývá "pod kapotou"), a jak zvolit ten správný pro daný workload.

# Problémy naivní synchronizace

## Race condition — proč sdílená data bez ochrany nefungují

**Race condition** nastane, pokud výsledek výpočtu závisí na pořadí, ve kterém vlákna přistupují ke sdíleným datům. Pořadí je nedeterministické → výsledek je nepředvídatelný.

**Příklad:**
```c
bool locked = false;
void func() {
    while (locked == true) { /* busy wait */ }
    locked = true;   // ← PROBLÉM: dvě vlákna mohou projít najednou!
    data++;
    locked = false;
}
```

**Proč to nefunguje:** Mezi testem `while (locked == true)` a nastavením `locked = true` může jiné vlákno projít testem a také nastavit `locked = true`. Oba pak vstoupí do kritické sekce.

**Fyzikální analogie:** Semafor na jednoproudé silnici, kde řidič vidí "zelená" a rozjede se — ale než dorazí na semafor, jiný řidič z druhé strany udělá totéž. Kolize.

## Busy waiting — výkonnostní katastrofa

```c
while (locked == true) { /* busy wait */ }
```

CPU stráví 100 % cyklů testováním proměnné. To je extrémní plýtvání — jiná vlákna nemohou být naplánována na toto jádro.

**Výjimka:** Velmi krátké kritické sekce na vícejádrovém CPU, kde je čekání kratší než overhead kontextového přepínání → spinlock (viz níže).

## ABA problém

Vlákno A přečte hodnotu X, vlákno B změní X→Y→X. Vlákno A vidí X a myslí, že se nic nezměnilo — ale stav systému se změnil. Relevantní pro lock-free algoritmy s CAS.

# Real-world synchronizace: mutex, semaphore, rw-lock, spinlock

## Mutex — základní vzájemné vyloučení

**Mutex** (Mutual Exclusion) zaručuje, že kritickou sekci vykonává vždy právě jedno vlákno.

**Jak funguje pod kapotou:**
1. Vlákno zavolá `lock()` → atomicky zkusí nastavit flag.
2. Pokud je mutex volný → získá ho, vstoupí do CS.
3. Pokud je mutex obsazen → vlákno jde do **sleep** (dá se na wait queue v OS).
4. Po `unlock()` OS probudí čekající vlákno.

**Overhead:**
- Lock/unlock = syscall (kontextový přepínání user↔kernel) → ~1000–5000 ns.
- Moderní "futex" (Linux) — lock je rychlý, pokud není spor; syscall jen při sporu.

**Kdy použít:** Kritické sekce se střední délkou (µs–ms range), kde je spor (contention) nízký.

## Semaphore — počítadlo přístupu

**Semaphore** generalizuje mutex — umožňuje N vláknům vstoupit do CS najednou.
- `sem_wait()` → dekrementuje čítač; pokud 0 → sleep.
- `sem_post()` → inkrementuje čítač; probudí čekající vlákno.

**Kdy použít:** Omezení paralelního přístupu (connection pool, rate limiting). Mutex = binární semaphore.

## Reader-Writer Lock (rw-lock) — oddělení čtecích a zápisových přístupů

**Problém:** Mutex neumožňuje souběžné čtení. Ale čtení je bezpečné paralelně — pouze zápis musí být exkluzivní.

**rw-lock:**
- Libovolný počet **readerů** může číst souběžně.
- Pouze jeden **writer** může zapisovat — a musí počkat na všechny readery.

**Výkon:** Pro read-heavy workload (80–90 % čtení) je rw-lock výrazně rychlejší než mutex.

**Problém rw-locku:** Writer starvation — pokud readery přicházejí neustále, writer se nikdy nedostane na řadu. Implementace to řeší různě (priority writeru).

## Spinlock — busy wait jako záměrná volba

**Spinlock** nedává vlákno do sleep — aktivně točí (spins) ve smyčce:
```c
while (!__atomic_test_and_set(&lock, __ATOMIC_ACQUIRE)) { }
```

**Kdy spinlock dává smysl:**
- Kritická sekce je **velmi krátká** (řádově desítky ns).
- Systém má **víc jader** než vláken (jinak spinnující vlákno blokuje ostatní).
- Cena spinování je nižší než cena kontextového přepínání (~1000+ ns).

**Kdy spinlock škodí:**
- Dlouhé kritické sekce → plýtvání CPU.
- Na jednojádrovém CPU → deadlock (spinning vlákno blokuje i vlákno, co drží lock).

**Adaptive spinlock:** Nejdřív spinuje N iterací, pak přejde do sleep. Nejlepší z obou světů.

## Srovnání mechanismů

| Mechanismus | Overhead locku | Vhodné pro |
|---|---|---|
| **Spinlock** | < 100 ns | Velmi krátké CS, multi-core |
| **Mutex (futex)** | ~200–1000 ns | Střední CS, obecný případ |
| **Semaphore** | ~200–1000 ns | N-way přístup, resource pool |
| **rw-lock** | ~500–2000 ns | Read-heavy workload |
| **RCU** | ~0 ns čtení | Read-mostly, viz níže |

# Read-mostly workload

## Problém s mutex pro read-heavy data

Pokud 95 % přístupů jsou čtení (typické pro konfigurační data, cache, routing tabulky), pak:
- Mutex zbytečně serializuje čtení — pouze jedno vlákno čte najednou.
- rw-lock pomáhá, ale writer pořád musí čekat a pak blokuje všechny readery.

**Fyzikální analogie:** Knihovna, kde každý čtenář musí čekat, až dostane klíč od místnosti — i když čte jinou knihu než ostatní. Nesmysl. Skutečné knihovny nechají vstoupit kohokoliv — uzamknou místnost, jen když přijde vrátný s novými knihami.

# Read-Copy-Update (RCU)

## Myšlenka — lock-free čtení pomocí epochy

**RCU** je synchronizační mechanismus původem z Linuxového kernelu, navržený pro **extrémně read-heavy** workload (přístup ke sdíleným datovým strukturám).

**Klíčová myšlenka:**
- **Readery** nikdy nečekají — přistupují k datům bez jakékoliv synchronizace.
- **Writery** vytvoří **kopii** dat, upraví kopii, a pak atomicky přepnou pointer na novou verzi.
- Stará verze je zachována, dokud ji všechna existující čtecí vlákna nedokončí (grace period).

## RCU v praxi — tři operace

```c
// Čtení (read-side critical section) — žádný lock:
rcu_read_lock();
struct config *cfg = rcu_dereference(global_cfg);  // atomický load
use(cfg);
rcu_read_unlock();

// Zápis (pouze jedno vlákno najednou, typicky pod mutex):
struct config *new_cfg = kmalloc(...);  // alokuj novou kopii
*new_cfg = *old_cfg;
new_cfg->value = new_value;            // uprav kopii
rcu_assign_pointer(global_cfg, new_cfg); // přepni pointer (memory barrier!)
synchronize_rcu();                      // počkej na grace period
kfree(old_cfg);                         // uvolni starý
```

**Klíčový detail:** `rcu_assign_pointer()` používá pouze **compiler a memory barriers** — **ne** atomické instrukce (jako CAS nebo atomic exchange). To je důvodem extrémní rychlosti read-path: čtení je prostý load s read-side memory barrier, ne atomická operace.

## Grace period — kdy je bezpečné uvolnit stará data

**Grace period** je interval, po jehož uplynutí je garantováno, že žádné vlákno nemá referenci na starou verzi dat.

**Jak se detekuje:** Každé vlákno má quiescent state — okamžik, kdy nemůže držet RCU referenci (kontext přepínání, idle). Po tom, co všechna vlákna projdou quiescent state → grace period skončila.

## Výhody a nevýhody RCU

**Výhody:**
- Readery jsou naprosto lock-free → nulový overhead na read path.
- Škáluje perfektně s počtem čtenářů.
- Čtení je race-condition free (atomic pointer load).

**Nevýhody:**
- Paměťový overhead — stará verze existuje až do konce grace period.
- Latence writeru závisí na délce grace period (může být ms–stovky ms).
- Složitost implementace.

**Kde se RCU používá:**
- Linux kernel (routing tabulky, task lists, modul systém).
- Userspace: `liburcu`, garbage collector v Go runtime.

## RCU a Java — proč nedává smysl

**Otázka:** Lze RCU použít v Javě?

**Odpověď: Ne, a tady proč.** Java má garbage collector — GC sám sleduje reference na objekty a automaticky uvolňuje paměť, jakmile na objekt neexistuje žádná reference. RCU řeší přesně tento problém (bezpečné uvolnění staré verze dat) jiným způsobem — explicitní grace period.

GC má ale mnohem vyšší overhead než RCU:
- **Stop-the-world pauzy** — GC občas zastaví všechna vlákna.
- **Marking overhead** — GC musí projít celý heap.

V C/C++ (bez GC) má RCU smysl, protože bez něj bys musel počítat reference (`shared_ptr`) — a to má overhead při každém přístupu. RCU eliminuje reference counting i pro read path.


> [!tip]- RW_lock vs RCU
> 
> ### **1. RW Lock (Read-Write Lock)**
> 
> V .NETu ho znáš jako `ReaderWriterLockSlim` (nebo v C++ jako `std::shared_mutex`). Je to standardní a nejčastější zbraň pro situace, kde se hodně čte a málo zapisuje.
> 
> **Jak to funguje (Abstrakce):**
> 
> - Máme dvě brány. Do "čtecí brány" může vstoupit nekonečně mnoho vláken naráz (sdílejí data).
>     
> - Do "zapisovací brány" může vstoupit jen jeden zapisovatel.
>     
> - **Klíčové pravidlo:** Zapisovatel musí počkat, až všichni čtenáři odejdou. A zatímco zapisovatel zapisuje, noví čtenáři musí stát venku a čekat.
>     
> 
> **Fyzikální analogie: Studovna v knihovně**
> 
> Představ si knihovnu s jednou vzácnou knihou.
> 
> - **RW Lock:** Každý čtenář, který vejde, si vezme u vchodu žeton z misky (počítadlo čtenářů). Když chce knihovník (zapisovatel) knihu aktualizovat, podívá se do misky. Pokud tam chybí žetony, ví, že někdo čte. Stoupne si ke dveřím, nikoho dalšího nepustí dovnitř a čeká, až poslední čtenář vrátí žeton a odejde. Pak zamkne zevnitř, přepíše knihu, odemkne a pustí dav.
>     
> 
> **Pod kapotou: Architektonický bottleneck (Cache Line Bouncing)**
> 
> Na papíře zní RW Lock skvěle. Většinu času všichni jen čtou, takže se vzájemně neblokují, ne? **Omyl.** Když vlákno v procesoru (třeba na NUMA uzlu 0) chce "jen číst", musí fyzicky vzít ten žeton. Musí provést hardwarovou operaci (tzv. `Interlocked.Increment` nebo atomický součet) na sdíleném počítadle čtenářů.
> 
> Tím, že provede zápis do počítadla, procesor musí okamžitě zneplatnit tuto Cache line ve všech ostatních 64 jádrech na serveru. Takže i když tvá vlákna "jen čtou", na úrovni hardwaru probíhá brutální ping-pong zpráv přes sběrnici (Cache Line Bouncing). Systém se zahltí vlastní synchronizací.
> 
> ### **2. RCU (Read-Copy-Update)**
> 
> RCU je "černá magie", kterou zpopularizoval Linuxový kernel. Nevymysleli ho teoretici, ale inženýři, kteří potřebovali extrémně rychlé směrovací tabulky na síťových kartách.
> 
> **Jak to funguje (Abstrakce):**
> 
> Čtenáři neberou **žádné žetony** a nikdy nečekají. Prostě přijdou a čtou data s naprosto nulovou softwarovou režií (Zero Overhead).
> 
> - **Lidský překlad (RCU postup):**
>     
>     1. **Read:** Čtenář si vezme aktuální pointer a rovnou čte.
>         
>     2. **Copy:** Zapisovatel se chce dotknout dat. Nezamkne je! Místo toho si **alokuje úplně novou paměť** a zkopíruje si stará data do sebe.
>         
>     3. **Update:** Zapisovatel přepíše tuto svou privátní kopii.
>         
>     4. **Atomický Swap:** Zapisovatel jedinou strojovou instrukcí na procesoru přepíše hlavní pointer tak, aby ukazoval na jeho novou kopii.
>         
> 
> **Fyzikální analogie: Muzeum s obrazy**
> 
> - **RCU:** V muzeu visí obraz (Verze 1), lidé (vlákna) na něj koukají. Kurátor chce obraz změnit. Nevyhazuje lidi z místnosti. Odejde do ateliéru, namaluje nový obraz (Verze 2) a pověsí ho do vedlejší místnosti. Potom bleskově u vchodu přepíše směrovou šipku z "Obraz -> Místnost A" na "Obraz -> Místnost B".
>     
> - **Grace Period (Doba odkladu):** Nyní se děje to hlavní. Noví návštěvníci jdou automaticky do Místnosti B. Ale v Místnosti A stále stojí pár lidí, kteří tam přišli dříve. Kurátor nesmí Místnost A vyhodit do povětří! Musí počkat u dveří (to je fáze tzv. `synchronize_rcu()`), dokud poslední starý návštěvník neodejde. Teprve pak Místnost A (starou paměť) fyzicky smaže (Deallocation).
>     
> 
> **Pod kapotou (Jak RCU žije v .NETu):**
> 
> V C/C++ se RCU musí složitě implementovat ručně. Zde přichází absolutní Big Picture pro tvou architekturu: **.NET má RCU de facto zabudované v sobě, díky Garbage Collectoru a immutable objektům.**
> 
> Pokud v C# vytvoříš `readonly class`, zapisovatel vyrobí novou instanci a provede `Interlocked.Exchange(ref pointer, novaInstance)`. To je ten okamžitý překlop. A co "Grace Period"? O tu se stará Garbage Collector. GC starou instanci automaticky smaže až ve chvíli, kdy si je stoprocentně jistý, že z ní už žádné čtecí vlákno nečte.
> 
> ### **Side-by-Side: Technické porovnání výkonu**
> 
> |**Metrika**|**RW Lock (ReaderWriterLockSlim)**|**RCU (Read-Copy-Update)**|
> |---|---|---|
> |**Režie čtení (Read Overhead)**|**Nízká, ale ne nulová.** Atomická instrukce ničí CPU Cache a limituje škálování na desítky jader.|**Absolutní nula.** Čtenář jen natáhne pointer z Cache a jede. Ideální pro stovky jader.|
> |**Režie zápisu (Write Overhead)**|**Vysoká.** Blokuje všechny nové čtenáře, musí přepínat kontext operačního systému (spánek vlákna).|**Extrémně vysoká.** Musí se alokovat nová paměť (kopírování dat na haldu) a zatěžuje to Garbage Collector.|
> |**Konzistence čtení**|**Striktní.** Čtenář vždy vidí zaručeně nejnovější data.|**Eventuální (Stale data).** Čtenář může číst z milisekundu staré verze dat, i když už je v systému novější.|
> |**Paměťová náročnost**|Minimální (data existují v RAM jen jednou).|Vysoká (v jeden okamžik může v RAM existovat více starých verzí stejného objektu, než je GC uklidí).|
> 
> ### **Celkový obrázek kontextu: Jaký zvolit kdy?**
> 
> Architektura je o volbě správných kompromisů. Neexistuje stříbrná kulka.
> 
> **Kdy nasadíš RW Lock:**
> 
> - **Poměr čtení/zápis:** Zhruba **10:1** až **100:1**.
>     
> - **Případy užití:** Cache uživatelských sezení, stav lokální in-memory databáze.
>     
> - **Proč:** Potřebuješ striktní konzistenci (nikdo nesmí číst stará data), a data, která měníš, jsou obrovská, takže by bylo šílené (paměťově i časově) je při každém updatu celá kopírovat na novou adresu.
>     
> 
> **Kdy nasadíš RCU (Immutable + Interlocked Exchange):**
> 
> - **Poměr čtení/zápis:** Zhruba **10 000:1** a více. Čtení je drtivě dominantní.
>     
> - **Případy užití:** Globální konfigurační soubory systému (Routing tabulky, Feature Flags, seznam povolených IP adres). Konfigurace se změní jednou za hodinu, ale čte ji každý request statisíckrát za vteřinu.
>     
> - **Proč:** Pokud nasadíš RW lock na globální konfiguraci, kterou čte každé jádro při každém requestu, vytvoříš v systému tzv. _Lock Convoy_ (kolonu na mýtné bráně). CPU se zahltí atomickými operacemi. RCU zajistí, že čtecí potrubí je absolutně čisté a 100% dostupné bez jediné nanosekundy zpoždění, za cenu toho, že při tom jednom zápisu za hodinu GC trochu zapracuje.

# RAII a bezpečné odemykání v C++

## Problém ručního unlock

```cpp
m.lock();
do_work();
m.unlock();  // ← nikdy se neprovede, pokud do_work() hodí exception!
```

**RAII (Resource Acquisition Is Initialization)** — v destruktoru se resource automaticky uvolní, i při výjimce:

```cpp
{
    std::lock_guard<std::mutex> guard(m);  // lock při konstrukci
    do_work();
} // destructor → automatický unlock (i při exception!)
```

## `std::unique_lock` — flexibilní varianta

`std::unique_lock` umí navíc: deferred lock, manuální unlock/lock, `try_lock`, move a čekání na `condition_variable`:

```cpp
std::unique_lock<std::mutex> lock(m);
cv.wait(lock, [] { return ready; });  // atomicky uvolní mutex a čeká
```

# Deadlock

## Definice

Deadlock nastane, když vlákna čekají navzájem na locky a žádné nemůže pokračovat:

```
Thread 1 drží lock A, čeká na lock B
Thread 2 drží lock B, čeká na lock A
→ žádné vlákno nepokročí → deadlock
```

## Coffman conditions — 4 nutné podmínky deadlocku

Deadlock **může nastat pouze tehdy, pokud platí všechny 4**:

1. **Mutual exclusion** — resource může v daný čas vlastnit jen jedno vlákno.
2. **Hold and wait** — vlákno drží resource a čeká na další.
3. **No preemption** — resource nelze odebrat vláknu násilně.
4. **Circular wait** — existuje cyklická závislost (T1 čeká na T2, T2 čeká na T1).

**Stačí porušit jednu podmínku → deadlock nemůže nastat.**

## Prevence deadlocku

```cpp
// Vždy zamykej mutexy ve stejném globálním pořadí (porušuje circular wait):
// Špatně: T1 lock(A), lock(B); T2 lock(B), lock(A)
// Dobře:  T1 i T2 vždy lock(A) pak lock(B)

// std::lock — atomické zamknutí více mutexů bez deadlocku:
std::lock(m1, m2);
std::lock_guard<std::mutex> g1(m1, std::adopt_lock);
std::lock_guard<std::mutex> g2(m2, std::adopt_lock);
```

Další strategie: `try_lock` s timeoutem, krátké kritické sekce, nevolat cizí callback pod lockem.

# Priority Inversion

## Definice

Priority inversion nastane, když **high-priority vlákno čeká na lock držený low-priority vláknem**, ale low-priority vlákno nedostane CPU, protože ho blokuje medium-priority vlákno.

```
High priority:   čeká na mutex
Low priority:    drží mutex, ale nedostane CPU
Medium priority: běží a brání Low v uvolnění mutexu
→ High priority je de facto blokované Medium priority vláknem
```

**Historický příklad:** Mars Pathfinder (1997) — reset systému způsobený priority inverzí [The First Bug on Mars: OS Scheduling, Priority Inversion, and the Mars Pathfinder | by Wahome | Medium](https://kwahome.medium.com/the-first-bug-on-mars-os-scheduling-priority-inversion-and-the-mars-pathfinder-53586a631525).

## Řešení — Priority Inheritance

Low-priority vlákno, které drží lock, dočasně **zdědí prioritu** waitujícího high-priority vlákna → dostane CPU → rychle uvolní lock → priority se vrátí zpět.

Alternativy: **Priority Ceiling** (mutex má nejvyšší prioritu ze všech threadů, které ho mohou uzamknout) nebo kratší kritické sekce.

# Convoy Effect

**Convoy effect** nastane, když jedno pomalé vlákno drží mutex a za ním se hromadí fronta vláken, která čekají (jako konvoj za pomalým autem). Výsledek: dramatické snížení throughputs i při nízkém contention.

**Příčina:** Jedno vlákno drží lock déle (I/O, page fault, OS scheduler) → ostatní vlákna se zablokují → po odemknutí OS probudí všechna → opět contention.

**Řešení:** Zkrátit kritické sekce, vyndat I/O z kritické sekce, nebo použít lock-free přístupy.

# Memory ordering pro C++ atomics

V C++11+ mají atomické operace explicitní **memory order** parameter, který řídí, jak se operace propíše do ostatních vláken:

| Memory order | Záruka | Overhead |
|---|---|---|
| `memory_order_relaxed` | Pouze atomicita, žádné ordering | Nejnižší |
| `memory_order_acquire` | Následující čtení nesmí být přesunuta před acquire | Střední |
| `memory_order_release` | Předchozí zápisy nesmí být přesunuty za release | Střední |
| `memory_order_acq_rel` | Acquire + Release (pro read-modify-write) | Střední |
| `memory_order_seq_cst` | Globální sekvenční pořadí (default) | Nejvyšší |

**Acquire/Release vzor** — publikace dat mezi vlákny:
```cpp
// Producer (writer):
data = 42;
flag.store(true, std::memory_order_release);   // předchozí zápisy viditelné

// Consumer (reader):
while (!flag.load(std::memory_order_acquire)); // následující čtení vidí data
use(data);  // garantovaně vidí data = 42
```

**Relaxed** — vhodné jen pro statistické čítače, kde na pořadí nezáleží:
```cpp
counter.fetch_add(1, std::memory_order_relaxed);
```

> Ke státnicím: Race condition = nedeterministický přístup ke sdíleným datům. Busy wait = plýtvání CPU. Spinlock = intentional busy wait pro krátké CS. Mutex = sleep + OS wakeup. rw-lock = paralelní čtení, exkluzivní zápis. RCU = lock-free čtení + copy-on-write + grace period + memory barriers (ne atomické instrukce!). V Javě nedává RCU smysl — GC řeší uvolnění paměti. Vybírej podle délky CS, poměru čtení/zápisů a počtu jader. RAII = automatický unlock v destruktoru (lock_guard, unique_lock). Deadlock = 4 Coffman podmínky, prevence: globální pořadí locků nebo std::lock. Priority inversion = High čeká na Low, Medium blokuje Low → řeší priority inheritance. Convoy effect = fronta za pomalým vláknem → krátit CS. Memory ordering: relaxed (jen atomicita), acquire/release (synchronizace dat), seq_cst (globální pořadí).
