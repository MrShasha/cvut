Synchronizace je nejzákeřnější zdroj výkonnostních problémů v multi-threaded systémech. Špatná synchronizace nezpůsobí jen pomalý kód — způsobí nesprávný kód. Tato kapitola ti ukáže, proč naivní přístupy selhávají, jaké mechanismy existují (a co se za nimi skrývá "pod kapotou"), a jak zvolit ten správný pro daný workload.

# Problémy naivní synchronizace

## Race condition — proč sdílená data bez ochrany nefungují

**Race condition** nastane, pokud výsledek výpočtu závisí na pořadí, ve kterém vlákna přistupují ke sdíleným datům. Pořadí je nedeterministické → výsledek je nepředvídatelný.

**Příklad:**
```c
bool locked = false;
void func() {
    while (locked == true) { /* busy wait */ }
    locked = true;   // ← PROBLÉM: dvě vlákna mohou projít najednou!
    data++;
    locked = false;
}
```

**Proč to nefunguje:** Mezi testem `while (locked == true)` a nastavením `locked = true` může jiné vlákno projít testem a také nastavit `locked = true`. Oba pak vstoupí do kritické sekce.

**Fyzikální analogie:** Semafor na jednoproudé silnici, kde řidič vidí "zelená" a rozjede se — ale než dorazí na semafor, jiný řidič z druhé strany udělá totéž. Kolize.

## Busy waiting — výkonnostní katastrofa

```c
while (locked == true) { /* busy wait */ }
```

CPU stráví 100 % cyklů testováním proměnné. To je extrémní plýtvání — jiná vlákna nemohou být naplánována na toto jádro.

**Výjimka:** Velmi krátké kritické sekce na vícejádrovém CPU, kde je čekání kratší než overhead kontextového přepínání → spinlock (viz níže).

## ABA problém

Vlákno A přečte hodnotu X, vlákno B změní X→Y→X. Vlákno A vidí X a myslí, že se nic nezměnilo — ale stav systému se změnil. Relevantní pro lock-free algoritmy s CAS.

# Real-world synchronizace: mutex, semaphore, rw-lock, spinlock

## Mutex — základní vzájemné vyloučení

**Mutex** (Mutual Exclusion) zaručuje, že kritickou sekci vykonává vždy právě jedno vlákno.

**Jak funguje pod kapotou:**
1. Vlákno zavolá `lock()` → atomicky zkusí nastavit flag.
2. Pokud je mutex volný → získá ho, vstoupí do CS.
3. Pokud je mutex obsazen → vlákno jde do **sleep** (dá se na wait queue v OS).
4. Po `unlock()` OS probudí čekající vlákno.

**Overhead:**
- Lock/unlock = syscall (kontextový přepínání user↔kernel) → ~1000–5000 ns.
- Moderní "futex" (Linux) — lock je rychlý, pokud není spor; syscall jen při sporu.

**Kdy použít:** Kritické sekce se střední délkou (µs–ms range), kde je spor (contention) nízký.

## Semaphore — počítadlo přístupu

**Semaphore** generalizuje mutex — umožňuje N vláknům vstoupit do CS najednou.
- `sem_wait()` → dekrementuje čítač; pokud 0 → sleep.
- `sem_post()` → inkrementuje čítač; probudí čekající vlákno.

**Kdy použít:** Omezení paralelního přístupu (connection pool, rate limiting). Mutex = binární semaphore.

## Reader-Writer Lock (rw-lock) — oddělení čtecích a zápisových přístupů

**Problém:** Mutex neumožňuje souběžné čtení. Ale čtení je bezpečné paralelně — pouze zápis musí být exkluzivní.

**rw-lock:**
- Libovolný počet **readerů** může číst souběžně.
- Pouze jeden **writer** může zapisovat — a musí počkat na všechny readery.

**Výkon:** Pro read-heavy workload (80–90 % čtení) je rw-lock výrazně rychlejší než mutex.

**Problém rw-locku:** Writer starvation — pokud readery přicházejí neustále, writer se nikdy nedostane na řadu. Implementace to řeší různě (priority writeru).

## Spinlock — busy wait jako záměrná volba

**Spinlock** nedává vlákno do sleep — aktivně točí (spins) ve smyčce:
```c
while (!__atomic_test_and_set(&lock, __ATOMIC_ACQUIRE)) { }
```

**Kdy spinlock dává smysl:**
- Kritická sekce je **velmi krátká** (řádově desítky ns).
- Systém má **víc jader** než vláken (jinak spinnující vlákno blokuje ostatní).
- Cena spinování je nižší než cena kontextového přepínání (~1000+ ns).

**Kdy spinlock škodí:**
- Dlouhé kritické sekce → plýtvání CPU.
- Na jednojádrovém CPU → deadlock (spinning vlákno blokuje i vlákno, co drží lock).

**Adaptive spinlock:** Nejdřív spinuje N iterací, pak přejde do sleep. Nejlepší z obou světů.

## Srovnání mechanismů

| Mechanismus | Overhead locku | Vhodné pro |
|---|---|---|
| **Spinlock** | < 100 ns | Velmi krátké CS, multi-core |
| **Mutex (futex)** | ~200–1000 ns | Střední CS, obecný případ |
| **Semaphore** | ~200–1000 ns | N-way přístup, resource pool |
| **rw-lock** | ~500–2000 ns | Read-heavy workload |
| **RCU** | ~0 ns čtení | Read-mostly, viz níže |

# Read-mostly workload

## Problém s mutex pro read-heavy data

Pokud 95 % přístupů jsou čtení (typické pro konfigurační data, cache, routing tabulky), pak:
- Mutex zbytečně serializuje čtení — pouze jedno vlákno čte najednou.
- rw-lock pomáhá, ale writer pořád musí čekat a pak blokuje všechny readery.

**Fyzikální analogie:** Knihovna, kde každý čtenář musí čekat, až dostane klíč od místnosti — i když čte jinou knihu než ostatní. Nesmysl. Skutečné knihovny nechají vstoupit kohokoliv — uzamknou místnost, jen když přijde vrátný s novými knihami.

# Read-Copy-Update (RCU)

## Myšlenka — lock-free čtení pomocí epochy

**RCU** je synchronizační mechanismus původem z Linuxového kernelu, navržený pro **extrémně read-heavy** workload (přístup ke sdíleným datovým strukturám).

**Klíčová myšlenka:**
- **Readery** nikdy nečekají — přistupují k datům bez jakékoliv synchronizace.
- **Writery** vytvoří **kopii** dat, upraví kopii, a pak atomicky přepnou pointer na novou verzi.
- Stará verze je zachována, dokud ji všechna existující čtecí vlákna nedokončí (grace period).

## RCU v praxi — tři operace

```c
// Čtení (read-side critical section) — žádný lock:
rcu_read_lock();
struct config *cfg = rcu_dereference(global_cfg);  // atomický load
use(cfg);
rcu_read_unlock();

// Zápis (pouze jedno vlákno najednou, typicky pod mutex):
struct config *new_cfg = kmalloc(...);  // alokuj novou kopii
*new_cfg = *old_cfg;
new_cfg->value = new_value;            // uprav kopii
rcu_assign_pointer(global_cfg, new_cfg); // přepni pointer (memory barrier!)
synchronize_rcu();                      // počkej na grace period
kfree(old_cfg);                         // uvolni starý
```

**Klíčový detail:** `rcu_assign_pointer()` používá pouze **compiler a memory barriers** — **ne** atomické instrukce (jako CAS nebo atomic exchange). To je důvodem extrémní rychlosti read-path: čtení je prostý load s read-side memory barrier, ne atomická operace.

## Grace period — kdy je bezpečné uvolnit stará data

**Grace period** je interval, po jehož uplynutí je garantováno, že žádné vlákno nemá referenci na starou verzi dat.

**Jak se detekuje:** Každé vlákno má quiescent state — okamžik, kdy nemůže držet RCU referenci (kontext přepínání, idle). Po tom, co všechna vlákna projdou quiescent state → grace period skončila.

## Výhody a nevýhody RCU

**Výhody:**
- Readery jsou naprosto lock-free → nulový overhead na read path.
- Škáluje perfektně s počtem čtenářů.
- Čtení je race-condition free (atomic pointer load).

**Nevýhody:**
- Paměťový overhead — stará verze existuje až do konce grace period.
- Latence writeru závisí na délce grace period (může být ms–stovky ms).
- Složitost implementace.

**Kde se RCU používá:**
- Linux kernel (routing tabulky, task lists, modul systém).
- Userspace: `liburcu`, garbage collector v Go runtime.

## RCU a Java — proč nedává smysl

**Otázka:** Lze RCU použít v Javě?

**Odpověď: Ne, a tady proč.** Java má garbage collector — GC sám sleduje reference na objekty a automaticky uvolňuje paměť, jakmile na objekt neexistuje žádná reference. RCU řeší přesně tento problém (bezpečné uvolnění staré verze dat) jiným způsobem — explicitní grace period.

GC má ale mnohem vyšší overhead než RCU:
- **Stop-the-world pauzy** — GC občas zastaví všechna vlákna.
- **Marking overhead** — GC musí projít celý heap.

V C/C++ (bez GC) má RCU smysl, protože bez něj bys musel počítat reference (`shared_ptr`) — a to má overhead při každém přístupu. RCU eliminuje reference counting i pro read path.

> Ke státnicím: Race condition = nedeterministický přístup ke sdíleným datům. Busy wait = plýtvání CPU. Spinlock = intentional busy wait pro krátké CS. Mutex = sleep + OS wakeup. rw-lock = paralelní čtení, exkluzivní zápis. RCU = lock-free čtení + copy-on-write + grace period + memory barriers (ne atomické instrukce!). V Javě nedává RCU smysl — GC řeší uvolnění paměti. Vybírej podle délky CS, poměru čtení/zápisů a počtu jader.
