# 1. JVM Internals: bytecode, JIT, safepoints, profiling

→ Detaily v [[8. JVM Internals — bytecode, JIT a safepoints]]

## Klíčové pojmy

| Pojem                          | Vysvětlení                                                                                |
| ------------------------------ | ----------------------------------------------------------------------------------------- |
| **JVM paměťové rozložení**     | Method Area, Heap, Stack (per-thread), PC Register, Native Method Stack                   |
| **JVM Stack frame**            | Jeden frame = jeden aktivní volání metody: local vars + operand stack + constant pool ref |
| **Stack-oriented machine**     | JVM bytecode je zásobníkový stroj: instrukce pracují s operand stack, ne s registry       |
| **Compressed OOP**             | 32-bit pointery na heap (místo 64-bit) pro heap < 32 GB → `XX:+UseCompressedOops`         |
| **Bytecode**                   | Kompaktní instrukce sady (opcode + operandy), interpretována nebo JIT-compilována         |
| **JIT kompilace**              | HotSpot JIT (C1 + C2). C1 = rychlá, menší optimalizace. C2 = agresivní, pomalá kompilace  |
| **Tiered compilation**         | Tier 0 (interpretace) → Tier 1 (C1 bez profil) → Tier 3 (C1 s profilem) → Tier 4 (C2)     |
| **OSR (On-Stack Replacement)** | Náhrada interpretovaného kódu JIT-compilovaným za běhu, uprostřed metody (long loops)     |
| **Safepoint**                  | Bod, kdy lze bezpečně zastavit vlákna (GC, deoptimizace). Vlákna se "synchronizují"       |
| **Time to safepoint**          | Prodleva od žádosti o safepoint po dosažení — kritické pro GC pauzy                       |
| **CPU profiling**              | Sampling (nízký overhead) nebo tracing (přesné). JFR, async-profiler                      |
| **Paměťový profiling**         | jmap -histo, heap dump, Eclipse MAT                                                       |
| **Warm-up**                    | JVM optimalizuje kód postupně → benchmarky nutno zahřát (JMH)                             |

## Nástroje

- `javap -c ClassName` → dekompilace bytecode
- `javap -verbose` → detailní info o class souboru
- `-XX:+PrintAssembly` → disassembly JIT kódu
- Java Flight Recorder (JFR) + JMC → profiling
- async-profiler → CPU + alokační sampling

# 2. Synchronizace: data races, JMM, locking, CAS

→ Detaily v [[9. Data races a synchronizace v JVM]]

## Java Memory Model (JMM)

| Pojem | Vysvětlení |
|---|---|
| **Data race** | Přístup ke sdílené proměnné z více vláken bez synchronizace, alespoň jeden je zápis |
| **Happens-before** | Vztah viditelnosti: A HB B → B vidí vše co A zapisoval. Definuje JMM |
| **`volatile`** | Zaručuje viditelnost + pořadí (memory barrier). Nestačí pro read-modify-write! |
| **CPU pipeline** | Instrukce jsou přeuspořádány OOO procesorem → jiné vlákno vidí jiné pořadí zápisů |
| **Superscalar execution** | Více instrukcí ve stejném clock cyklu → složitější interakce vláken |

## Locking v JVM

| Typ | Overhead | Kdy aktivní |
|---|---|---|
| **Biased locking** | ~0 (jen check mark word) | Objekt vždy zamklo jen jedno vlákno |
| **Thin locking** | ~30 ns (1 CAS) | Nízká konkurence, rychlé locky |
| **Fat locking (monitor)** | ~1000+ ns (OS mutex) | Vysoká konkurence, čekání |

**Reentrant lock:** Explicnitní lock s trylock, timeout, Condition (náhrada wait/notify).

## Atomické operace

| Pojem | Vysvětlení |
|---|---|
| **CAS (Compare-And-Set)** | `CMPXCHG` — hardware atomická operace: pokud value == expected → zapiš new |
| **`AtomicInteger` atd.** | Java obálky nad CAS — lock-free incrementAndGet apod. |
| **Atomic field updaters** | CAS na `volatile int` pole — méně overhead než `AtomicInteger` objekt |
| **ABA problém** | CAS uspěje i když hodnota byla A→B→A. Řešení: `AtomicStampedReference` |

## Non-blocking algoritmy

- **Lock-free:** Vždy alespoň jedno vlákno pokračuje (CAS smyčka).
- **Wait-free:** Každé vlákno dokončí v konečném počtu kroků (silnější záruka).
- **TreibER stack (LIFO):** CAS smyčka na `AtomicReference<Node>` head.
- **`ConcurrentHashMap`:** Lock-free čtení, CAS pro první prvek v bucketu, synchronized na hlavičce pro kolize.

# 3. Synchronizace v C/C++: mutex, semaphore, rw-lock, spinlock, RCU

→ Detaily v [[5. Synchronizace a RCU (C-C++)]]

## Přehled synchronizačních primitiv

| Primitiv | Popis | Kdy použít |
|---|---|---|
| **Mutex** | Vzájemné vyloučení (1 vlákno najednou). OS-level sleep při čekání | Krátké kritické sekce s nízkou konkurencí |
| **Semaphore** | Čítač povolení (signaling + mutual exclusion). `sem_wait`/`sem_post` | Producent-konzument, řízení přístupu k pool zdrojů |
| **RW-lock** | Více čtenářů NEBO jeden writer. `pthread_rwlock_rdlock`/`wrlock` | Read-mostly workload |
| **Spinlock** | Aktivní čekání (busy-wait). Žádný sleep → nulový OS overhead | Velmi krátké sekce, NE pro user-space (mrhá CPU) |
| **RCU (Read-Copy-Update)** | Read-side: lock-free (paměťová bariéra). Write-side: kopíruj, modifikuj, zamkni, swap, čekej na grace period | Kernel, read-mostly s občasným writováním |

## Read-mostly workload — performance bottlenecky

- Writer drží exkluzivní zámek → čtenáři čekají → contention.
- Prioritní inverze: nízko-prioritní writer blokuje vysoko-prioritní čtenáře.
- **RCU výhoda:** Čtenáři nikdy nečekají na writery. Writer čeká na grace period (dokud čtenáři neopustí kritickou sekci).

# 4. Paměť JVM: datové struktury, kolekce, autoboxing

→ Detaily v [[11. Paměť JVM, datové struktury a kolekce]]

## Paměťový overhead

```
int   → 4 B         Integer → 16 B   (4× overhead)
long  → 8 B         Long    → 24 B   (3× overhead)
```

- **Autoboxing:** `list.add(42)` → `Integer.valueOf(42)` → nový heap objekt → GC pressure!
- **Řešení:** Type-specific kolekce (Eclipse Collections, Trove): `IntArrayList`, `IntIntHashMap`

## Analýza paměti

| Pojm | Vysvětlení |
|---|---|
| **Shallow size** | Jen vlastní pole objektu (pointery = 4 B, ne jejich objekty) |
| **Retained size** | Objekty, které by byly uvolněny spolu s tímto objektem (dosažitelný subgraf) |
| **Statická analýza** | Snapshot: `jmap -histo:live <PID>`, heap dump |
| **Dynamická analýza** | Alokační profil v čase: JFR, async-profiler (alloc mode) |
| **Memory leak** | Logicky mrtvý, ale dosažitelný objekt (static kolekce, event listeners, ThreadLocal) |

## Hash tabulky

| Schéma | Cache-friendliness | Clustering |
|---|---|---|
| **Chaining** (Java HashMap) | Nízká (pointery) | N/A |
| **Linear probing** | Vysoká (pole) | Primary clustering |
| **Robin Hood hashing** | Vysoká | Minimální variance |

- **Load factor 0.75** = kompromis kolize vs. paměť (Java default).
- **Growth factor 2** → amortizovaně O(1) insert.

# 5. Alokace objektů, Bloom filtry, typy referencí

→ Detaily v [[12. Alokace objektů, Bloom filtry a typy referencí]]

## Alokace v JVM

| Pojem | Vysvětlení |
|---|---|
| **Bump-the-pointer** | Alokace = posunutí ukazatele (~20 ns). Možné díky compacting GC |
| **TLAB** | Per-thread chunk Eden → lock-free alokace (synchronizace jen při refill) |
| **Escape analysis** | JIT detekuje objekty, které neuniknou metodu → skalární náhrada (žádná heap alokace!) |
| **Humongous objects** | > 50 % G1 regionu → alokovány do Old gen → GC problém |
| **NUMA** | Multi-socket servery: lokální RAM ~100 ns, remote ~200-300 ns. `-XX:+UseNUMA` |

## Bloom filtr

- **Struktura:** Bitové pole délky m + k hash funkcí.
- **Insert:** Pro každou hash funkci nastav bit na 1.
- **Contains:** Pokud jakýkoliv bit = 0 → "určitě NENÍ". Všechny = 1 → "možná JE".
- **False positive:** Bitové pole "přeplněno" → chybné "možná JE". $P_{FP} \approx (1 - e^{-kn/m})^k$
- **Optimální k:** $k = \frac{m}{n} \ln 2$
- **Counting Bloom filtr:** Čítače místo bitů → umožňuje mazání (4–8× větší).
- **Použití:** RocksDB/Cassandra (ušetři disk I/O), Chrome malicious URL check.

## Typy referencí

| Typ | Kdy GC uvolní | Použití |
|---|---|---|
| **Strong** | Nikdy (pokud existuje reference) | Standardní |
| **Soft** | Při memory pressure (OOM hrozí) | Paměťové cache |
| **Weak** | Při první GC | `WeakHashMap`, caching kde klíč je "živý" jen pokud ho jiný drží |
| **Phantom** | Po finalizaci (get() = vždy null) | Cleanup nativních zdrojů |

# 6. Networking a C10K

→ Detaily v [[10. Síťová komunikace a C10K problém]]

## C10K problém

- **Thread-per-request selhání:** 10 000 vláken × 512 KB stack = 5 GB jen na stacky. + context switch overhead.
- **Event-based architektura:** 1 vlákno, event loop, epoll → obsluhuje 10 000+ spojení.

## I/O mechanismy

| Mechanismus | Overhead | Škálovatelnost |
|---|---|---|
| `select` | O(n) scan | Max 1024 FD |
| `poll` | O(n) scan | Neomezený FD, stále pomalý |
| `epoll` (Linux) | O(1) notifikace, O(k) aktivní | 100 000+ FD |
| `kqueue` (macOS/BSD) | O(1) | Ekvivalent epoll |

## Java NIO

- **Direct ByteBuffer** (`allocateDirect`) = native memory, bez GC přesouvání, nulová kopie pro I/O.
- **Channel** = non-blocking I/O endpoint.
- **Selector** = Java abstrakce nad epoll/kqueue.

## Virtual threads (Java 21)

- Miliony lehkých vláken spravovaných JVM.
- Při I/O blokování → JVM automaticky park virtual thread (unmount z carrier thread).
- Blocking kód + event-based škálování. Žádný callback hell.
- **Pinning problém:** `synchronized` blok pinuje virtual thread k carrier thread.

# 7. Cache-efektivní algoritmy a false sharing

→ Detaily v [[6. Cache, paměť a cache-efektivní algoritmy]]

## Cache architektura

| Cache miss typ | Příčina | Řešení |
|---|---|---|
| **Compulsory (cold)** | První přístup | Prefetching, sekvenciální access |
| **Capacity** | Cache je příliš malá | Zmenšit working set, cache-oblivious algo |
| **Conflict** | Mapovací kolize (direct-mapped/set-assoc) | Padding, jiné rozložení dat |

- **Cache line** = typicky 64 B. Hardware vždy přenáší celý cache line.
- **Spatial locality:** Přístup k sousedním adresám → cache hit.
- **Temporal locality:** Opakovaný přístup ke stejné adrese → cache hit.

## Násobení matic — cache tiling

```
Naive:  B[j][k] = column access → stride = N → cache miss každý přístup
Tiled:  Zpracuj blok (tile) A, B, C najednou → drží v L1/L2 cache

Pro tiled: výkon může být 10× lepší pro velké matice
```

## False sharing

- Dvě vlákna zapisují do různých proměnných, které **sdílí jeden cache line**.
- Každý zápis invaliduje cache line u druhého vlákna → "ping-pong" cache line.
- **Řešení:** Padding na 64 B hranici. Java 8: `@Contended` anotace. Nebo oddělení proměnných.

## NUMA

- Dvě CPU sockety → každý má svou RAM. Přístup na druhý socket = 2–3× pomalejší.
- `-XX:+UseNUMA` → JVM alokuje TLAB na NUMA uzlu vlákna.

## Multi-core cache coherency

- **MESI protokol:** Modified, Exclusive, Shared, Invalid — cache line stavy.
- Write → ostatní kopie Invalid. Invalid → fetch ze sdílené cache/RAM.

# 8. Profiling a optimalizace C/C++: HW counters, PGO, compiler passes

→ Detaily v [[3. Překladač, LLVM a profile-guided optimization]] a [[4. Benchmarking a měření výkonu]]

## Hardware performance counters

Přístup přes `perf` (Linux) nebo Intel VTune:

| Čítač | Co měří |
|---|---|
| `cache-misses` | L1/L2/L3 cache miss (velmi drahé) |
| `branch-misses` | Špatné předpovědi větví (pipeline flush) |
| `instructions` / `cycles` | IPC (instructions per cycle) |
| `page-faults` | TLB miss → OS stránkování |

```bash
perf stat -e cycles,instructions,cache-misses ./program
perf record -g ./program ; perf report
```

## PGO (Profile-Guided Optimization)

1. **Instrumentovaný build:** Binárka sbírá profil za běhu.
2. **Spuštění na reprezentativních datech** → profil (větve, call freq, inlining hints).
3. **Optimalizovaný build:** Kompilátor použije profil → lepší inlining, branch layout.

**Výsledek:** Typicky 5–15 % zlepšení výkonu pro compute-heavy kód.

## LLVM optimalizační průchody

| Úroveň | Příklady průchodů |
|---|---|
| **High-level (IR)** | Inlining, dead code elimination, constant folding, LICM, vectorization |
| **Low-level (codegen)** | Register allocation, instruction selection, loop unrolling, peephole |

**Kompilační pipeline:**
```
Zdrojový kód → [Parser] → AST → [Sémantická analýza] → IR → 
[Optimalizační průchody] → Low-level IR → [Codegen] → Strojový kód
```

**Optimalizace co musí dělat programátor:**
- Aliasing (`__restrict__`, `const` correctness).
- Data layout (SoA vs. AoS pro SIMD/cache).
- Avoid UB (undefined behavior blokuje optimalizace).

## Q1 — JVM: Virtual machine, výhody, nevýhody, JIT kompilátor
**Hnětynka | červen 2025**

> *(Java) Virtual machine, jak funguje, výhody, nevýhody. JIT kompilátor. Doplňující: co je potřeba pro implementaci JVM na nové platformě, jaké jiné VM znám, jestli se JVM používá na Androidu. U JITu: jak ví u null check elimination, že tam null nikdy nepřijde?*

→ Detaily: [[8. JVM Internals — bytecode, JIT a safepoints]]

**Jak JVM funguje:**
- JVM = zásobníkový virtuální stroj. Javový kód se přeloží do bytecodu (`.class` soubory), který JVM interpretuje nebo JIT-compiluje do nativního strojového kódu.
- Paměťové rozložení: **Heap** (objekty, GC), **Stack** (frame per volání metody: local vars + operand stack), **Method Area** (metadata tříd, konstanty), **PC Register** (ukazatel na aktuální instrukci), **Native Method Stack**.
- **JVM Stack frame** = lokální proměnné + operand stack + reference na constant pool.

**Výhody JVM:**
- **Platform independence** — "Write once, run anywhere". Bytecode běží na jakékoliv platformě s JVM.
- **Automatic memory management** — GC spravuje heap, žádné memory leaks (logické leaky jsou ale možné).
- **JIT optimalizace** — za běhu profilení a optimalizace pro konkrétní hardware.
- **Bezpečnost** — bytecode verifier kontroluje bezpečnost před spuštěním.

**Nevýhody JVM:**
- **Warm-up** — JIT potřebuje čas k profilování a kompilaci; začátek je pomalejší.
- **Memory overhead** — JVM samotné spotřebovává paměť (metaspace, overhead každého objektu ~16B header).
- **GC pauzy** — Stop-the-world pauzy ovlivňují latenci.
- **Nižší předvídatelný výkon** — JIT může deoptimizovat, GC může přerušit.

**JIT kompilátor (tiered compilation):**
- **Tier 0:** Čistá interpretace (nejpomalejší).
- **Tier 1–3:** C1 kompilátor — rychlá kompilace s různou úrovní profilování.
- **Tier 4:** C2 kompilátor — agresivní optimalizace (inlining, escape analysis, SIMD).
- JIT profiluje: které cesty jsou "hot" → optimalizuje jen hot spots.

**Jak JIT ví u null check elimination, že null nikdy nepřijde?**
- JIT sbírá profil: pozoruje typy a hodnoty za běhu (type profiling, value profiling).
- Pokud v tisících voláních `obj` nebyl nikdy `null` → JIT spekulativně odstraní null check + generuje **deoptimization trap**: pokud null přijde v budoucnu → deoptimize zpět na interpretaci.
- Mechanismus: **uncommon trap** (HotSpot terminologie) — "tato větev se dosud nikdy nestala, vygeneruju ji jako deopt."

**Co je potřeba pro implementaci JVM na nové platformě?**
- Implementovat JIT backend pro cílovou ISA (x86/ARM/RISC-V codegen).
- Implementovat GC s podporou OS paměťového modelu.
- Implementovat JNI (Java Native Interface) pro volání nativního kódu.
- Základní knihovny (java.lang, java.io...).

**Jiné JVM / VM implementace:**
- **GraalVM** — polyglot VM (Java, JS, Python, Ruby). Má i AOT kompilátor (Native Image).
- **OpenJ9** (IBM) — alternativa HotSpot, nižší footprint.
- **ART (Android Runtime)** — NA Androidu se nepoužívá HotSpot JVM. ART používá **AOT kompilaci** (přeloží bytecode při instalaci aplikace) + JIT od Android 7 (profil-guided JIT).

> [!note] Ke státnicím
> JVM = zásobníkový stroj, bytecode → JIT → nativní kód. Tiered: Tier 0 (interpret) → Tier 4 (C2 agresivní). Null check elimination = spekulativní optimalizace + deopt trap. Android = ART (AOT + JIT), ne HotSpot. Výhody: portabilita, GC, JIT opt. Nevýhody: warm-up, GC pauzy, overhead.

## Q2 — Efektivní servery a C10K problém
**Vokřínek | leden 2022**

> *Efektivní servery — C10K problém. Vysvětlete o co se jedná, proč není vhodný „klasický" přístup, vysvětlete rozdíl přístupů. Jaký má vliv použití procesů, vláken, I/O operací.*

→ Detaily: [[10. Síťová komunikace a C10K problém]]

**Co je C10K problém:**
- Jak obsluhovat 10 000 souběžných síťových spojení na jednom serveru?
- Dan Kegel (1999) pojmenoval problém: tehdy "klasické" servery kolabovaly kolem 10 000 klientů.

**Proč klasický (thread-per-request) přístup selhává:**
- **Paměť:** Každé OS vlákno = stack ~512 KB – 2 MB. 10 000 vláken = **5–20 GB** jen na stacky.
- **Context switch overhead:** OS musí přepínat mezi 10 000 vlákny (ukládání/obnova registrů, TLB flush). Při I/O-bound workloadu vlákna **čekají** (na síť, disk, DB) → CPU je idle, ale vlákna existují a blokují.
- **Kernel scheduling overhead:** OS scheduler s 10 000 vlákny je signifikantně pomalejší než se 100 vlákny.

**Vliv procesů vs. vláken vs. I/O operací:**

| Přístup | Paměťový overhead | Izolace | Komunikace | Škálovatelnost |
|---|---|---|---|---|
| **Procesy** (fork per request) | Velmi vysoký (celý address space) | Silná | IPC (drahá) | Nejhorší |
| **Vlákna** (thread per request) | Střední (jen stack) | Slabá (sdílená paměť) | Sdílená paměť | Střední |
| **Non-blocking I/O + event loop** | Minimální (1 vlákno) | Žádná (1 vlákno) | Callback/future | Nejlepší |

**Přístupy řešení C10K:**
1. **Thread pool** — omezený počet vláken (100–1000), fronta requestů. Pořád blokuje pool při I/O.
2. **Non-blocking I/O + epoll** — jedno vlákno obsluhuje tisíce spojení. Výzva: callback hell.
3. **Reaktivní programování** — (Reactor pattern, Netty, Node.js) — event-driven, composable futures.
4. **Virtual threads (Java 21)** — miliony lehkých JVM vláken, automatický park při I/O blokování. Kombinuje jednoduchost vláken + škálovatelnost event-based.

**epoll jako klíčový OS mechanismus:**
- `select`/`poll` = O(n) scan přes všechny FD → nepoužitelné pro 10K+ spojení.
- `epoll` (Linux) = kernel udržuje seznam sledovaných FD, notifikuje jen aktivní → **O(1) na notifikaci**.

> [!note] Ke státnicím
> C10K = 10 000 spojení, thread-per-request selhává kvůli paměti (5–20 GB stacky) a context switch. Procesy > vlákna > non-blocking I/O pro škálovatelnost. epoll = O(1) notifikace. Virtual threads (Java 21) = nejlepší z obou světů.

## Q3 — Synchronizace: primitiva, média, škálovatelnost
**Sojka | září 2021**

> *Synchronizace, synchronizační média, synchronizační primitiva, vysvětlit škálovatelnost. Doplňující: CAS a atomické operace (Java + C++), mutex, futex (co je a jak je implementován), memory barriers a fence instrukce, Java synchronized, RCU — jak funguje, jak je implementováno.*

→ Detaily: [[5. Synchronizace a RCU (C-C++)]] | [[9. Data races a synchronizace v JVM]]

**Synchronizační primitiva — přehled:**

| Primitiv | Mechanismus | Overhead | Kdy použít |
|---|---|---|---|
| **Atomic ops (CAS)** | Hardware instrukce (`CMPXCHG`) | ~1–5 ns | Lock-free algoritmy, čítače |
| **Spinlock** | Aktivní čekání na CAS | Mrhá CPU (busy-wait) | Velmi krátké sekce, kernel |
| **Mutex** | OS-level, thread spí při čekání | ~1000 ns (syscall) | Obecné kritické sekce |
| **Futex** | Fast userspace mutex | ~50 ns (uncontested) | Implementace mutexu v Linuxu |
| **Semaphore** | Čítač + blokování | Podobný mutexu | Producent-konzument, pooling |
| **RW-lock** | Více čtenářů / 1 writer | Nízký pro čtení | Read-mostly workload |
| **RCU** | Lock-free čtení, wait na grace period | ~0 pro čtení | Kernel, read-mostly struktury |

**CAS (Compare-And-Set) — základ atomických operací:**
```
CMPXCHG(adresa, expected, new):
  atomicky: if *adresa == expected → *adresa = new, vrať true
            else vrať false
```
Java: `AtomicInteger`, `AtomicReference`, `VarHandle`. C++: `std::atomic<T>::compare_exchange_strong/weak`.

**Futex (Fast Userspace Mutex) — jak je implementován:**
- Kombinuje userspace CAS (rychlá cesta) + kernel syscall (pomalá cesta).
- **Uncontested lock:** CAS v userspace → zamčeno bez syscallu (~50 ns).
- **Contested lock:** CAS selže → `futex(WAIT)` syscall → OS thread přejde do sleep. Unlock: CAS + `futex(WAKE)` pokud čekají vlákna.
- `pthread_mutex_t` v Linuxu je implementován přes futex.

**Memory barriers a fence instrukce:**
- CPU přeuspořádává paměťové přístupy (out-of-order execution). Memory barrier = instrukce, která toto zabrání.
- `mfence` (x86) = full memory barrier (StoreStore + LoadLoad + StoreLoad).
- `sfence` = store barrier, `lfence` = load barrier.
- Java: `volatile` → implicitní bariéry. C++: `std::atomic` + `memory_order_seq_cst`.

**RCU (Read-Copy-Update) — jak funguje:**
- **Čtení (read-side):** `rcu_read_lock()` = zabrání preempci (kernel) nebo oznámí vstup do kritické sekce. Žádný lock! Jen paměťová bariéra.
- **Zápis (write-side):** Kopíruj → modifikuj kopii → atomicky zamkni pointer na novou verzi → `synchronize_rcu()` = čekej na **grace period** (dokud všechna vlákna, která viděla starý pointer, neopustí critical section) → uvolni starou verzi.
- **Implementace grace period:** Kernel RCU sleduje, zda každý CPU prošel alespoň jedním context switchem (po context switchu = určitě mimo RCU critical section). Uživatelský prostor: quiescent state tracking.

**Škálovatelnost synchronizace:**
- Mutex = globální bottleneck → neškáluje na mnoho jader (lock contention).
- CAS smyčka = škáluje lépe, ale při vysoké konkurenci dochází k livelock/spins.
- RCU = read-side lineárně škáluje s počtem jader (žádná koordinace čtenářů).
- Fine-grained locking (`ConcurrentHashMap`) = škáluje lépe než global lock.

> [!note] Ke státnicím
> CAS = hardware atomická instrukce, základ lock-free. Futex = CAS v userspace + kernel WAIT/WAKE. Memory barrier = zákaz přeuspořádání paměťových přístupů. RCU = lock-free čtení, write-side čeká na grace period (všechna vlákna mimo critical section). Škálovatelnost: RCU > fine-grained lock > global mutex.

## Q4 — Virtualizace: simulace, nativní virtualizace, paravirtualizace
**Pevný | červen 2018**

> *Vysvětlete rozdíl mezi simulací, úplnou nativní virtualizací a paravirtualizací. Dále řekněte, kdy je nutné pro virtualizaci, aby se host přizpůsobil hostiteli.*

→ Detaily: [[14. Virtualizace]]

**Simulace (emulace):**
- Emuluje **jinou hardwarovou architekturu** v softwaru. Guest OS nevyžaduje žádnou úpravu.
- Příklady: QEMU bez KVM (emuluje ARM na x86), MAME (herní konzole).
- **Výkon:** Velmi pomalý — každá instrukce je softwarově interpretována. 10–100× pomalejší než nativní.
- **Kdy použít:** Vývoj embedded systémů, cross-platform testování, retrogaming.

**Plná (nativní) virtualizace:**
- Guest OS běží **nezměněný** jako by byl na fyzickém hardware. Guest neví, že je virtualizovaný.
- Hypervisor zachytává privilegované instrukce (VM exit) a emuluje je.
- **Bez HW podpory (binary translation):** Hypervisor překládá privilegované instrukce Guest OS za běhu → overhead.
- **S HW podporou (Intel VT-x / AMD-V):** Guest OS běží v VMX non-root Ring 0, privilegované instrukce automaticky způsobují VM exit → hypervisor zachytí a zpracuje. Moderní standard.
- Příklady: VMware ESXi, KVM, Hyper-V, VirtualBox.
- **Výkon:** ~5–10 % overhead oproti nativnímu (s VT-x).

**Paravirtualizace:**
- Guest OS **ví, že je virtualizovaný** a místo privilegovaných instrukcí volá **hypercalls** (hypervisor API).
- Guest musí být upraven (kernel patch nebo speciální driver).
- **Výkon:** Lepší než plná virtualizace bez HW podpory (přímé hypercalls místo trap+emulate). S VT-x je rozdíl menší.
- Příklady: Xen (PV mode), **Virtio** (paravirtualizovaný I/O driver — klíčový příklad!).
- Virtio = paravirtualizovaný síťový/diskaří driver: Guest OS volá Virtio API → hypervisor backend → fyzický hardware. Efektivnější než plná emulace NIC.

**Přehledová tabulka:**

| Typ | Guest upraven? | Výkon | Příklady |
|---|---|---|---|
| Simulace (emulace) | Ne | Velmi pomalý (10–100×) | QEMU bez KVM, MAME |
| Plná virtualizace (bez HW) | Ne | Pomalý (binary translation) | VMware Workstation (starý) |
| Plná virtualizace (s VT-x) | Ne | Rychlý (~5 % overhead) | KVM+VT-x, ESXi, Hyper-V |
| Paravirtualizace | Ano | Rychlý, nižší overhead | Xen PV, Virtio drivers |
| Kontejnerizace (OS-level) | Ne (sdílí kernel) | Téměř nativní | Docker, LXC |

**Kdy je nutné, aby se host přizpůsobil hostiteli (= paravirtualizace)?**
- Když hypervisor **neposkytuje hardware emulaci** pro I/O (disk, síť) — guest musí používat speciální Virtio driver.
- Historicky na Xen: bez HW asistence musel být Linux kernel patchovaný pro běh v Ring 1.
- **Příklad Virtio:** Aby KVM guest efektivně využíval síť, musí mít nainstalovaný `virtio-net` driver (místo emulovaného Intel e1000). Tento driver komunikuje přímo přes virtqueue (sdílená paměť) s hypervisorem.
- Moderně: Většina distribucí Linuxu má Virtio drivery v kernelu → paravirtualizovaný I/O bez nutnosti explicitní úpravy.

> [!note] Ke státnicím
> Simulace = SW interpretace jiné ISA (pomalá). Plná virtualizace = guest neví, používá VT-x/AMD-V (VM exit/entry). Paravirtualizace = guest ví, volá hypercalls (Virtio je klíčový příklad). Host se musí přizpůsobit = paravirtualizace (Virtio driver, Xen PV kernel). Moderní: VT-x + Virtio I/O = nejlepší kompromis.
