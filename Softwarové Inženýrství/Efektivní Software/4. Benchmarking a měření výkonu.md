Benchmarking je věda, ne magie. Špatně navržený benchmark je horší než žádný benchmark — dává ti falešnou jistotu. Tato kapitola tě naučí navrhnout experiment, který skutečně měří to, co chceš měřit, a jehož výsledky jsou reprodukovatelné, srovnatelné a statisticky smysluplné.

# Principy správného benchmarkingu

## Čtyřfázový optimalizační cyklus

Benchmarking je vstupní brána do optimalizačního cyklu:

```
1. MĚŘ    → benchmark, profiler → najdi bottleneck
2. ROZUMĚJ → proč je to pomalé? (cache miss? branch? I/O? lock?)
3. ZMĚŇ   → jedna změna najednou
4. OVĚŘ   → benchmark znovu → je to skutečně lepší?
```

**Zlaté pravidlo:** Vždy změř nejdřív, optimalizuj až potom. Premature optimization je kořenem všeho zla (Knuth).

## Proč naivní měření lže

Zdrojů šumu je mnoho:

- **CPU frequency scaling (turbo boost, power saving)** — CPU mění frekvenci za běhu. Jeden benchmark může běžet na 2,5 GHz, druhý na 4,5 GHz.
- **Thermal throttling** — CPU zahřeje → sníží frekvenci → výsledky degradují uprostřed benchmarku.
- **OS scheduler interference** — tvůj proces je přerušen jiným procesem, irq handlerem, GC.
- **NUMA effects** — data alokovaná na jiném NUMA uzlu než CPU → vyšší latence.
- **Cache state** — první spuštění = cold cache, druhé = warm cache. Měříš cold nebo warm?
- **JIT warm-up (JVM)** — první iterace je interpretovaná, po N iteracích JIT zkompiluje → dramatická změna výkonu.
- **Branch predictor state** — stav prediktoru závisí na předchozím kódu.

## Benchmark design — jak to dělat správně

- **Izolovanost** — benchmark měří jednu věc, ne vše najednou.
- **Reprodukovatelnost** — stejný hardware, stejný OS stav, fixovaná CPU frekvence (`cpupower frequency-set`).
- **Reprezentativní vstup** — vstupní data musí odpovídat reálnému workloadu.
- **Dostatečná délka** — benchmark musí běžet dostatečně dlouho, aby byl statisticky významný.

# Měření doby běhu

## Timestamping — jak správně měřit čas

Různé hodiny mají různou přesnost a overhead:

| Hodiny | Přesnost | Overhead | Kdy použít |
|---|---|---|---|
| `clock_gettime(CLOCK_MONOTONIC)` | ~1 ns | nízký | C/C++ benchmarky |
| `std::chrono::high_resolution_clock` | ~1 ns | nízký | C++ |
| `System.nanoTime()` | ~1 ns | nízký | Java |
| `RDTSC` (CPU timestamp counter) | 1 cyklus | velmi nízký | mikrobenchmarky |

**Pozor na RDTSC:** Není garantovaně monotónní přes jádra (bez `RDTSCP`). Moderní CPU s invariant TSC jsou OK.

## Warm-up fáze — povinná při JVM měření

```
Cold start → interpretovaný bytecode (pomalé)
  ↓ ~1 000–10 000 invokací
C1 kompilátor → základní JIT kompilace
  ↓ ~10 000–100 000 invokací
C2 kompilátor → agresivní optimalizace (inlining, vectorization)
  ↓ TEPRVE TEĎ měř!
```

**Pravidlo:** Pro JVM benchmarky vždy nejdřív "zahřej" JVM — spusť měřený kód mnohokrát bez měření, pak teprve měř.

**Nástroj:** JMH (Java Microbenchmark Harness) — automaticky řeší warm-up, opakování a statistiku.

## Opakování iterací a exekucí

Jeden běh nestačí — výsledek je zatížen šumem. Musíš opakovat na dvou úrovních:

- **Iterace** — N opakování měřeného kódu v jednom spuštění (průměr přes iterace eliminuje náhodný šum).
- **Exekuce** — M spuštění celého programu (eliminuje variabilitu způsobenou cold start, OS stavem).

**Multi-level repetition:**
```
Pro každý run (m = 1..M):
  Warm-up fáze (ignoruj výsledky)
  Pro každou iteraci (n = 1..N):
    Změř čas t_n
  Zaznamenej statistiky (mean, std, median)
Porovnej statistiky přes runs
```

## Měření speedup

$$\text{Speedup}(n) = \frac{T_{\text{reference}}}{T_{\text{optimized}}}$$

**Jak správně porovnávat:**
- Porovnávej mediány (robustnější vůči outlierům než průměr).
- Uveď confidence interval (95% CI).
- Geomean pro agregaci speedup přes více benchmarků.

# Měření spotřeby energie a paměti

## Měření spotřeby energie — RAPL

**RAPL (Running Average Power Limit)** je Intel/AMD mechanismus pro měření spotřeby energie:
- `perf stat -e power/energy-pkg/` — spotřeba celého socketu.
- `perf stat -e power/energy-ram/` — spotřeba DRAM.

**Proč měřit energii?** Na serverech energie = peníze. Na mobilních zařízeních energie = výdrž baterie. Algorithmus s nižší latencí nemusí být energeticky efektivnější (může běžet na vyšší frekvenci).

## Měření paměti

- **Peak RSS** (Resident Set Size) — maximální fyzická paměť procesu: `valgrind --tool=massif`, `/usr/bin/time -v`.
- **Heap profiler** — kde se alokuje: `heaptrack`, `Valgrind Massif`.
- **Cache usage** — `perf stat -e cache-misses,cache-references`.

# Statistická analýza výsledků

## Průměr vs. medián — kdy co použít

| Metrika | Vhodné pro | Nevhodné pro |
|---|---|---|
| **Průměr (mean)** | normální distribuce, bez outlierů | distribuce s outliers (GC pauses) |
| **Medián** | skewed distribuce, s outliers | jemné rozdíly (low variance data) |
| **P99 (99. percentil)** | latence tail (worst case uživatele) | average case analýzu |
| **Geomean** | agregace speedup přes různé benchmarky | absolutní časy |

**Fyzikální analogie:** Průměrná latence serverů je nesmyslná, pokud 1 % requestů trvá 10× déle. P99 říká, co zažije 1 % uživatelů — a to je to, co zákazníka trápí.

## Variance a šum

Pokud je variance příliš vysoká, výsledky jsou nespolehlivé. Možné příčiny:
- CPU frequency scaling → fixuj frekvenci.
- OS interference → pin proces na jádro (`taskset`), vypni hyperthreading.
- NUMA → alokuj data na stejném uzlu jako CPU (`numactl`).
- Disk/síť → nemá co dělat v CPU benchmarku → izoluj.

## Jak reportovat výsledky

Dobrý benchmark report obsahuje:
1. **Hardware specifikace** (CPU model, frekvence, RAM, OS).
2. **Nastavení benchmarku** (počet iterací, warm-up, vstupní data).
3. **Statistiky** (mean/median, std dev, min, max, P95/P99).
4. **Confidence interval** nebo p-value pro srovnání.
5. **Zdrojový kód benchmarku** (reprodukovatelnost).

> Ke státnicím: Benchmarking = měř–rozuměj–změň–ověř. Šum pochází z CPU scaling, GC, OS scheduler, cold cache, JIT warm-up. Warm-up je nutný pro JVM. Iterace i exekuce opakuj. Medián > průměr pro skewed distribuce. RAPL pro energii. P99 pro latenci.
