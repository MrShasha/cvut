# OSI model a síťová komunikace

## OSI model — přehled vrstev

OSI model popisuje, jak data putují od aplikace na fyzické médium a zpět:

```
Vrstva 7: Aplikační    (HTTP, gRPC, WebSocket)
Vrstva 6: Prezentační  (TLS/SSL, serializace)
Vrstva 5: Relační      (session management)
Vrstva 4: Transportní  (TCP, UDP) ← sockety jsou zde
Vrstva 3: Síťová       (IP, routing)
Vrstva 2: Linková      (Ethernet, MAC adresy)
Vrstva 1: Fyzická      (kabely, Wi-Fi, optika)
```

**Pro efektivního inženýra jsou klíčové vrstvy 4 a 7.**

## TCP vs. UDP

| | TCP | UDP |
|---|---|---|
| Spojení | Connection-oriented (3-way handshake) | Connectionless |
| Spolehlivost | Garantované doručení, správné pořadí | Žádná záruka |
| Overhead | Vyšší (ACK, congestion control) | Nižší |
| Kdy použít | HTTP, databáze, SSH | Video streaming, gaming, DNS |

## Sockety a identifikace spojení

**Socket** = endpoint síťového spojení, identifikovaný **4-tuplí:**
`(zdrojová IP, zdrojový port, cílová IP, cílový port)`

- IPv4: 32-bit adresa, IPv6: 128-bit adresa.
- Porty: 16-bit čísla (0–65535). Systémové porty: 0–1023, uživatelské: 1024–65535.

# C10K problém

## Thread-per-request architektura — kde selhává

**Thread-per-request** = pro každý příchozí request vytvoř nové vlákno (nebo vezmi z poolu):

```
Klient 1 → Vlákno 1 → zpracování (čeká na DB 200ms)
Klient 2 → Vlákno 2 → zpracování (čeká na DB 200ms)
...
Klient 10000 → Vlákno 10000 → ???
```

**Proč to zkolabuje na 10 000 spojení:**
- **Paměť:** Každé vlákno = stack (~512 KB – 2 MB). 10 000 vláken = 5–20 GB jen na stacky.
- **Kontextové přepínání:** OS musí přepínat mezi 10 000 vlákny → overhead roste.
- **Většina vláken čeká na I/O:** CPU je idle, ale vlákna existují a blokují.

**Fyzikální analogie:** Restaurace s 10 000 stoly, kde každý stůl má vlastního číšníka. Číšník čeká u stolu, dokud zákazník nejí — jen sedí a nic nedělá. Lepší model: jeden číšník obsluhuje více stolů a jde k zákazníkovi jen když ho potřebuje.

## Event-based architektura — jeden vlákno, mnoho spojení

**Event loop** = jedno (nebo málo) vláken, která neustále kontrolují, která spojení jsou ready pro I/O, a zpracovávají je:

```
while (true) {
    ready_events = wait_for_io_events(all_sockets)  // epoll
    for event in ready_events:
        handle(event)  // rychlé zpracování, nesmí blokovat!
}
```

**Klíčový požadavek:** Handlery nesmí blokovat — nesmí čekat na DB, disk nebo síť. Pokud blokují, celý event loop čeká.

**Kde to používají:** nginx, Node.js, Redis, Kafka — single-threaded event loop, obsluhují desítky tisíc spojení.

# Non-blocking I/O: select, poll, epoll

## Blocking vs. non-blocking I/O

**Blocking I/O:**
```c
int bytes = read(fd, buffer, size);  // Vlákno čeká, dokud nejsou data → CPU idle
```

**Non-blocking I/O:**
```c
fcntl(fd, F_SETFL, O_NONBLOCK);
int bytes = read(fd, buffer, size);  // Okamžitě vrátí -1 (EAGAIN) pokud žádná data
```

Ale jak víš, kdy jsou data připravena? → Potřebuješ mechanismus pro monitoring více socketů najednou.

## `select()` — historický základ

```c
fd_set readfds;
FD_SET(socket1, &readfds);
FD_SET(socket2, &readfds);
select(maxfd + 1, &readfds, NULL, NULL, &timeout);  // Blokuje až do aktivity
// Pak zkontroluj FD_ISSET(socket1, &readfds) pro každý socket
```

**Omezení:** Maximum 1024 file descriptorů. O(n) scan přes všechny FD. Nepraktické pro 10 000+ spojení.

## `poll()` — zlepšení select

Odstraňuje limit 1024 FD, ale stále O(n) scan — pro velký počet spojení je stále pomalé.

## `epoll` (Linux) — O(1) event notification

**epoll** = Linux mechanismus pro efektivní monitoring velkého počtu FD:

```c
int epfd = epoll_create1(0);

struct epoll_event ev = {.events = EPOLLIN, .data.fd = socket1};
epoll_ctl(epfd, EPOLL_CTL_ADD, socket1, &ev);  // Zaregistruj socket

// Event loop:
while (true) {
    int n = epoll_wait(epfd, events, MAX_EVENTS, timeout);  // Čekej
    for (int i = 0; i < n; i++) {
        handle(events[i].data.fd);  // Zpracuj jen aktivní sockety
    }
}
```

**Proč epoll škáluje:** OS udržuje seznam sledovaných FD v kernelu. `epoll_wait` vrátí jen aktivní FD → O(1) pro přijatí notifikace, O(k) pro zpracování kde k = počet aktivních FD.

**macOS/BSD ekvivalent:** `kqueue`. **Windows ekvivalent:** `IOCP` (I/O Completion Ports).

## Edge-triggered vs. level-triggered epoll

- **Level-triggered (default):** epoll notifikuje dokud je FD ready (i pokud ho ignoruješ).
- **Edge-triggered (EPOLLET):** notifikuje jen při změně stavu (nová data přišla). Efektivnější, ale složitější (musíš přečíst vše najednou).

# Java NIO: native buffery, kanály a selektory

## Native memory buffers — ByteBuffer

Java NIO zavádí `ByteBuffer` jako abstrakci nad paměťovým bufferem:

- **Heap buffer** (`ByteBuffer.allocate(size)`) — v JVM heap (managed). Podléhá GC.
- **Direct buffer** (`ByteBuffer.allocateDirect(size)`) — **mimo JVM heap** (native memory). Není přesouván GC → efektivní pro I/O (OS může přímo číst/zapisovat bez kopírování).

**Proč direct buffer pro I/O:** JVM heap buffer musí být před každým I/O operací zkopírován do native memory (OS nemůže pracovat s přesouvanými heap objekty). Direct buffer = nulová kopie.

## Channels — non-blocking I/O v Javě

**`Channel`** = non-blocking obdoba `InputStream`/`OutputStream`:
- `SocketChannel` — TCP spojení.
- `ServerSocketChannel` — příjem TCP spojení.
- `FileChannel` — soubory.

```java
ServerSocketChannel server = ServerSocketChannel.open();
server.configureBlocking(false);  // Non-blocking mód
server.bind(new InetSocketAddress(8080));
```

## Selector — Java epoll abstrakce

**`Selector`** = Java obálka nad `select`/`poll`/`epoll`/`kqueue`:

```java
Selector selector = Selector.open();
server.register(selector, SelectionKey.OP_ACCEPT);  // Zaregistruj zájem

while (true) {
    selector.select();  // Čekej na eventi (jako epoll_wait)
    Set<SelectionKey> keys = selector.selectedKeys();
    
    for (SelectionKey key : keys) {
        if (key.isAcceptable()) { /* Nové spojení */ }
        if (key.isReadable()) { /* Data k přečtení */ }
        if (key.isWritable()) { /* Lze zapisovat */ }
    }
    keys.clear();
}
```

**Výhoda:** Jedno vlákno obsluhuje tisíce spojení. **Nevýhoda:** Komplexní kód, callback hell.

# Thread pools a virtual threads

## Thread pools — kompromis

Thread pool = předem vytvořená skupina vláken. Přijatý request se přidá do fronty, první volné vlákno ho zpracuje.

```java
ExecutorService pool = Executors.newFixedThreadPool(100);
pool.submit(() -> handleRequest(request));
```

**Výhody:** Omezení počtu vláken → kontrola paměti a overhead. Znovupoužití vláken (bez thread creation overhead).

**Nevýhody:** Pořád existuje limit. I/O-bound tasky blokují vlákno z poolu → ostatní requesty čekají.

## Virtual threads (Project Loom, Java 21+) — změna paradigmatu

**Virtual thread** = lehký thread spravovaný JVM (ne OS). JVM může mít miliony virtual threadů, mapovaných na malý počet OS vláken (carrier threads).

**Klíčová vlastnost:** Při blokujícím I/O volání JVM **automaticky odmontuje** virtual thread z carrier threadu a přidělí carrier thread jinému virtual threadu.

```
Virtual Thread 1: čeká na DB → JVM ho odloží (unmount z carrier thread)
Virtual Thread 2: zpracovává CPU task → běží na uvolněném carrier thread
Carrier Thread: nikdy neblokuje → 100 % CPU využití
```

**Výhoda oproti NIO/epoll:** Kód vypadá jako **synchronní, blokující** kód:

```java
// Virtual thread kód (vypadá blocking, JVM ho automaticky park/unpark):
try (var conn = db.connect()) {
    var result = conn.query("SELECT ...");  // Blocking? JVM to vyřeší.
    return processResult(result);
}
```

**Žádný callback hell, žádný reaktivní kód** — stejná jednoduchost jako thread-per-request, ale škálování jako event-based.

**Kdy virtual threads nestačí:** CPU-bound workload (virtual threads neřeší paralelismus, jen I/O-bound čekání). Pinning (synchronized bloky "pinují" virtual thread k carrier threadu, blokují ostatní).

> Ke státnicím: OSI model — vrstvy 4 (TCP/UDP) a 7 (HTTP) jsou klíčové. C10K — thread-per-request selhává kvůli paměti a context switch overhead. epoll = O(1) event notification (Linux). Java NIO = Selector + Channel + ByteBuffer (direct buffer pro nulovou kopii). Thread pool = kompromis. Virtual threads (Java 21) = miliony lehkých threadů, JVM automaticky park/unpark při I/O blokování → blocking kód + event-based škálování.
