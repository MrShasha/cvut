> [!info]- Rozdíl mezi JVM a .NET CLR
> ### **1. Způsob startu (Boj o první milisekundu)**
> 
> - **.NET (Kompilace od nuly):** Když aplikaci spustíš, .NET **neinterpretuje**. První spuštění metody okamžitě spustí JIT kompilátor, který vygeneruje základní strojový kód (Tier 0).
>     
> - **Java (Interpretace):** Když aplikaci spustíš, JVM rovnou čte Bytecode a **interpretuje** ho. Ke kompilaci na strojový kód přistoupí až mnohem později.
>     
> - **Co to znamená v praxi (Cold Start):** .NET aplikace nastartuje a odpoví na první HTTP request mnohem rychleji než Java. Java má brutální _Cold Start_ (zahřívací fázi), což ji v moderním cloudu (Serverless/AWS Lambda) dlouho diskvalifikovalo.
>     
> 
> ### **2. Chování při extrémní zátěži (Peak Throughput)**
> 
> - **.NET (RyuJIT Tier 1):** Kompilátor udělá skvělou práci s inliningem a rozbalením smyček na základě nasbíraných dat z provozu. Je ale spíše konzervativní – generuje stabilní a předvídatelný strojový kód.
>     
> - **Java (C2 Compiler):** Je to agresivní "spekulant". Vsadí vše na to, že data budou vypadat tak, jak vypadala doteď. Kód na procesoru maximálně ořeže a nadrátuje natvrdo. Pokud se splete, musí zablokovat vlákno a kód zahodit (Deoptimalizace).
>     
> - **Co to znamená v praxi:** Pokud máš monolit nebo enterprise backend, který běží bez restartu 6 měsíců, Java (díky C2 kompilátoru) dokáže po pár dnech "vymačkat" z křemíku o něco vyšší surový výkon než .NET.
>     
> 
> ### **3. Práce s CPU Cache (Největší inženýrský rozdíl)**
> 
> Toto je kritický rozdíl pro tvé Bentleyho pravidla a optimalizaci pro L1 Cache.
> 
> - **.NET (Podpora Value Types):** Od prvního dne má .NET klíčové slovo `struct`. Můžeš vytvořit pole 10 000 struktur a paměťový alokátor je poskládá **těsně za sebe do jednoho bloku RAM**. Procesor je načte do L1 Cache naráz. Nulový GC overhead, maximální propustnost.
>     
> - **Java (Zatím všechno je reference):** V Javě primitivní datové typy nejdou takto snadno sdružovat (kromě polí základních čísel). Pokud máš v Javě pole 10 000 objektů, je to ve skutečnosti **pole 10 000 pointerů**, které ukazují na náhodná místa na haldě (Heap). Při iteraci procesor neustále skáče v paměti a maže L1 Cache (Cache Misses). Zvedá to tlak na Garbage Collector. (Java tohle řeší už roky v rámci chystaného _Project Valhalla_, ale .NET to má v základu desítky let).
>     
> 
> ### **Celkový obrázek (Big Picture)**
> 
> Obě platformy zkopírovaly to nejlepší jedna od druhé.
> 
> - Java byla dlouho králem masivních, dlouho běžících enterprise serverů, kde nezáleží na paměti a startu, ale na finální propustnosti.
>     
> - .NET byl navržen více "blízko křemíku" (díky podpoře pointerů v `unsafe` kontextu a `struct`), což z něj dělá lepší volbu pro systémy, kde záleží na nízké latenci, práci s pamětí a herních enginech (Unity).
>     
>

# Virtuální stroj a JVM paměťové rozložení

## Proč virtuální stroj?

**VM model** (.NET CLR, JVM) = zdrojový kód se zkompiluje do platformově nezávislého bytecode → bytecode běží v hybridním execution modelu (interpretovaně nebo jako nativní kód po JIT kompilaci).

**Výhody:**
- Platformová nezávislost (RISC/CISC, 32/64-bit, Windows/Linux).
- Reflection (introspekce a modifikace za runtime).
- Automatická správa paměti (GC).
- Bezpečnostní sandbox.

**Nevýhody:**
- Pomalejší start (interpretace + JIT kompilační latence).
- Méně kontroly nad low-level optimalizacemi.
- GC pauzy.

## JVM paměťové rozložení

JVM paměť je rozdělena na dvě kategorie:

### Thread-specific (per-thread):
- **Program Counter (PC)** — adresa aktuálně vykonávané bytecode instrukce.
- **JVM Stack** — zásobník stackových rámců (frames). Každé volání metody = nový frame.
- **Native Method Stack** — pro JNI (Java Native Interface) volání. Tohle jsou metody, které jsou napsané v nativním kódu C/C++, ale můžeš je z Javy zavolat. Tyhle metody nemůže JIT optimalizovat.

### Sdílené přes všechna vlákna:
- **Heap** — kde jsou alokovány všechny objekty. Sdíleno, spravováno GC.
- **Method Area (Metaspace v Java 8+)** — metadata tříd (bytecode, konstantní pool, statická pole).

## Stack frame — anatomie volání metody

Každý frame obsahuje:
- **Local variable array** — pole lokálních proměnných (index 0 = `this` pro instance metody).
- **Operand stack** — zásobník pro výpočty (JVM je stack-oriented machine).
- **Constant pool reference** — odkaz do konstantního poolu třídy.
- **Return address** — kam se vrátit po dokončení metody.

## Ordinary Object Pointer (OOP) a Compressed OOP

**OOP** (Ordinary Object Pointer) = pointer na JVM objekt v heap. Každý objekt má **header**:
- **Mark word** (8 B) — GC info, hash code, lock stav (thin/fat/biased locking).
- **Class reference** (4 nebo 8 B) — pointer na metadata třídy.

**Compressed OOP** (`-XX:+UseCompressedOops`, default pro heap < 32 GB): místo 64-bit pointeru se ukládá 32-bit offset (snižuje paměťový overhead, zlepšuje cache hit rate).

**Lidský překlad:** Každý Java objekt je větší než datová třída v C — má 8–16 B overhead jen na header. Integer jako objekt zabírá 16 B místo 4 B. Proto je `List<Integer>` paměťově neefektivní.

# JVM Bytecode

## Stack-oriented model výpočtu

JVM je **zásobníkový virtuální stroj** (stack-oriented) — na rozdíl od registrového modelu (x86, ARM). Instrukce pracují s operandovým zásobníkem.

**Příklad:** `a + b`

```
iload_1    // push a na zásobník
iload_2    // push b na zásobník
iadd       // pop a a b, push (a+b)
istore_3   // pop výsledek, ulož do lokální proměnné
```

**Proč zásobníkový model?** Jednodušší bytecode generování (bez nutnosti přidělování registrů). JIT pak přeloží zásobníkové instrukce na registrové instrukce pro daný CPU.

## Klíčové bytecode instrukce

| Skupina | Instrukce | Popis |
|---|---|---|
| Load/Store | `iload`, `istore`, `aload`, `astore` | Přesun dat zásobník ↔ lokální proměnné |
| Aritmetika | `iadd`, `isub`, `imul`, `idiv` | Celočíselná aritmetika |
| Konverze | `i2l`, `l2i`, `i2d` | Typové konverze |
| Řízení toku | `ifeq`, `goto`, `tableswitch` | Podmíněné a nepodmíněné skoky |
| Volání | `invokevirtual`, `invokespecial`, `invokestatic`, `invokeinterface` | Volání metod |
| Objekty | `new`, `getfield`, `putfield`, `checkcast` | Práce s objekty |

## Disassembler a decompiler

- **`javap -c MyClass.class`** — zobrazí bytecode (disassembluje).
- **`javap -verbose`** — zobrazí konstantní pool, atributy.
- **Decompiler** (IntelliJ, CFR, Procyon) — zpětně převede bytecode na čitelný Java kód.

**Kdy je to užitečné:** Pochopení, co překladač skutečně vygeneroval (anonymní třídy, lambdy, try-with-resources — všechno má překvapivý bytecode overhead).

# Just-in-Time kompilace a tiered compilation

## Hybridní execution model

JVM neinterpretuje bytecode celý čas — adaptivně přechází mezi úrovněmi:

```
Úroveň 0: Interpretace
  → pomalé, ale okamžitý start
  ↓ (po ~1 000 invokacích)
Úroveň 1–3: C1 kompilátor (client compiler)
  → rychlá kompilace, základní optimalizace
  → vloží profilovací kód (Counters, MDO)
  ↓ (po ~10 000–15 000 invokacích)
Úroveň 4: C2 kompilátor (server compiler)
  → agresivní optimalizace (inlining, escape analysis, loop vectorization)
  → používá profil z C1 fáze
```

**Tiered compilation** = kombinace C1 a C2 pro optimální trade-off mezi startup latencí a peak throughput.

## C2 optimalizace — co JIT dělá za tebe

- **Inlining** — nejdůležitější optimalizace. Bez inliningu není možná žádná jiná.
- **Escape analysis** → objekty, které "neuniknou" z metody, jsou alokovány na stacku (bez GC pressure).
- **Loop unrolling a vectorization** — SIMD instrukce pro numerické výpočty.
- **Devirtualization** — pokud profil ukazuje, že virtual call vždy volá stejnou implementaci → přímý call.
- **Null check elimination** — pokud je garantováno, že pointer není null.

## On-Stack Replacement (OSR)

**OSR** = nahrazení interpretovaného kódu JIT kódem **za běhu metody** (uprostřed smyčky). Kritické pro long-running metody s jednou velkou smyčkou.

**Jak funguje:** JVM detekuje, že metoda je "horká" (backward branch counter), zkompiluje ji, a pak přepne na JIT kód v bezpečném místě (safepoint ve smyčce).

## Deoptimizace

C2 dělá **spekulativní optimalizace** (např. devirtualization). Pokud se předpoklad ukáže jako neplatný za runtime → **deoptimizace** = návrat na interpretovaný nebo C1 kód.

**Příklad:** JIT předpokládal, že polymorfní volání vždy míří na `ArrayList.add`. Pak přijde `LinkedList.add` → deoptimizace.

# Safepoints

## Co je safepoint

**Safepoint** = bod v kódu, kde je garantováno, že JVM zná **konzistentní stav všech vláken** (všechna vlákna jsou pozastavena nebo v bezpečném stavu).

**Proč safepoints potřebujeme:**
- GC potřebuje stopnout všechna vlákna pro přesné vyhledání live objektů (stop-the-world).
- JIT deoptimizace.
- Thread stack inspection (jmap, jstack).
- Biased locking revocation.

## Globální vs. lokální safepoint

- **Globální safepoint** — všechna vlákna musí dojít do safepoint. GC stop-the-world pause.
- **Lokální safepoint** (Java 10+) — pozastaví jen jedno konkrétní vlákno.

## Time to Safepoint (TTSP)

**TTSP** = čas, který uplyne od požadavku na safepoint do doby, kdy všechna vlákna dorazí do safepoint.

**Problém:** Vlákno nemůže být přerušeno kdekoliv — musí dojít do bezpečného bodu (smyčka, volání metody). Pokud vlákno běží long-running nativní kód nebo počítá bez bezpečných bodů → velký TTSP → GC pauza je delší.

**Jak detekovat:** `-Xlog:safepoint` — logování safepoint událostí a jejich trvání.

# CPU a paměťový profiling v JVM

## Sampling vs. tracing

| | Sampling | Tracing |
|---|---|---|
| Princip | Periodicky pozastaví vlákna → zaznamená call stack | Vloží kód do každé metody |
| Overhead | Nízký (1–5 %) | Vysoký (10–100×) |
| Přesnost | Statistická | Přesná |
| Vhodné pro | Nalezení horkých míst | Přesné měření konkrétní metody |
| Nástroje | async-profiler, Java Flight Recorder | JMH, JVM TI agenti |

## Warm-up fáze — proč musíš čekat

Před měřením výkonu Java kódu musíš nechat JVM "zahřát":
1. Prvních N iterací běží pod interpretací → pomalé.
2. C1 zkompiluje kód → rychlejší.
3. C2 zkompiluje kód po dostatku profil dat → maximum výkonu.

**Pokud neměříš až po warm-up, měříš výkon interpretéru, ne JIT.**

**JMH (Java Microbenchmark Harness)** automaticky řeší warm-up, opakování a statistiku. Je to standardní nástroj pro Java mikrobenchmarky.

## async-profiler — low-overhead profiling

`async-profiler` používá `AsyncGetCallTrace` API pro sampling bez nutnosti safepoint → nízký overhead, přesné výsledky.

```bash
./profiler.sh -d 30 -f flamegraph.html <PID>
```

Výsledek = flamegraph → vizualizace, kde CPU tráví čas.

> Ke státnicím: JVM = stack-oriented bytecode + hybridní execution (interpret → C1 → C2). OOP = object header (mark word + klass ref) + compressed OOP pro < 32 GB heap. Frame = local variables + operand stack + constant pool ref. JIT: inlining, escape analysis, devirtualization, loop vectorization, OSR. Safepoint = bezpečný bod pro GC, deoptimizaci, stack inspection. TTSP = čas čekání na safepoint = GC pause. Profiling: sampling (async-profiler) pro nalezení horkých míst.
