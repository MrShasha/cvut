Paměť v JVM je komplexnější, než se zdá. Každý "jednoduchý" Java objekt nese hidden overhead. Špatná volba datové struktury může způsobit 5–10× větší paměťový footprint a dramaticky zhoršit GC chování (GC pressure). Tato kapitola tě naučí měřit skutečnou paměťovou spotřebu, vybírat efektivní datové struktury a porozumět, co autoboxing dělá s výkonem.

# Statická a dynamická analýza paměti

## Dva typy analýzy

**Statická analýza paměti:**
- Analyzuje využití paměti v daném okamžiku (snapshot).
- Vhodné pro: analýza efektivity datových struktur, inspekce obsahu paměti.
- Nástroje: `jmap -histo:live <PID>`, heap dump + MAT (Memory Analyzer Tool).

**Dynamická analýza paměti:**
- Analyzuje chování paměti v průběhu času (alokace, dealokace, živost).
- Vhodné pro: nalezení memory leaků, analýza alokačního vzoru, GC pressure.
- Nástroje: Java Flight Recorder (JFR), async-profiler (alokační profil), Eclipse MAT.

## Object histogram — rychlý přehled

```bash
jmap -histo:live <PID>
```

Výstup:
```
 num     #instances         #bytes  class name
   1:       1234567      987654321  [B        (byte arrays)
   2:        234567       18765432  java.lang.String
   3:         12345        1975200  java.util.HashMap$Node
```

**Upozornění:** `-histo:live` způsobuje **globální GC** (stop-the-world) pro přesné výsledky. Nepoužívej na produkci bez vědomí dopadu.

## Memory leak v Javě

**Logický memory leak** = objekt je dosažitelný (GC ho neuvolní), ale aplikace ho již nepoužívá.

**Typické příčiny:**
- **Static kolekce** — `static List<User> cache` roste donekonečna.
- **Event listenery** — zaregistrován, ale nikdy odregistrován → objekt zůstane živý.
- **ThreadLocal proměnné** — v thread poolech přežívají requesty.
- **Vnitřní třídy** — drží referenci na vnější třídu (anonymous listeners, inner classes).

**Detekce:** Porovnání heap dumpu před a po podezřelé operaci v Eclipse MAT → nalezení objektů, které by měly být uvolněny, ale nejsou.

# Shallow vs. retained size

## Shallow size — vlastní paměť objektu

**Shallow size** = paměť, kterou objekt zabírá přímo — pouze hodnoty svých polí (reference jsou jen pointery, 4–8 B).

```java
class Node {
    int value;        // 4 B
    Node next;        // 4 B (compressed OOP pointer)
    // + object header: ~16 B
}
// Shallow size Node: ~24 B
```

**Kdy použít:** Měření overhead datové struktury samotné (bez dat, která drží).

## Retained size — celková paměť dosažitelná z objektu

**Retained size** = paměť, která by byla uvolněna, pokud by byl objekt smazán — tj. vlastní paměť + paměť všech objektů dosažitelných výhradně přes tento objekt.

```java
class Cache {
    HashMap<String, byte[]> data = new HashMap<>();
    // Retained size = Cache (16B) + HashMap (~64B) + všechny Node objekty + všechny String klíče + všechna byte[] pole
    // Může být GB!
}
```

**Kdy použít:** Identifikace skutečných paměťových "žroutů" — retained size ukazuje, kolik paměti uvolníš, pokud objekt odstraníš.

**Pravidlo:** Pro nalezení memory leaků hledáš objekty s velkým retained size, které by podle logiky aplikace měly být uvolněny.

# Java primitiva, objekty a autoboxing

## Paměťový overhead objektů

```
int        → 4 B (stack nebo v poli)
Integer    → 16 B (object header 12B + int 4B) → 4× overhead!

long       → 8 B
Long       → 24 B → 3× overhead

boolean    → 1 B (nebo 4 B v JVM závisí na implementaci)
Boolean    → 16 B → 16× overhead
```

**Header každého objektu:**
- **Mark word** (8 B): GC info, hash, locking state.
- **Class pointer** (4 B s compressed OOP, 8 B bez): odkaz na metadata třídy.
- **Padding**: zarovnání na 8 bajtů.

## Autoboxing — skrytý overhead

**Autoboxing** = automatický převod primitiv na boxed typy (a zpět) kompilátorem:

```java
List<Integer> list = new ArrayList<>();
list.add(42);   // Autoboxing: 42 → Integer.valueOf(42) → nový objekt na heap!

int sum = 0;
for (int x : list) {  // Unboxing v každé iteraci: Integer → int
    sum += x;
}
```

**Problémy autoboxingu:**
1. **Paměť:** Každý `Integer` = 16 B místo 4 B.
2. **GC pressure:** Tisíce krátkodobých `Integer` objektů → časté GC.
3. **Cache miss:** `ArrayList<Integer>` = pole pointerů na Integer objekty rozptýlené po heap (vs. `int[]` = kontinuální array).

**Příklad katastrofy:**
```java
// Špatně (10 000 Integer objektů na heap):
List<Integer> ids = new ArrayList<>();
for (int i = 0; i < 10000; i++) ids.add(i);

// Lépe:
int[] ids = new int[10000];
```

# Performance-oriented kolekce

## Proč standardní Java kolekce nejsou vždy optimální

`java.util.HashMap<Integer, Integer>`:
- Klíče: `Integer[]` (pole pointerů na Integer objekty).
- Hodnoty: `Integer[]` (pole pointerů na Integer objekty).
- Každý Integer = 16 B + pointer 4 B = 20 B overhead na **každý** pár klíč-hodnota.
- Pro milion párů: ~20 MB jen na boxing overhead.

## Type-specific kolekce — primitivní typy bez boxingu

Knihovny jako **Eclipse Collections**, **Trove**, **Koloboke** nabízí kolekce pro primitivní typy:

```java
// Eclipse Collections:
IntList list = IntLists.mutable.of(1, 2, 3);  // int[], ne Integer[]
IntIntHashMap map = new IntIntHashMap();       // int[] keys + int[] values

// Výhoda: žádný boxing, kontinuální paměť, lepší cache locality
```

## Open addressing hashing — efektivnější než chaining

**Chaining (standardní Java HashMap):**
- Každý bucket = linked list nebo TreeNode → pointer na každý prvek → cache miss.

**Open addressing:**
- Všechna data uložena přímo v hash tabulce (array) → žádné pointery, cache-friendly.
- Při kolizi: skoč na další slot (linear probing, quadratic probing).

```
Array: [_ | K1:V1 | _ | K2:V2 | K3:V3 | _ | ...]
                             ↑kolize K2, K3 → linear probing
```

## Collision resolution schémata

| Schéma | Popis | Výhody | Nevýhody |
|---|---|---|---|
| **Linear probing** | Kolize → přejdi na `(hash+1)%n` | Cache-friendly | Primary clustering |
| **Quadratic probing** | Kolize → `(hash+i²)%n` | Méně clustering | Méně cache-friendly |
| **Double hashing** | Kolize → `(hash+i*hash2)%n` | Minimum clustering | Komplex, 2 hash funkce |
| **Robin Hood hashing** | Bogatý ustupuje chudému | Lepší variance vyhledávání | Složitější insert/delete |

## Load factor a výkon hash tabulky

**Load factor** = `počet prvků / kapacita tabulky`.

- Nízký LF (< 0.5): málo kolizí, ale plýtvání pamětí.
- Vysoký LF (> 0.75): více kolizí → degradace výkonu.

Java `HashMap` defaultně resize při LF = 0.75.

# Resizing strategie kolekcí

## Amortizovaná složitost

Při přidávání prvku do `ArrayList` nebo `HashMap` může dojít k **resize** — vytvoření nového pole a zkopírování všech prvků. Toto je O(n) operace.

Ale **amortizovaně** je každé přidání O(1):
- Pokud kapacitu zdvojujeme, celkový počet kopírovaných prvků je O(n) pro n přidání.
- Amortizovaně: O(n)/n = O(1) na přidání.

**Fyzikální analogie:** Výstavba dálnice. Každý rok zdvojen ná kapacita → první rok 1 km, druhý 2 km, třetí 4 km. Celkem za 10 let 1023 km. Průměrně na rok: 102 km. Přestože v 10. roce bylo nutné postavit 512 km najednou.

## Growth factor — vliv na výkon a paměť

**Growth factor** = koeficient, o který se kapacita zvětší při resize.

| Growth factor | Počet resize (pro 1M prvků) | Průměrná plnost po resize | Paměťový overhead |
|---|---|---|---|
| 1.5 | ~34 | ~67 % | Nižší |
| 2.0 (Java, C++) | ~20 | ~50 % | Střední |
| 4.0 | ~10 | ~25 % | Vysoký |

**Java `ArrayList`:** growth factor = 1.5 (od Java 6). `HashMap`: growth factor = 2.

## Kdy resize bolí

Resize je problematický, když:
1. **Přesně v kritický moment** (latency spike) — resize způsobuje O(n) kopírování.
2. **GC pressure** — starý array je garbage, nový array alokován → GC.
3. **Neznáme dopředu velikost** — zbytečné opakované resize.

**Řešení:** Pokud znáš dopředu počet prvků, inicializuj kolekci s odpovídající kapacitou:
```java
List<Item> list = new ArrayList<>(expectedSize);
Map<K, V> map = new HashMap<>((int)(expectedSize / 0.75) + 1);  // Kapacita pro LF=0.75
```

> Ke státnicím: Statická analýza = snapshot (jmap -histo). Dynamická = alokační profil v čase. Shallow = jen vlastní pole. Retained = celý dosažitelný subgraf. Autoboxing = skrytý boxing (int→Integer = 16B) → GC pressure + cache miss. Type-specific kolekce pro primitiva. Open addressing > chaining pro cache locality. Load factor 0.75 je kompromis kolize vs. paměť. Growth factor 2 → amortizovaně O(1) insert.
