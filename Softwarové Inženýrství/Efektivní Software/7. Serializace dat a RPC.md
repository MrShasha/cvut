Serializace je technicky triviální — převádíš datovou strukturu na bajty a zpět. Ale v produkčním systému, kde serializuješ miliony zpráv za sekundu, rozhoduje volba formátu o tom, zda se vmestíš do latency budget nebo zda přidáváš servery. Tato kapitola ti ukáže, co se skutečně děje při serializaci a proč JSON v high-throughput systému způsobí technický dluh.

# Úvod do serializace

## Co je serializace a kde se děje

**Serializace** = převod datové struktury v paměti (objekty, struktury) na bajty (serializovaná forma). **Deserializace** = opak.

Serializace je potřeba vždy, když data opouštějí paměťový prostor procesu:
- **Síťová komunikace** (REST API, gRPC, message broker).
- **Perzistence** (ukládání do souboru, databáze).
- **Inter-process communication** (IPC, shared memory formát).
- **Remote Procedure Calls (RPC)** — volání funkce na jiném stroji/procesu.

## Kde serializace sedí v zásobníku

```
Tvoje aplikace (objekty v paměti)
    ↓ Serializace (knihovna)
Bajty (síťový/souborový formát)
    ↓ Transport (TCP, UDP, soubor)
Bajty na druhém konci
    ↓ Deserializace (knihovna)
Objekty v paměti druhé aplikace
```

**Fyzikální analogie:** Pošta. Máš věc (objekt). Musíš ji zabalit (serializovat) do krabice (bajty), odeslat, druhá strana vybalí (deserializuje). Kvalita balení rozhoduje o rychlosti, spolehlivosti a bezpečnosti.

## Klíčové metriky serializace

- **Rychlost serializace** — čas převodu objekt → bajty.
- **Rychlost deserializace** — čas převodu bajty → objekt.
- **Velikost zprávy** — méně bajtů = nižší síťový overhead.
- **Schema evoluce** — jak přidáváš nová pole bez rozbití starých klientů.
- **Čitelnost pro lidi** — debug + monitoring.

# XML a JSON — pomalé, ale čitelné

## XML — textový formát s obrovským overhead

```xml
<person>
  <name>Jan Novák</name>
  <age>30</age>
</person>
```

**Problémy XML:**
- **Verbose** — tagy jsou zopakované, data tvoří malý zlomek celkové velikosti.
- **Parsování je pomalé** — musíš číst bajt po bajtu, udržovat stav parseru, budovat DOM.
- **Žádné schema** v základu → validace přes XSD = další overhead.

## JSON — lepší než XML, stále pomalé

```json
{"name": "Jan Novák", "age": 30}
```

**Problémy JSON:**
- **Čísla jako text** — `30` je string "30" → parsing na integer = extra operace.
- **Žádné typy** — `{"timestamp": "2024-01-01"}` — je to string nebo datum?
- **Parsování je stále O(n) pomalé** — každý bajt musí být zkontrolován.
- **Velikost** — field names se opakují v každé zprávě.

**Kdy JSON dává smysl:** REST API pro lidi/browsers, konfigurace, jednoduché data. Nikdy pro high-throughput machine-to-machine komunikaci.

## Výkonnostní čísla (orientační)

| Formát | Serializace | Deserializace | Velikost zprávy |
|---|---|---|---|
| JSON | ~1× (baseline) | ~1× | ~1× |
| Protobuf | ~5–10× rychlejší | ~5–10× rychlejší | ~3–10× menší |
| Cap'n'proto | ~20× rychlejší | ~100× rychlejší* | ~2× menší |

*Cap'n'proto: zero-copy = deserializace je v podstatě pointer cast.

# Protocol Buffers, Cap'n'proto a Apache Avro

## Protocol Buffers (protobuf) — Google standard

**Protobuf** = binární serializační formát od Google. Používá schema definici (`.proto` soubor), ze které generuje kód.

```protobuf
message Person {
  string name = 1;
  int32 age = 2;
}
```

**Jak funguje:**
- Každé pole má číselný tag (1, 2...) — v binárním formátu se posílá tag + délka + data.
- Pole jména jsou **odstraněna** — místo `"name"` se posílá jen `1`.
- Čísla jsou zakódována ve variable-length encoding (varint) — malá čísla mají méně bajtů.

**Schema evoluce:**
- Přidávání nových polí je zpětně kompatibilní (staré klienty neznámá pole ignorují).
- Přejmenování pole nevadí (tagy jsou čísla, ne jména).

**Kde se používá:** gRPC (standard pro Kubernetes, Google, Cloudflare), interní komunikace ve velkých firmách.

## Cap'n'proto — zero-copy serializace

**Cap'n'proto** jde ještě dál — data jsou uložena přímo ve formátu, který je čitelný bez parsování. **Deserializace = pointer cast**, ne kopírování dat.

```cpp
// Deserializace je v podstatě:
auto reader = capnp::FlatArrayMessageReader(data);
auto person = reader.getRoot<Person>();
auto name = person.getName(); // Přímý pointer do raw dat — žádná kopie!
```

**Kdy Cap'n'proto použít:** Extrémně latency-sensitive systémy (trading, real-time systémy, IPC v rámci jednoho stroje).

**Nevýhody:** Složitější API, méně ekosystém než protobuf.

## Apache Avro — schema v datech

**Avro** posílá schema spolu s daty (nebo odkazuje na schema v registru). Vhodné pro:
- **Data streaming** (Apache Kafka — Avro je de facto standard).
- **Big Data** (Hadoop, Spark, Hive).
- **Schema registry** — centrální úložiště schemata, versionování.

**Rozdíl od protobuf:** Avro nevyžaduje generování kódu — schema je dynamicky čitelné za runtime.

## Srovnání formátů

| Formát | Schema | Generuje kód | Zero-copy | Vhodné pro |
|---|---|---|---|---|
| **JSON** | Volitelné (JSONSchema) | Ne | Ne | REST API, config |
| **Protobuf** | Ano (.proto) | Ano | Ne | gRPC, internal API |
| **Cap'n'proto** | Ano (.capnp) | Ano | Ano | IPC, ultra-low latency |
| **Avro** | Ano (JSON schema) | Volitelné | Ne | Kafka, Big Data |
| **XML** | XSD | Ne | Ne | Enterprise legacy |

# Bezpečnost a RPC

## Deserialization attacks

Deserializace nedůvěryhodných dat je jedním z nejnebezpečnějších vektorů útoku (**OWASP A8**).

**Příklad:** Java native serialization (`ObjectInputStream`) → pokud deserializuješ objekt z nedůvěryhodného zdroje, útočník může podstrčit gadget chain, která vykoná libovolný kód.

**Pravidla:**
- **Nikdy nedeserializuj nedůvěryhodná data nativní serializací** (Java, Python pickle, .NET BinaryFormatter).
- Preferuj formáty s explicitním schema (protobuf, Avro) — nelze podstrčit arbitrární typy.
- Validuj velikost zprávy před deserializací (ZIP bomb, giant allocations).

## CORBA — historický kontext RPC

**CORBA** (Common Object Request Broker Architecture) je starý standard pro distribuované objekty. Zmínka v ESW protože ukazuje, proč byly standardní RPC frameworky příliš komplexní a proč moderní alternativy (gRPC, REST) vyhrály.

**Problémy CORBA:**
- Extrémně složitá specifikace → buggy implementace.
- IDL (Interface Definition Language) → vendor lock-in.
- Horší výkon než moderní alternativy.

**Moderní alternativy:**
- **gRPC** (protobuf + HTTP/2) — google standard, polyglot, binary.
- **REST** (JSON + HTTP) — jednoduché, human-readable, wide ecosystem.
- **GraphQL** — flexible queries, ale overhead parsování.

> Ke státnicím: Serializace = objekt → bajty. JSON/XML = pomalé (textové parsování), vhodné pro lidi. Protobuf = binární, tagged fields, schema evoluce, gRPC standard. Cap'n'proto = zero-copy deserializace. Avro = Kafka standard, schema registry. Deserializace nedůvěryhodných dat = kritická zranitelnost (OWASP A8) → nikdy native Java serialization.
