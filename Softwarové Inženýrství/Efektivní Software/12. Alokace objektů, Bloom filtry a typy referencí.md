# JVM alokace objektů a TLABs

## Alokace na heap — bump-the-pointer

V klasickém alokátoru (C `malloc`) je hledání volného místa složité — freelist, binning, fragmentation management. JVM heap alokace je radikálně jednodušší díky **bump-the-pointer**:

```
[----využito----][volné místo                    ]
                 ↑ pointer

new Object() →
[----využito----][Obj][volné místo               ]
                      ↑ pointer posunut o velikost Obj
```

**Proč to JVM může dělat:** GC pravidelně kompaktuje heap — přesouvá živé objekty na začátek, čímž vytváří velký kontinuální volný blok. Žádná fragmentace → bump-the-pointer je vždy validní.

**Výhoda:** Alokace = pouhé posunutí pointeru. Extrémně rychlé (~20 ns).

## TLABs — Thread-Local Allocation Buffers

**Problém:** Mnoho vláken alokuje souběžně → potřeba synchronizace při manipulaci s heap pointerem.

**Řešení — TLAB:** Každé vlákno dostane svůj privátní kus Young generation (Eden space). Vlákno alokuje do svého TLAB bez synchronizace (bump-the-pointer jen ve svém TLAB).

```
Thread 1 → TLAB 1: [--využito--][volné   ]
Thread 2 → TLAB 2: [---využito---][volné ]
Thread 3 → TLAB 3: [volné              ]
```

- TLAB je plný → vlákno žádá JVM o nový TLAB (synchronizace jen při refill, ne při každé alokaci).
- Velikost TLAB: defaultně adaptivní (JVM profil alokačního chování vlákna).

**Výsledek:** Alokace v JVM je rychlá jako stack alokace v C (žádný lock).

## Velké objekty — Humongous allocation

Objekty větší než ~50 % TLAB (nebo 50 % G1 region) jsou alokovány přímo do **Old generation** (nebo v G1 do speciálních Humongous regionů).

**Proč to je problém:**
- Humongous objekty nejsou kolekovány v Young GC → zůstávají až do Full GC.
- Způsobují fragmentaci Old generation.
- Velké byte[] (JSON parsing, buffers) jsou typičtí pachatelé.

# Escape analysis a skalární náhrada

## Co je escape analysis

**Escape analysis** = JIT optimalizace, kdy JVM analyzuje, zda objekt "unikne" ze svého kontextu (metody nebo vlákna):

```java
void process(int x) {
    Point p = new Point(x, x * 2);  // Alokuje se na heap?
    return p.x + p.y;               // p nikde neunikne!
}
```

Pokud `p` neunikne (není vrácen, nepředán jiné metodě, není přiřazen do pole/static), JVM ho **nealokuje na heap**.

## Skalární náhrada (scalar replacement)

Objekt, který neunikne, je "rozebrán" na jeho primitivní složky — ty jsou uloženy na stack nebo do registrů:

```java
// Po skalární náhradě se `new Point(x, x*2)` přeloží přibližně na:
void process(int x) {
    int p_x = x;       // Jen stack proměnné, žádný heap objekt!
    int p_y = x * 2;
    return p_x + p_y;
}
```

**Výsledek:** Nulová heap alokace, nulový GC pressure pro short-lived lokální objekty.

**Kdy JIT nestihl EA:** Příliš komplikovaný call graph, virtual dispatch, volání přes reflexi. V těchto případech se stále alokuje na heap.

# Data locality a NUMA

## Cache lokalita objektů

Objekty alokované "najednou" jsou v bump-the-pointer alokátoru fyzicky blízko sebe v paměti → lepší cache lokalita při iteraci.

```java
// Cache-friendly (objekty blízko u sebe po alokaci):
List<Point> points = new ArrayList<>();
for (int i = 0; i < 1000; i++) {
    points.add(new Point(i, i));  // Alokovány postupně, blízko sebe v heap
}

// Ale po GC mohou být přesunuty → lokalita se může zhoršit!
```

## NUMA — Non-Uniform Memory Access

V **multi-socket serverech** (2+ CPU socketů) každý socket má svou RAM. Přístup do "cizí" RAM (přes QPI/UPI interconnect) je 2–3× pomalejší.

```
Socket 0: CPU0-CPU15 + RAM0 (64 GB)
Socket 1: CPU16-CPU31 + RAM1 (64 GB)

CPU0 přistupuje do RAM0: ~100 ns (lokální)
CPU0 přistupuje do RAM1: ~200-300 ns (NUMA remote)
```

**NUMA problém v JVM:**
- JVM heap je standardně alokován na jednom NUMA uzlu.
- Vlákna na jiném socketu přistupují remote → výkon degraduje.

**Řešení:** `-XX:+UseNUMA` příznak JVM → JVM alokuje části heapu na každém NUMA uzlu, TLAB jsou přiřazeny vláknům na základě jejich NUMA affinity.

# Bloom filtry

## Co je Bloom filtr

**Bloom filtr** = pravděpodobnostní datová struktura pro membership testing: "Je prvek X v množině?"

**Odpověď:** Buď **"určitě NE"** nebo **"možná ANO"** (false positive je možný).

**Struktura:**
- Bitové pole délky m (inicializováno na 0).
- k nezávislých hash funkcí.

**Vkládání prvku:**
```
insert(x):
  for i in 1..k:
    bit[hash_i(x) % m] = 1
```

**Dotaz na membership:**
```
contains(x):
  for i in 1..k:
    if bit[hash_i(x) % m] == 0: return FALSE  // Určitě není
  return TRUE  // Možná je (všechny bity nastaveny, ale možná jinými prvky)
```

## False positive — pravděpodobnost a ovládání

**False positive** = Bloom filtr řekne "možná ANO", ale prvek ve skutečnosti není v množině.

Pravděpodobnost FP závisí na:
- **m** (velikost bitového pole) — větší m → méně FP.
- **k** (počet hash funkcí) — optimální k závisí na m/n (n = počet prvků).
- **n** (počet vložených prvků) — více prvků → více nastavených bitů → více FP.

$$P_{FP} \approx \left(1 - e^{-kn/m}\right)^k$$

**Optimální k:** $k = \frac{m}{n} \ln 2 \approx 0.693 \cdot \frac{m}{n}$

**Praktické nastavení:** Cílová FP rate (např. 1 %) + počet prvků n → výpočet potřebného m a k.

## Counting Bloom filtr

**Standardní Bloom filtr:** nelze mazat prvky (setnutý bit nelze "odnastavit" — nevíme, zda ho nenastavil jiný prvek).

**Counting Bloom filtr:** místo bitů jsou čítače. Insert → increment. Delete → decrement. Contains → check > 0.

**Nevýhoda:** 4–8× větší paměťový overhead (čítač místo bitu).

## Kde se Bloom filtry používají

- **Databáze** (Cassandra, HBase, RocksDB): Before disk lookup, check Bloom filter → pokud "určitě NE" → ušetři disk I/O.
- **Webové prohlížeče:** Google Chrome malicious URL check (Bloom filter lokálně, positive → server ověření).
- **Cache systémy:** Zjisti, zda klíč existuje před nákladným výpočtem.

# Typy referencí v Javě

## Motivace — komunikace s GC

Standardní Java reference = **strong reference**. GC objekt nikdy neuvolní, pokud existuje strong reference.

Ale někdy chceme říct GC: "Tohle můžeš uvolnit, pokud potřebuješ paměť" → slabší typy referencí.

## Weak reference — "Pokud GC potřebuje, uvolni to"

**`WeakReference<T>`** — objekt je uvolněn při **první GC**, která ho vidí jako slabě dosažitelný (= žádná strong reference na něj):

```java
WeakReference<BigObject> ref = new WeakReference<>(new BigObject());
// ... v průběhu:
BigObject obj = ref.get();  // Vrátí null, pokud GC objekt uvolnil!
if (obj == null) {
    // Objekt byl uvolněn, vytvoř ho znovu
}
```

**Použití:** `WeakHashMap` — auto-cleanup záznamů, jejichž klíče nemají jinou referenci. Ideální pro caching, kde klíč je "živý" jen pokud ho někdo používá.

## Soft reference — "Uvolni jen pokud opravdu dochází paměť"

**`SoftReference<T>`** — GC objekt uvolní jen pokud **hrozí OutOfMemoryError**:

```java
SoftReference<byte[]> cache = new SoftReference<>(loadExpensiveData());
byte[] data = cache.get();  // null jen při opravdovém memory pressure
if (data == null) {
    data = loadExpensiveData();  // Znovu načti
    cache = new SoftReference<>(data);
}
```

**Použití:** Paměťově citlivé cache — JVM sám rozhodne, kdy cache vyčistit, když dochází paměť.

## Phantom reference — "Řekni mi, až bude objekt uvolněn"

**`PhantomReference<T>`** — `get()` vždy vrátí `null`. Slouží pouze pro notifikaci po finalizaci:

```java
ReferenceQueue<Object> queue = new ReferenceQueue<>();
PhantomReference<Object> ref = new PhantomReference<>(obj, queue);

// Až GC objekt uvolní:
Reference<?> r = queue.poll();  // ref se dostane do fronty
if (r != null) {
    // Proveď cleanup (např. uvolni nativní zdroje)
}
```

**Použití:** Správa nativních zdrojů (Direct buffers, nativní handle) bez `finalize()`.

## Přehled referencí

| Typ | Uvolnění | `get()` po uvolnění | Použití |
|---|---|---|---|
| Strong | Nikdy (pokud existuje) | N/A | Standardní |
| Soft | Při memory pressure | `null` | Paměťové cache |
| Weak | Při první GC | `null` | WeakHashMap, caching po klíči |
| Phantom | Po finalizaci | Vždy `null` | Nativní resource cleanup |

> Ke státnicím: TLABs = per-thread privátní chunk Eden space → lock-free alokace. Bump-the-pointer = alokace = posunutí pointeru (~20 ns). Escape analysis = JIT detekce neunikajících objektů → skalární náhrada (žádná heap alokace!). NUMA = multi-socket servery mají nerovnoměrné přístupy do RAM (-XX:+UseNUMA). Bloom filtr = bitové pole + k hash funkcí, only "definitely not" or "maybe yes". FP rate → ovládáno m a k. Weak reference = uvolnění při GC. Soft reference = uvolnění při memory pressure. Phantom reference = notifikace po uvolnění.
