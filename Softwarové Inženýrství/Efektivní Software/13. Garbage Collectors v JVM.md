Garbage collector je jeden z nejkritičtějších komponent JVM pro produkční výkon. "Stop-the-world" pauzy způsobují latency spikes, nedostatečný throughput GC limituje škálování aplikace. Tato kapitola vysvětluje, jak GC algoritmicky fungují pod kapotou, proč je G1 výchozím GC v moderní Javě a kdy sáhnout po alternativách jako ZGC nebo Shenandoah.

# Základy automatické správy paměti

## GC roots — co je "živé"

**GC root** = reference, od které GC začíná prohledávání dosažitelnosti. Objekt je "živý" (live) = dosažitelný z alespoň jednoho GC root přes libovolně dlouhou cestu referencí.

**Typy GC roots:**
- **Stack proměnné** — lokální proměnné a parametry aktuálně běžících metod.
- **Static pole** — `static` pole tříd.
- **JNI reference** — reference z nativního C/C++ kódu.
- **System classes** — `String`, `Class`, bootstrap classloader.
- **Synchronized monitors** — objekty, na které je aktivní `synchronized`.

**Fyzikální analogie:** Vychozí bod je okraj sítě (GC roots). Algoritmus "táhne" všechny uzle dosažitelné přes řetěz referencí. Co není dosažitelné → vyhozeno.

## Mark phase

```
Mark phase:
  worklist = [all GC roots]
  while worklist not empty:
    obj = worklist.pop()
    if not obj.marked:
      obj.marked = true
      for ref in obj.references:
        worklist.push(ref)

Sweep phase:
  for all objects in heap:
    if not obj.marked:
      free(obj)  // Uvolnit
    else:
      obj.marked = false  // Reset pro příští GC
```

# Generational hypothesis a heap organizace

## Generational hypothesis

**Hypotéza:** Většina objektů zemře mladá (*"die young"*). Analýzy produkčních aplikací potvrzují: 80–98 % objektů se stane garbage do první kolekce.

**Fyzikální analogie:** Papírové lístky na stole — většina je vyhozena ten samý den, jen malá část přežívá měsíce.

**Implikace:** Pokud většina objektů zemře mladá, stačí **kolektovat jen mladé objekty** — to je rychlé. Staré objekty kolektuj jen výjimečně.

## Organizace heap (Generational GC)

```
Heap:
┌─────────────────────────────────────────────────────┐
│ Young Generation (1/3 heapu)    │ Old Generation     │
│ ┌──────────┬────────┬────────┐  │ (2/3 heapu)        │
│ │  Eden    │ Surv0  │ Surv1  │  │                    │
│ └──────────┴────────┴────────┘  │                    │
└─────────────────────────────────────────────────────┘
```

- **Eden:** Nové objekty alokované přes TLAB.
- **Survivor 0 & 1 (From/To):** Objekty, které přežily alespoň jednu Young GC.
- **Old generation (Tenured):** Objekty, které přežily dostatek Young GC (tenuring threshold, defaultně 15 cyklů).

## Young GC (Minor GC)

Spustí se, když Eden je plný. Kolektuje jen Young generation:
1. **Mark** — najdi živé objekty v Eden a aktivní Survivor prostor.
2. **Copy** — zkopíruj živé objekty do druhého Survivor prostoru (nebo do Old gen pokud jsou starší).
3. **Flip** — Survivor prostory se prohodí (From/To).

**Výsledek:** Eden je prázdný, živé objekty v jednom Survivor prostoru.

**Rychlost:** Young GC trvá typicky 1–50 ms (malý region, copying collector).

# Design dimensions GC

## Mark-Sweep

**Fáze:**
1. Mark — označ živé objekty.
2. Sweep — uvolni neoznačené.

**Nevýhoda:** Fragmentace! Uvolněné kusy jsou rozesety po heapu → `malloc` potřebuje freelist → alokace pomalá + nelze alokovat velké kontinuální bloky.

## Mark-Compact

**Fáze:**
1. Mark — označ živé objekty.
2. Compact — přesuň živé objekty na začátek heapu → žádná fragmentace.

**Výhoda:** Žádná fragmentace → bump-the-pointer alokace.
**Nevýhoda:** Compaction = přesouvání objektů → musí se aktualizovat všechny reference → pomalé (2–3× průchod heap).

## Copying collector

**Heap rozdělen na 2 poloostrovy (From/To):**
1. Alokuj do From.
2. GC: zkopíruj živé objekty z From do To.
3. Flip — To se stane From.

**Výhoda:** Žádná fragmentace (objekty jsou zkompaktovány po kopírování). Rychlé (jen živé objekty zpracovány).
**Nevýhoda:** Polovina heap je vždy prázdná → 2× menší efektivní heap.

*Young generation v JVM používá copying collector (Eden + Survivor spaces).*

## Stop-the-world vs. Concurrent

| | Stop-the-world (STW) | Concurrent |
|---|---|---|
| Aplikace běží? | NE — všechna vlákna pozastavena | ANO — GC běží souběžně |
| Jednoduchost | Jednoduchá implementace | Složité (floating garbage, write barriers) |
| Latence | Pauzy viditelné uživateli | Minimální pauzy |
| Overhead | Nižší celkový overhead | Vyšší (write barriers, tri-color marking) |

**Tri-color marking** (concurrent GC): Objekty jsou barveny: bílá (nenavštíveno) → šedá (navštíveno, reference nezpracovány) → černá (navštíveno, reference zpracovány). Write barrier: při změně reference aktualizuj barvu.

# Parallel collector

## Remembered Set

**Problém:** Při Young GC potřebuješ najít všechny reference z Old generation do Young generation (jinak by Ti unikly živé objekty v Eden).

**Problém:** Prohledat celý Old generation při každém Young GC = příliš pomalé.

**Řešení — Remembered Set (RSet):** Při každém zápisu reference z Old do Young (write barrier), přidej zdroj do RSet. Při Young GC stačí prohledat jen RSet (malý subset Old gen).

```
Old gen objekt O zapisuje referenci na Young gen objekt Y:
  → write barrier: RSet.add(O)

Young GC:
  Roots = GC roots + RSet entries
  → správné zachycení živých objektů v Young gen
```

## Parallel Scavenge (PS) — throughput-first

**Algoritmus:**
- Young GC: Copying collector + více vláken (N vláken = N CPU jader).
- Cíl: Maximální throughput (minimální čas strávený v GC jako % celkového času).
- Nevhodné pro: Low-latency aplikace (STW pauzy mohou být stovky ms).

## Parallel Mark-Compact (Old gen)

- Parallelní mark phase (N vláken).
- Parallelní compact phase.
- **Full GC** = Young + Old gen → trvá sekundy pro velké heapy. Katastrofa pro latenci!

**Kdy použit PS:** Batch zpracování, analytické workloady kde latency nezáleží, ale throughput ano.

# Garbage-First (G1) collector

## Region-based heap

G1 odstraňuje pevné dělení na Young/Old/Survivor — místo toho je heap rozdělen na **rovnoměrné regiony** (defaultně 1–32 MB každý):

```
Heap:
┌───┬───┬───┬───┬───┬───┬───┬───┬───┬───┬───┬───┐
│ E │ S │ O │ O │ E │ F │ H │ H │ E │ O │ S │ F │
└───┴───┴───┴───┴───┴───┴───┴───┴───┴───┴───┴───┘
E=Eden, S=Survivor, O=Old, F=Free, H=Humongous
```

Každý region má dynamicky přiřazenou roli (Eden/Survivor/Old/Free). G1 může přiřadit více regionů Young nebo Old dle potřeby.

## Garbage-First princip

G1 si drží statistiku o **množství garbage** v každém Old region. Při Mixed GC vybere regiony s **nejvyšším podílem garbage** (= nejvíce "vydělá" za čas kolekce) → *Garbage-First*.

## GC cykly v G1

### Young-only fáze

- **Young GC (STW):** Evakuace Eden a Survivor regionů do nových Survivor nebo Old regionů. Trvá desítky ms.
- G1 udržuje cílovou dobu pauzy (`-XX:MaxGCPauseMillis=200`) — přizpůsobuje počet Young regionů.

### Space-reclamation fáze (Mixed GC)

Spouští se, když Old generation dosáhne threshold (`-XX:InitiatingHeapOccupancyPercent`, defaultně 45 %):

1. **Initial Mark (STW):** Označ GC roots (rychlé, součást Young GC).
2. **Concurrent Mark:** Souběžně s aplikací označ živé objekty v Old gen (write barrier).
3. **Remark (STW):** Zpracuj nedokončené references (krátká pauza).
4. **Cleanup (STW):** Identifikuj regiony bez živých objektů → okamžitě uvolni.
5. **Mixed GC (STW):** Evakuuj Young + výběr Old regionů s nejvíce garbage.

### Full GC

- Fallback pokud G1 nestíhá uvolňovat paměť → serialní (nebo parallelní) Full GC.
- Stop-the-world, trvá sekundy. Signál, že G1 potřebuje tuning (větší heap, menší alokační rate).

## Humongous objekty

Objekty > 50 % velikosti regionu → alokovány do **Humongous regionů** (sada po sobě jdoucích regionů).

**Problém:** Humongous objekty jsou kolekovány jen v Old gen GC (nebo Full GC). Velké `byte[]` (JSON parsing) mohou způsobit průběžné Full GC.

**Řešení:** Zvětšit G1 region size (`-XX:G1HeapRegionSize`), nebo přepsat kód pro opakované použití bufferů (objektový pool).

## ZGC a Shenandoah — low-latency alternativy

**ZGC (Java 15+ production-ready):**
- Téměř plně concurrent (pauzy < 10 ms i pro TB heap).
- Load barriers místo write barriers → složitější, ale efektivnější concurrent marking.
- Concurrent relocation (přesouvá objekty za běhu aplikace!).

**Shenandoah:**
- Podobné cíle jako ZGC, ale jiná implementace.
- Concurrent compaction → eliminuje STW compaction pauz.

> Ke státnicím: GC roots = stack vars, static fields, JNI, monitors. Generational hypothesis → Young (Eden + 2 Survivor) + Old gen. Young GC = copying collector (rychlé, < 50 ms). Old GC = mark-compact (pomalé). RSet = write barrier záznamy Old→Young referencí (pro efektivní Young GC). G1 = region-based heap, garbage-first princip, cílová doba pauzy. G1 fáze: Young-only → concurrent mark → Mixed GC. Humongous = > 50 % regionu → problém pro GC. ZGC/Shenandoah = sub-10ms pauzy i pro velké heapy.
