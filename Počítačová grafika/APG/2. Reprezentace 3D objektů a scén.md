## Hraniční vs. objemová reprezentace

### Základní dělení

```
Reprezentace 3D objektu
│
├── Hraniční (B-Rep / BBR)  ── popisuje POVRCH objektu
│   ├── Polygonální mesh (trojúhelníky, quad)
│   ├── Parametrické plochy (Bézier, NURBS)
│   ├── Implicitní plochy f(x,y,z) = 0
│   ├── Výšková pole
│   └── Bodová mračna (point clouds)
│
└── Objemová (IBR / volumetrická)  ── popisuje OBJEM objektu
    ├── Voxelová mřížka
    ├── Oktalový strom (Octree)
    ├── CSG strom
    └── Výsledky CT/MRI
```

### Srovnání B-Rep vs. IBR

|Vlastnost|Hraniční (B-Rep)|Objemová (IBR)|
|---|---|---|
|Co popisuje|Povrch (hranici)|Objem (vnitřek)|
|Test "bod uvnitř"|Složitý (ray casting)|Triviální|
|Paměťová náročnost|Nízká|Vysoká|
|GPU renderování|Přímé, jednoduché|Nepřímé (ray casting) nebo konverze|
|Geometrické operace (boolean)|Složité|Přirozené|
|Příklady|Mesh, NURBS|Voxely, Octree, CT data|

### Objemová reprezentace — voxely a Octree

**Voxel** = volumetrický pixel; základní buňka pravidelné 3D mřížky.

```
Pravidelná voxelová mřížka:
  Každá buňka = plná / prázdná (nebo skalární hodnota)
  Adresování: linearizace 3D → 1D (řádkový major, Morton/Z-křivka, ...)
  Paměť: 1000×1000×1000×4 B = 4 GB  ← problém velkých dat
```

**Octree** = adaptivní hierarchická dělení prostoru na 8 oktantů:

- Vnitřní uzel = smíšená buňka (nutno dál dělit)
- Plný list = plně obsazená buňka
- Prázdný list = prázdná buňka

Booleovské operace na octree (sjednocení A∪B, průnik A∩B, rozdíl A\B): složitost O(N_A · log N_B).

### CSG (Constructive Solid Geometry)

CSG strom — binární strom popisující historii tvarování:

```
        [rozdíl -]
       /           \
  [průnik ∩]     [válec]
  /         \
[kvádr]   [koule]

Listy   = primitiva (koule, kvádr, válec, kužel, ... v implicitní formě)
Uzly    = množinové operace (∪ OR, ∩ AND, \ MINUS, ⊕ XOR) nebo transformace
Kořen   = výsledný model
```

**Vlastnosti CSG:**

- Jednoduchý test "je bod uvnitř solidu" (rekurzivní vyhodnocení stromu)
- Modelování připomíná manufacturing (vrtání = MINUS)
- Paměťově kompaktní
- Pro GPU rendering nutná konverze na B-Rep (tessellace, marching cubes) nebo ray casting

### Implicitní plochy

Plocha = nulová množina funkce: `f(x, y, z) = 0`

```
Koule poloměru r:     x² + y² + z² - r² = 0
Rovina:               ax + by + cz + d = 0
Torus:                (√(x²+y²) - R)² + z² - r² = 0
```

**Metaballs** = zaoblené objekty tvořené součtem radiálních bázových funkcí — tání/splývání tvarů přirozenou cestou.

### Mračna bodů (Point Clouds)

Každý bod = (x, y, z) + normálový vektor + barva. Získávají se skenováním (LiDAR, fotogrammetrie). Paměťově náročné; zobrazují se hierarchicky (k určité úrovni LOD).

## Síťová (mesh) reprezentace

**Polygonální síť** = klasická B-Rep; popisuje povrch objektu pomocí vrcholů, hran a stěn.

- Preferují se **trojúhelníky** — vždy planární, HW-friendly, zjednodušují rendering, ořezávání i detekci kolizí
- Alternativa: quad mesh (pro subdivize), N-gony (modelování)

### Proč trojúhelníky?

```
Trojúhelník je vždy:
  ✓ Planární (3 body určují rovinu jednoznačně)
  ✓ Konvexní
  ✓ Zpracovatelný GPU primitivem
  ✗ Špatně aproximuje hladké plochy při malém počtu
```

### Pořadí vrcholů — orientace stěny

Orientace stěny určena pořadím vrcholů při pohledu z vnějšku:

```
CCW (counter-clockwise) = přední strana (front face)
CW  (clockwise)         = zadní strana (back face)

Pohled zepředu:
    v0
   /  \
  v2 - v1    → CCW = normála míří K pozorovateli
```

Toto konvention je kritické pro **back-face culling** (GPU zahazuje CW stěny).

## Geometrická a topologická data

### Geometrická data (u vrcholů)

|Atribut|Typ|Poznámka|
|---|---|---|
|Pozice|`vec3`|Souřadnice ve 3D prostoru|
|Normála|`vec3`|Pro Phong shading (interpolace přes plochu)|
|Tangenta|`vec3`|Pro normal/bump mapping|
|Bitangenta (binormal)|`vec3`|Kolmá na normálu i tangentu|
|UV souřadnice|`vec2`|Texturové mapování|
|Barva (vertex color)|`vec4`|Volitelně|
|Skinning: joint ID + váhy|`ivec4 + vec4`|Pro kosterní animaci|

**Tangentový prostor** — nutný pro normal mapping:

```
Δp1 = p1 - p0,  Δp2 = p2 - p0
Δu1 = u1 - u0,  Δu2 = u2 - u0

[Δp1]   [Δu1  Δv1] [T]
[Δp2] = [Δu2  Δv2] [B]

→ T (tangenta) a B (bitangenta) = ze souřadnic textury sousedních vrcholů
```

### Topologická data

Topologie = **konektivita** — které vrcholy tvoří jaké trojúhelníky, které trojúhelníky spolu sousedí.

Oddělení geometrie od topologie je výhodné:

- Geometrické transformace upravují pouze pozice vrcholů (topologie zůstává)
- GPU zpracovává geometrii a topologii zvlášť

### Typy hran (sémantická klasifikace)

```
Ostré hrany (sharp)    = reálné hranice objektu
Vnitřní hrany          = pomocné (pro tesselaci)
Siluetové hrany        = generovány za renderování (závisí na pohledu)
```

### Typy stěn

```
Planární    vs  Parametrické  (Bézier patch, NURBS)
Jednostranné vs  Oboustranné  (orientace = CCW/CW)
Stěny s dírami: 1 vnější hranice + N vnitřních hranic, spojeny pomocnými hranami
```

## Efektivní kódování meshů

### 1. Prostý seznam trojúhelníků (Triangle Soup / Polygon Soup)

```
Každý trojúhelník = 3 vrcholy uloženy přímo (bez sdílení):

Vertices: [v0,v1,v2, v3,v4,v5, v6,v7,v8, ...]
           ← tri 0 →  ← tri 1 →  ← tri 2 →

Redundance: sdílený vrchol uložen N-krát (typicky 4–6×)
Výhoda:    jednoduchá struktura
Nevýhoda:  paměťové plýtvání, žádná topologická info
```

### 2. Indexovaný seznam trojúhelníků (Indexed Mesh) ← STANDARD

```
Vertex buffer:   [v0, v1, v2, v3, v4, ...]    (každý vrchol 1×)
Index buffer:    [0,1,2, 0,2,3, 1,3,4, ...]   (3 indexy / trojúhelník)

Výhoda:  sdílení vrcholů → méně dat, GPU cache efektivita
         V vrcholech, F plochách → přibližně V + 3F čísel
Nevýhoda: nepřímý přístup (nutno dereferencovat index)
```

### 3. Triangle Strip

```
Sekvence vrcholů, kde každý nový vrchol + 2 předchozí = nový trojúhelník:

Vrcholy:  v0 v1 v2 v3 v4 v5
Tria:     (v0,v1,v2) (v1,v2,v3) (v2,v3,v4) (v3,v4,v5)
          (střídá se orientace → renderer automaticky opravuje)

Výhoda:  N vrcholů = N-2 trojúhelníků (ideál: 1 vertex / trojúhelník)
Nevýhoda: složité generování optimálních stripů pro libovolnou mesh
Nástroj:  NvTriStrip
```

### 4. Triangle Fan

```
Všechny trojúhelníky sdílí 1 centrální vrchol v0:

Vrcholy:  v0 v1 v2 v3 v4
Tria:     (v0,v1,v2) (v0,v2,v3) (v0,v3,v4)

Vznik:    triangulace konvexního polygonu
```

### 5. Kompaktní (bezindexová) reprezentace

```
Pouze pole vertices — vrcholy opakující se sdílejí sousedy implicitně
podle pozice v poli (závisí na konvenci).
Šetří paměť na indexy, ale ztrácí explicitní topologii.
```

### Srovnání kódování

|Metoda|Vrcholů / trojúhelník|Topologie|GPU cache|
|---|---|---|---|
|Prostý seznam|3|✗|špatná|
|Indexovaný seznam|~1 (sdílení)|✗ explicitní|dobrá|
|Triangle Strip|~1|✗|výborná|
|Triangle Fan|~1|✗|výborná|

### Optimalizace meshů

```
1. Výpočet normál vrcholů:
   n_vertex = normalize(Σ n_face_i)  pro sousední plochy
   (mezní úhel: ostrá hrana = oddělit vrcholy)

2. Sdílení vrcholů (welding):
   Sloučit duplicitní vrcholy (stejná pozice, normála, UV)

3. Optimalizace pořadí pro GPU vertex cache:
   Maximalizace cache hit rate → méně vertex shader invokací

4. Komprese (Unity příklad):
   High quality: 20 bit pozice / 8 bit normála / 16 bit UV
   Low quality:  10 bit pozice / 6 bit normála /  8 bit UV
```

## Okřídlená hrana a manifoldy

### Manifold a 2-manifold

**2-manifold** = pro každý povrchový bod existuje okolí topologicky ekvivalentní s rovinou (lze "rozvinout" do roviny).

Praktická definice: každá hrana inciduje **přesně se dvěma** plochami.

```
Non-manifold příklady:
  - Hrana sdílená 4 plochami (nekonečně tenká stěna)
  - Vrchol spojující dvě oddělené části (bowtie)
```

Manifoldy jsou důležité pro CAD/CAM a manufacturing.

### Euler–Poincaré vzorec

Pro 2-manifold platí:

```
V - E + F - R = 2(S - H)

V  = počet vrcholů (vertices)
E  = počet hran (edges)
F  = počet stěn (faces)
R  = počet prstenců (rings = díry ve stěnách)
H  = počet průchozích děr objektem (holes through object)
S  = počet samostatných slupek (shells)

Zjednodušení pro jednoduchý mnohostěn (S=1, H=0, R=0):
  V - E + F = 2   ← Eulerova rovnost

Příklad: Krychle: V=8, E=12, F=6 → 8-12+6 = 2  ✓
```

### Okřídlená hrana (Winged-Edge)

Datová struktura pro explicitní uložení topologické sousednosti:

```
Pro každou hranu ukládáme ("křídla"):
  ┌──────────────────────────────────┐
  │  Hrana e:                        │
  │  - start_vertex, end_vertex      │
  │  - left_face,  right_face        │
  │  - prev_left,  next_left         │  ← hrany v levé ploše
  │  - prev_right, next_right        │  ← hrany v pravé ploše
  └──────────────────────────────────┘

Tři tabulky (DPG varianta):
  Tabulka vrchol → hrana
  Tabulka strana → hrana
  Tabulka hrana → hrana   ← klíčový element
```

**Schopnosti winged-edge:**

- Traversal sítě (procházení sousedních ploch, vrcholů, hran) v O(1)
- Editování mesh a zajišťování konzistence
- **Pouze pro manifoldy** (každá hrana má právě 2 sousední plochy)

**Pro non-manifoldy:** winged half-edge (každá hrana = 2 poloviční hrany s vlastní orientací)

```
Half-edge struktura:
  Každá hrana = 2 orientované poloviční hrany (twin)
  Každá half-edge: next, prev, twin, vertex, face
  Umožňuje: libovolný počet ploch na hranu
```

## Hladké a dělící povrchy

### Parametrické plochy

- **Bézierovy patche** — po částech parametrické, řízeny kontrolními body
- **NURBS** (Non-Uniform Rational B-Splines) — standard v CAD
- **Coons surface, Gordon surface** — interpolace hranic

Před GPU renderingem se tessellují na trojúhelníky (tessellation shaders: hull + domain).

### Dělící (subdivision) plochy

Progresivní zjemnění hrubé mesh — každý krok rozdělí polygony a přemístí vrcholy:

|Schéma|Vstup|Typ|Spojitost|
|---|---|---|---|
|**Loop**|trojúhelníky|aproximační|C² (mimo extra vrcholy)|
|**Catmull-Clark**|quady (libovolné)|aproximační|C²|
|**Doo-Sabin**|libovolné|aproximační|C¹|
|**Butterfly**|trojúhelníky|interpolační|C¹|

**Catmull-Clark** (nejrozšířenější):

```
Pro každou plochu F (N vrcholů):
  Ploškový bod VF = průměr(V1,...,VN)

Pro každou hranu E s plochami F1, F2:
  Hranový bod VE = (E1 + E2 + VF1 + VF2) / 4

Pro každý vrchol V s N okolními plochami:
  Vrcholový bod VV = (1/N)·Σ(VFi) + (2/N)·Σ(VEi) + ((N-3)/N)·V
```

HW podpora: hull shader + tessellation shader (DirectX 11 / OpenGL 4.0).

## Graf scény

### Definice a motivace

**Graf scény** = n-ární strom (nebo DAG) organizující objekty scény:

- Logické/sémantické seskupení
- Kompozice transformací
- Sdílení instancí (instancing)
- Aktivace/deaktivace podstromů
- Částečné prostorové třídění

**Scéna obsahuje:**

- Nezobrazované objekty: kamery, světla
- Zobrazované objekty: geometrie, materiály, textury
- Logická struktura: skupiny, instance
- Transformace: hierarchicky skládané

### Typy uzlů

```
Kořen (Root)
  │
  ├── Transformační uzel [T1]    ← matice 4×4 nebo TRS
  │     ├── Skupinový uzel [G1]  ← logické seskupení
  │     │     ├── Geometrický uzel ← mesh data
  │     │     └── Materiálový uzel ← shader, textury
  │     └── Světelný uzel        ← omezená oblast vlivu
  │
  └── Transformační uzel [T2]
        └── Ukazatel (instance) → sdílená geometrie
```

|Typ uzlu|Obsah|Poznámka|
|---|---|---|
|Geometrický|Mesh, LoD|Geometrická data|
|Transformační|Matice / TRS|Skládají se od kořene k listu|
|Skupinový|Logické seskupení|Dědičnost vlastností|
|Materiálový|Shader, textury|Poslední nalezená hodnota (nedědí se skládáním)|
|Světelný|Světelný zdroj|Platnost omezena podstromem|
|Instanční (ukazatel)|Odkaz na jiný podstrom|DAG místo stromu|

### Dědičnost vlastností

**Transformace** se skládají (násobí matice) od kořene k listu:

```
Pre-order traversal (DFS zleva doprava, shora dolů):

  ┌ Sestup → push transformace na zásobník
  │ Geometrický uzel → aplikuj složenou transformaci = T_root × ... × T_parent
  └ Výstup → pop transformace ze zásobníku

Příklad:
  Kořen → T1 → {Stůl, T2 → Váza}
  Stůl:  transformace = T1
  Váza:  transformace = T1 × T2
```

**Materiálové vlastnosti** se neskládají — platí **poslední nalezená** hodnota (přepsání, ne akumulace).

### Instancing — DAG místo stromu

Pokud se objekt ve scéně opakuje, definujeme geometrii **jednou** a na další výskyty odkazujeme ukazateli:

```
        [Kořen]
       /        \
    [T1]         [T2]
     │             │
  [VZOR]←────[INSTANCE]
  geometrie    (ukazatel)

VZOR:    transformace T1
INSTANCE: transformace T2  ← NE T1·T2!
          (ukazatel neakumuluje transformace předchůdců vzoru)
```

**Výhody instancingu:**

- Šetření paměti (geometrie uložena 1×)
- Propagace změn (změna vzoru = změna všech instancí)
- Označení: "zhroucený strom" (collapsed tree) — uzel má více předchůdců → DAG

**Omezení:** ukazatel nesmí ukazovat na svého předchůdce (cyklus → nekonečná rekurze).

### Mesh Graph

```
Zobecnění indexované polygonální sítě:
  - Orientovaný acyklický graf (DAG)
  - Jen vrcholy nesou geometrická data
  - Podporuje vícerozměrné hierarchie
```

### Praktické aspekty

- **Světelné zdroje v podstromu** → přirozené omezení působnosti světla
- **Více souborů** pro různé typy dat (textury, geometrie, materiály) → VRML, glTF
- **Polygon soup** = opak scénového grafu; množina polygonů bez hierarchie → neefektivní pro editaci

## ⚠️ Časté státnicové chytáky

### 1. CCW = front face — nezaměnit!

Studenti přehazují orientaci. Counter-clockwise (z pohledu pozorovatele) = přední strana = normála míří K pozorovateli. GPU zahazuje CW stěny při back-face cullingu.

### 2. Winged-edge — nezmínit vůbec

Z výsledků státnic: nejčastěji vynechávané téma. Winged-edge je strukturou pro topologická sousedství; **platí jen pro 2-manifoldy** (každá hrana = 2 sousední plochy). Pro non-manifoldy použij half-edge.

### 3. Indexovaný seznam vs. prostý seznam (triangle soup)

Studenti říkají "sdílení vrcholů" ale neřeknou klíčové slovo **indexed list** / **indexed buffer**. Examinátory zajímá: "Jak konkrétně jsou data uložena?"

### 4. Geometrická data vrcholu — tangenty!

Studenti jmenují: pozice, normála, UV. Zapomínají: **tangenta**, **bitangenta** (nutné pro normal mapping) a **skinning data** (joint ID + váhy pro kosterní animaci).

### 5. CSG pro GPU rendering

CSG strom sám o sobě není přímo renderovatelný na GPU. Nutno buď konvertovat na B-Rep (tessellace, marching cubes) nebo použít speciální ray casting shader.

### 6. Graf scény: kumulativní transformace od kořene

Studenti ví o hierarchii, ale zapomínají říct: transformace se **násobí (skládají) od kořene k listu** pomocí zásobníku při DFS traversal. Materiálové vlastnosti se neskládají — platí poslední nalezená.

### 7. Graf scény: instancing = DAG, ne strom

Pokud uzel odkazuje na sdílenou geometrii, graf přestane být strom a stane se **DAG** (directed acyclic graph). Chybějící zmínka o DAG je přesný typ chyby zaznamenaný u státnic.

### 8. Typy uzlů grafu scény — material a light uzly

Zmínit geometrický a transformační uzel nestačí. Úplný seznam: geometrický, transformační, skupinový, **materiálový** a **světelný** uzel. Světelný uzel v podstromu přirozeně omezuje působnost světla.

### 9. Objemová reprezentace — "existuje, ale bez detailů"

Zmínit "voxely a octree" nestačí. Examinátory zajímá: octree = 8 potomků, adaptivní dělení, operace OR/AND/MINUS mají složitost O(N_A · log N_B).

### 10. Euler–Poincaré: zaměnit R a H

R = díry **ve stěnách** (rings = otvory v ploše), H = **průchozí díry objektem** (topologické handles jako u torusu). Krychle: V=8, E=12, F=6, R=0, H=0, S=1 → 8-12+6-0 = 2(1-0) = 2 ✓.

### 11. Triangle strip vs. fan — kdy který

Strip: sekvence trojúhelníků sdílející hranu → lineární pruh. Fan: všechny trojúhelníky sdílí 1 centrální vrchol → vzniká triangulací konvexního polygonu.

### 12. Point cloud — vůbec nezmíněn

Zaznamenaná chyba: zapomenout na mračna bodů jako samostatnou B-Rep variantu (digitalizace reálných objektů, LiDAR, fotogrammetrie).
